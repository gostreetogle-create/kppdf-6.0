# 02_Договор/00-spr/00-glossary.md — Глоссарий терминов модуля «Договор»

> ⚠️ **STUB.** Создан декомпозицией (см. [MODULE-DECOMPOSITION-PLAN.md §2](../../99_Справочники/MODULE-DECOMPOSITION-PLAN.md)). Наполнение содержанием — Аналитик Run 2 (ТЗ-007).

## Назначение

Глоссарий терминов модуля `Договор` — единый источник для всех downstream-документов. Все термины должны совпадать с [`../../99_Справочники/GLOSSARY-MASTER.md`](../../99_Справочники/GLOSSARY-MASTER.md).

## Термины (8 ключевых, из `МОДУЛЬ-ДОГОВОР.md` §0)

| Термин | Что значит |
|---|---|
| **Договор (Contract)** | Юридический документ между нашей организацией и клиентом. Уникальный номер **Д-XXXX**. |
| **Спецификация** | Неотъемлемое приложение к Договору. Содержит позиции товаров/услуг из КП (snapshot с ценами и количеством). Юридически — часть Договора. |
| **Рамочный договор** | Тип Договора, у которого **нет** Спецификации с ценами — фиксируется только номенклатура и количество. Цены согласовываются позже по факту поставки. |
| **Договор поставки со Спецификацией** | Тип по умолчанию. К нему автоматически формируется Спецификация с ценами. |
| **Шаблон договора** | Визуальный макет А4 документа (как шаблон КП). Управляется администратором. |
| **Parent КП (parentProposalId)** | Ссылка на КП, из которого был создан Договор. Один КП = один Договор. **НЕ NULL** — Договор не может существовать в системе без КП-родителя. |
| **Parent Договор (parentContractId)** | Ссылка на старую версию Договора (если создаётся новая версия). Может быть `NULL` для первой версии. |
| **Неустойка** | Фиксированный процент за просрочку (настраивается per Договор через справочник значений 0.05/0.1/0.15/0.2% в день). См. OQ-Д-004. |

## Типы статусов (cross-ref)

Полный список статусов — в [`../03-zhiznennyj-cikl/03-statusy.md`](../03-zhiznennyj-cikl/03-statusy.md). Краткая сводка:

- `DRAFT` (Черновик)
- `SENT` (Отправлен клиенту)
- `SIGNED_CLIENT` (Подписан клиентом)
- `IN_PROGRESS` (В работе)
- `COMPLETED` (Завершён)
- `ARCHIVED` (Архив)
- `TERMINATED` (Расторгнут, добавлен 24.06.2026)

## Принципы наполнения (Аналитик Run 2)

1. **Один термин — одно определение.** Если термин пересекается с КП/Производством — дать cross-ref, не дублировать.
2. **Snapshot-семантика:** все поля юр-лиц, ФИО подписантов, реквизиты — фиксируются в момент подписания. Изменения в Org НЕ влияют задним числом.
3. **Связь с общим глоссарием:** новые термины, специфичные для Договоров → добавлять **сюда** + **в GLOSSARY-MASTER.md**.
