# 02_Договор/00-spr/00-products.md — Товары / услуги / работы в Спецификации Договора

> ⚠️ **STUB.** Создан декомпозицией (см. [MODULE-DECOMPOSITION-PLAN.md §2](../../99_Справочники/MODULE-DECOMPOSITION-PLAN.md)). Наполнение — Аналитик Run 2 (ТЗ-007).

## Назначение

Справочник номенклатуры Спецификации Договора (товар / услуга / работа) с snapshot-семантикой. Каждая позиция фиксируется в момент конвертации КП → Договор (`ContractItem`) и **не меняется задним числом**.

## Типы номенклатуры в Спецификации

### 1. Товар (ITEM)

| Поле | Тип | Snapshot? |
|---|---|---|
| `productId` | FK → Product | нет (FK) |
| `productSku` | String | **да** |
| `productName` | String | **да** |
| `quantity` | Decimal(15,3) | **да** |
| `priceWithoutVat` | Decimal(15,2) | **да** (NULL для рамочного) |
| `vatRate` | Decimal(5,2) | **да** (default 20) |
| `total` | Decimal(15,2) | **да** = `(quantity × priceWithoutVat) × (1 + vatRate/100)` |
| `lineDiscountPercent` | Decimal(5,2) | **да** (опционально, по В5.4) |

### 2. Услуга (SERVICE)

Аналогичные поля, но `productId` может быть `NULL` (если услуга ad-hoc, не из справочника).

### 3. Работа (WORK)

Аналогичные поля. Обычно из справочника, `productId` обязателен.

### 4. Монтаж (INSTALLATION)

Аналогичные поля. `productId` обязателен (из справочника). Создаёт отдельную задачу в Производстве (см. `МОДУЛЬ-ПРОИЗВОДСТВО.md` §6.3).

## Snapshot-семантика

> **Критическое правило:** все snapshot-поля фиксируются в момент создания `Contract`. Изменения в справочнике `Product` (переименование, изменение базовой цены) **НЕ влияют** на подписанную Спецификацию.

**Почему:** юридически подписанный Договор нельзя «исправить задним числом» через изменение справочника. Если менеджер ошибся в Договоре → «Создать новую версию» (`parentContractId` chain).

## Рамочный vs Со Спецификацией

| Тип Договора | `priceWithoutVat` | `quantity` |
|---|---|---|
| **Со Спецификацией** | обязателен | обязателен |
| **Рамочный** | `NULL` | обязателен (но без цены → согласовывается по факту поставки) |

## Связь с другими модулями

- **`ContractItem.parentContractId` → `Contract.id`** (ON DELETE CASCADE при архивировании).
- **`ContractItem.productId` → `Product.id`** (FK, не snapshot — нужен для текущих отчётов, но **не для финансовых расчётов** — те по snapshot-полям).
- **`SCHEMA-CONSOLIDATED.md` §2** — `ContractItem` entity полностью.
