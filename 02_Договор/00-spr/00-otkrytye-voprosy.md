## OQ-Д-001. Версионирование Договора — snapshot vs delta?

**Context.** `МОДУЛЬ-ДОГОВОР.md` §5.4 описывает «Создать новую версию» — создаётся `Д-XXXX-N`, старая версия получает приписку «superseded». Но: сохранять ли **все версии** в БД (полная история) или только последнюю (с потерей истории)?

| Variant | Описание |
|---|---|
| A | Хранить все версии (parentContractId chain) — полный audit trail |
| B | Хранить только последнюю, старую помечать `isActive=false` + `supersededAt` |
| C | Хранить все + parallel soft-delete через `isActive` (комбинация) |

**Рекомендация Архистратора:** A — для юридической прослеживаемости. Полная история версий нужна для возможных споров (кто что подписал и когда). Дополнительная нагрузка на БД — минимальна (десятки-сотни версий на клиента, не миллионы).

---

## OQ-Д-002. Реестр нескольких подписантов с разными ролями (юрист + главбух + коммерческий) — в v1 или v2?

**Context.** `МОДУЛЬ-ДОГОВОР.md` §9 явно: «Множественные подписанты с разными ролями — пока один подписант». Но это часто блокирует реальные продажи в среднем бизнесе (юрист должен визировать до подписания клиентом).

| Variant | Описание |
|---|---|
| A | v1: один подписант (signedByContractorAt — одна дата). v2: расширение. |
| B | v1: цепочка подписантов через `ContractSignature` (subentity, ordered) — больше кода, но реальнее |
| C | Только soft-approval workflow (checkbox «юрист визировал» без подписи) |

**Рекомендация Архистратора:** A в v1 (минимум в первой версии) — соответствует минимальному MVP. Пересмотреть в v2 при появлении реальных запросов клиентов.

---

## OQ-Д-003. Авто-архивирование «Завершённых» Договоров — через 30 дней ИЛИ через 90 дней ИЛИ ручное?

**Context.** `МОДУЛЬ-ДОГОВОР.md` §5.3 правило 3: «Архив — автоматически через 30 дней или вручную админом». Но в `МОДУЛЬ-ПРОИЗВОДСТВО.md` §2 аналогичное правило = **90 дней**. Нужна гармонизация модулей.

| Variant | Описание |
|---|---|
| A | 30 дней для Договоров (быстрее прячем из workspace) |
| B | 90 дней (унифицировано с Производством) |
| C | Ручное только, без автоматики |

**Рекомендация Архистратора:** B (90 дней) — единое правило для всех долгоживущих документов. Унификация снижает cognitive load и риск ошибок «забыли перевести вручную».

---

## OQ-Д-004. Неустойки — фиксированные проценты или настраиваемые per Договор?

**Context.** `МОДУЛЬ-ДОГОВОР.md` §11 упоминает «неустойки фиксированные %», но без конкретики (за что, сколько, как настраиваются). Это часто per-клиентское.

| Variant | Описание |
|---|---|
| A | Фиксированные в шаблоне Договора (например, 0.1% в день для всех) |
| B | Настраиваемые per Договор через dropdown % (выбор из справочника) |
| C | Полностью настраиваемое поле (текстовый ввод %) |

**Рекомендация Архистратора:** B — справочник фиксированных значений (0.05%, 0.1%, 0.15%, 0.2% в день) + менеджер выбирает при создании Договора. Юридическую валидность оставим юристу (он же согласовывает шаблон).

---

## OQ-Д-005. Пролонгация рамочных договоров — auto ИЛИ по запросу?

**Context.** `МОДУЛЬ-ДОГОВОР.md` §8 описывает автоматическую пролонгацию. Но: для многих клиентов пролонгация НЕ автоматическая — требуется пересмотр условий ежегодно.

| Variant | Описание |
|---|---|
| A | Всегда автоматически (если `validUntil` истёк → новый `validUntil = old + 1 год`) |
| B | Per-Договорный флаг `autoProlongate: Boolean` (менеджер при создании) |
| C | Default = false (ручная пролонгация через «Создать новый Договор») |

**Рекомендация Архистратора:** B — гибкий per-Договорный флаг. По умолчанию = true (привычный паттерн), но клиент-специфичные Договоры могут быть false.
