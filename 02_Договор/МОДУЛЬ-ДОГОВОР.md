# Модуль «Договор»

> Это документ модуля «Договор» в шестой версии проекта. Описан так же, как `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`: простым языком для нового менеджера. Если попалось непонятное место — задайте вопрос в файле `OPEN-QUESTIONS-LEGACY.md`.
>
> → **Связанные модули:** модуль КП (откуда приходят данные), модуль Производство (получает ссылку на Договор), модуль Финансы (OrderClosing привязан к Договору через производственный заказ). Границы MVP — раздел 9 в конце.

---

## 0. Глоссарий терминов модуля

| Термин | Что значит |
|---|---|
| **Договор (Contract)** | Юридический документ между нашей организацией и клиентом. Уникальный номер **Д-XXXX**. |
| **Спецификация** | Неотъемлемое приложение к Договору. Содержит позиции товаров/услуг из КП (snapshot с ценами и количеством). Юридически — часть Договора. |
| **Рамочный договор** | Тип Договора, у которого нет Спецификации с ценами — фиксируется только номенклатура и количество. Цены согласовываются позже по факту поставки. |
| **Договор поставки со Спецификацией** | Тип по умолчанию. К нему автоматически формируется Спецификация с ценами. |
| **Шаблон договора** | Визуальный макет А4 документа (как шаблон КП). Управляется администратором. |
| **Parent KП (parentProposalId)** | Ссылка на КП, из которого был создан Договор. Один КП = один Договор. НЕ NULL — Договор не может существовать в системе без КП-родителя. |

## 1. Для чего нужен этот модуль

**Договор** — это юридическое закрепление того, о чём договорились в КП. КП — это коммерческое предложение (по сути — оферта). Договор — это согласие обеих сторон с обязательствами, подписанное.

### Для кого он работает

| Роль | Что делает в модуле |
|---|---|
| **Менеджер по продажам** | Конвертирует КП в Договор, выбирает тип («со Спецификацией» / «рамочный»), отправляет клиенту на подпись |
| **Директор / руководитель** | Подписывает Договор от лица нашей организации, если менеджер сам не имеет таких прав |
| **Юрист** *(если есть в штате)* | Проверяет корректность формулировок в шаблоне перед отправкой клиенту (опционально в первой версии) |
| **Администратор системы** | Настраивает шаблоны договоров, справочники условий |

### Главная боль, которую решает

Менеджеры прошлых версий жаловались: «КП отправили, клиент согласился, всё на словах — а потом начинается разнобой: кто-то утверждает, что цена другая; кто-то говорит, что состав другой». Договор — это **юридически зафиксированная договорённость**, после которой не должно быть разночтений.

## 2. Жизненный цикл Договора (бизнес-логика)

Статусов шесть. На каждом этапе менеджер видит, в каком состоянии находится Договор.

### Полная карта статусов

| Статус | Что значит | Что можно делать дальше |
|---|---|---|
| **Черновик** | Только что сконвертирован из КП. Ещё не отправлен клиенту | Редактировать, отправить клиенту, архивировать (если ошибся) |
| **Отправлен клиенту** | Договор ушёл клиенту на подпись | Получить подпись («Подписан клиентом»), отозвать/скорректировать (создать новую версию) |
| **Подписан клиентом** | Клиент подписал. Договор действует | Начать производство / отгрузить со склада (если есть готовое), вернуть в работу (ошибка подписи) |
| **В работе** | Договор действует, по нему ведутся работы (производство/поставка). Связанный Производственный заказ в активных статусах | Следить за ходом Производственного заказа, отметить завершение |
| **Завершён** | Все обязательства по Договору выполнены: товар отгружен, услуги оказаны, оплата пришла | Перейти в архив (автоматически через 30 дней или вручную админом) |
| **Архив** | Мягкое удаление. Не показывается в основном списке | Восстановить (если ошибка) |
| **Расторгнут** (добавлено 24.06.2026 — правка #3 МАСТЕР-АУДИТ-V6.md) | Договор аннулирован до полного исполнения. Order в Финансах переходит в `CANCELLED`. | Только после письменного уведомления клиента; Refund в Модуле Финансы |

**Диаграмма переходов (ASCII):**

```
                   ┌───────────────┐
                   │  [Черновик]   │ ← конвертирован из КП
                   └──────┬────────┘
                          │ Отправить клиенту
                          ▼
                   ┌───────────────┐
                   │ [Отправлен   │
                   │   клиенту]   │
                   └──────┬────────┘
                          │ Клиент подписал
                          ▼
                   ┌───────────────┐
   ┌─────────── │  [Подписан    │ ◀──── вернуть в работу
   │            │   клиентом]   │       (ошибка подписи)
   │ Отменить   └──────┬────────┘
   │  подпись          │ Договор в работе
   │                   ▼
   │            ┌───────────────┐
   │            │ [В работе]   │ ← связан с активным
   │            └──────┬────────┘    Производственным заказом
   │                   │ Производство завершено, отгрузка сделана
   │                   ▼
   │            ┌───────────────┐
   └──────────▶│  [Завершён]  │
                └──────┬────────┘
                       │ Архивировать (авто через 30 дней
                       │ или вручную)
                       ▼
                  [Архив]

  В любом статусе (кроме «Архив»): можно перевести в «Расторгнут» (TERMINATED — добавлено 24.06.2026)
  → [Расторгнут] ◀── Order.cancel + Refund в Модуле Финансы
```

## 3. КЛЮЧЕВОЙ СЦЕНАРИЙ: Конвертация КП → Договор

Это главный путь создания Договоров в системе. Договоры НЕ создаются вручную с нуля — только из одобренного КП.

**Ситуация.** Менеджер Иванов отправил КП-0042 клиенту «Спорткомплект». Клиент через неделю ответил «да, всё устраивает, высылайте договор». Менеджер ставит КП в статус «Принято». Теперь нужно создать Договор.

**Что происходит:**

1. Менеджер открывает КП-0042 в статусе «Принято».
2. В правой панели (группа «Кнопки действий») нажимает **«Конвертировать в договор»**.
3. Появляется мини-форма: «Выберите тип договора: Договор поставки со Спецификацией (по умолчанию) / Рамочный договор без Спецификации».
4. Менеджер выбирает «Договор поставки со Спецификацией».
5. Система **мгновенно**:
   - Создаёт новый документ Договор с уникальным номером **Д-0042** (нумерация по порядку, обычно совпадает с исходным КП-XXXX).
   - Присваивает статус «Черновик».
   - Копирует **все позиции из КП вместе с ценами, наценками и скидками** (snapshot) — они становятся Спецификацией, приложением к Договору.
   - Копирует реквизиты нашей организации и клиента из КП.
   - Подставляет условия оплаты из КП/Организации.
6. У КП-0042 статус меняется на **«Конвертировано»** (КП становится архивным «источником» для Договора).
7. Менеджер открывает Договор в редакторе, проверяет Спецификацию.
8. Нажимает **«Отправить клиенту»** — Договор переходит в статус «Отправлен клиенту». Система формирует PDF.
9. Клиент подписывает (физически или через ЭДО — последнее в следующей версии).
10. Менеджер ставит статус «Подписан клиентом» — Договор становится действующим.

По времени — **2 минуты**, не больше.

### Что НЕ происходит при конвертации

- **Не меняются** товары, количество, базовые цены в позициях (snapshot, неизменны с момента добавления в КП).
- **Не копируется статус КП** — в Договоре всегда начинается с «Черновик», даже если КП был в «Принято».
- **Не удаляется КП** — он остаётся в системе со статусом «Конвертировано».

## 4. Экраны модуля

### 4.1 Список Договоров (`/contracts`)

Таблица со всеми Договорами. Колонки:

- **Номер** Д-XXXX
- **Клиент** (наименование из Org)
- **Наша организация** (продавец)
- **Дата создания**
- **Статус** (с цветной меткой)
- **Сумма** (итого с НДС)
- **Связанный КП** (parentProposalId, ссылка «КП-XXXX от дата»)
- **Связанный Производственный заказ** (если есть)
- **Кнопка «Открыть»**

Фильтры:

- **По статусу** — все / черновик / отправлен / подписан / в работе / завершён / архив
- **По клиенту** — dropdown с поиском по ИНН/наименованию
- **По дате создания** — диапазон дат
- **Галочка «Только мои»** (для менеджера — только Договоры, созданные им лично)

### 4.2 Редактор Договора (`/contracts/[id]`)

3-зонный макет (Каркас-Kit — как в КП, та же глобальная дизайн-система):

- **В центре**: живой предпросмотр А4-документа.
- **Слева** (большая панель): dropdown разделов пакета — Товары/Услуги из КП-источника (read-only, для справки), Шаблоны документов.
- **Справа** (компактнее): группа «Основное» (название, продавец, клиент, шаблон договора), группа «Параметры» (срок действия, условия оплаты), группа «Кнопки действий» (Сохранить, Отправить, Архивировать, Создать новую версию, Добавить Спецификацию).

### 4.3 Печать / PDF

Кнопка «Печать PDF» в правой панели. Формирует PDF-документ по выбранному шаблону. Таблицы разрываются правильно между страницами, заголовки повторяются. Если длинный — несколько страниц, Спецификация выносится в отдельный лист.

## 5. Бизнес-правила

### 5.1 Правила создания Договора

1. **Один КП = один Договор.** Повторно сконвертировать нельзя (если уже сконвертирован — ошибка).
2. > **✅ СПОР-11 (24.06.2026):** Статус КП «Конвертировано» — **финальный**. КП не возвращается в «Принято» или другие статусы. Правки после конвертации — только через «Создать новую версию» КП. Это предотвращает хаос «КП превратилось в договор, но менеджер продолжил править».

**Конвертировать можно ТОЛЬКО из статусов КП «отправлено» и «принято»**. Из «черновик» — нельзя (нечего конвертировать). **Из «оплачено» — КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** в первой версии (юридически логично: сначала договор → потом принято → потом оплата. Договор «задним числом» к уже заплаченным деньгам — это юридически кривой документ, который может оспорить клиент или налоговая. Решение зафиксировано по итогам критики Q1). См. как это правило логически согласуется с порядком статусов в разделе 6.1.
3. **Только из статуса «Принято»** самый частый путь.
4. **Шаблон выбирается в редакторе КП ДО конвертации** (dropdown «Шаблон договора»). По умолчанию — Договор поставки со Спецификацией.
5. **В Договоре позиции НЕ редактируются** (только через новую версию Договора — см. раздел 7). Менеджер не может править суммы после конвертации (только в статусе «Черновик», до отправки клиенту).

### 5.2 Правила для Спецификации (приложения)

1. **По умолчанию** — копируется вместе с ценами, наценками, скидками.
2. **В Рамочном договоре** — копируется только номенклатура (название, единица измерения) и количество. Цена = NULL.
3. **Спецификация живёт ВНУТРИ Договора** (как раздел/приложение А4), а не отдельным документом.
4. **Спецификация — snapshot** позиций КП. Изменение справочника товаров после конвертации не влияет на Спецификацию.

### 5.3 Правила для статусов

1. **«Подписан клиентом»** — менеджер ставит вручную. Автоматического получения подписи нет (ЭДО — следующая версия).
2. **«Отменить подпись»** — если менеджер ошибся (подписал не тот договор), кнопка «Отменить подпись» возвращает в «Отправлен клиенту».
3. **«Завершён»** — ставится автоматически когда связанный Производственный заказ в статусе «Завершён» (открытый вопрос: кнопка вручную или автоматически — см. раздел 9).

### 5.4 Правила для версий Договора

1. **«Создать новую версию»** — кнопка в правой панели, доступна во всех статусах кроме «Архив».
2. Создаёт **новый документ Д-XXXX-N** (где N = версия) с теми же позициями, но обновляемыми. Старая версия получает приписку «superseded».
3. Поля `parentContractId`, `version Int` — уже в модели данных (для будущего versioning).
4. **В первой версии модуля** — простая in-place правка для черновика. Версионирование полноценно появится, когда дойдём до цикла 42 (P2).

### 5.5 Правила для юридических реквизитов

1. **Реквизиты нашей организации и клиента** — snapshot в момент создания Договора. Изменение реквизитов в Org НЕ меняет подписанный Договор (это юридически невозможно задним числом).
2. **ФИО подписанта** — snapshot в момент конвертации. Если в справочнике был Иванов, а потом Петров стал директором — в подписанных Договорах остаётся Иванов.
3. **Печать и логотип** — берутся из Organization в момент формирования PDF (это картинка, она не «юридически старая», можно обновить).

## 6. Связи с другими модулями

### 6.1 Связь с КП

- **Один КП → один Договор.** Связь через `Contract.parentProposalId → Proposal.id`. `ON DELETE RESTRICT` (нельзя удалить КП, пока есть Договор).
- При конвертации:
  - Статус КП → «Конвертировано».
  - Статус Договора → «Черновик».
- **Обратная связь из Договора в КП** — НЕ описана. КП остаётся в статусе «Конвертировано», дальнейшие изменения Договора не отражаются в КП-источнике.

**✅ Решено (V-024):** Конвертация КП в Договор из статуса «Оплачено» — **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНА** в первой версии. Юридическое обоснование: в РФ «фактический договор» без подписания — это юридически слабая конструкция; если клиент заплатил без договора — это его риск, но система **не должна помогать** создавать кривые документы «задним числом». Решение зафиксировано в КП-доке (теперь кнопка «Конвертировать в договор» доступна только в «отправлено» и «принято»).

**[GAP] V-026:** Меняется ли статус КП после «Подписан клиентом»/«Завершён» в Договоре? **Не в первой версии**: КП остаётся в статусе «Конвертировано» навсегда. Договор живёт своей жизнью. Если хочется — открытый вопрос в разделе 9.

### 6.2 Связь с Производственным заказом

- В `ProductionOrder` есть поле `parentContractId → Contract.id` (FK, обычный, не строгий — можно nullable).
- Производственный заказ СОЗДАЁТСЯ при статусе КП «Оплачено», а не Договора. Но в ProductionOrder можно хранить ссылку на Договор — для удобства отчётов и поиска.
- Если КП→Договор создан, а потом КП «оплачено» → ProductionOrder с parentContractId указывающим на Договор.

### 6.3 Связь с Финансами

- В `OrderClosing` поле `productionOrderId → ProductionOrder.id` (из ANALYSIS-KP-priority.md). Через эту связь можно добраться до Договора, если нужно.

### 6.4 Связь с Картотекой сделки (ранее — «Пакет документов»)

- **Картотека = ВИРТУАЛЬНОЕ ПРЕДСТАВЛЕНИЕ** (`SELECT ... UNION ...`), а не отдельная сущность в БД. Все документы сделки (КП, Договоры, Производственные заказы) опционально тегируются общим `packageTag` (по `customerId`), а директор/админ через фильтр `packageTag=X` видят «всё по сделке» в одном окне. Это проще, чем отдельная сущность-суперконтейнер, и не ломается при удалении/архивации документов. Менеджер видит свои Договоры в списке `/contracts` (фильтр по `customerId`).

### 6.5 Связь со Справочником организаций

- `Contract.contractorId → Organization.id` (наша организация-продавец, FK).
- `Contract.customerId → Organization.id` (клиент, FK).
- `ON DELETE RESTRICT` для обеих — нельзя удалить организацию, пока есть её Договоры.

## 9. Что НЕ делаем в первой версии (границы MVP)

- **Электронная подпись (ЭЦП)** — ставим статус «Подписан клиентом» вручную.
- **Интеграция с ЭДО** (СБИС, КонтурДиадок и т.п.) — следующая версия.
- **Автоматические акты** (акты сверки, акты выполненных работ) — отдельный модуль Финансы.
- **Автоматический биллинг / invoice** — следующая версия.
- **Множественные подписанты с разными ролями** (юрист, главный бухгалтер, коммерческий директор) — пока один подписант.
- **Печатные формы с вложениями** (можно ли прикрепить файл к Договору) — нет, файлы живут в Пакете документов.
- **Языковые версии договоров** (русский + английский) — отложено для мультиязычности.
- **Drag-and-drop блоков** в шаблоне Договора — нет (КП-док тоже отложил).
- **Мобильная версия** редактора Договора — нет.

## 10. Открытые вопросы — ✅ ВСЕ 6 ЗАКРЫТЫ (24.06.2026)

> Источник: `OPEN-QUESTIONS-MASTER.md`. Каждое решение применено в §5 (бизнес-правила) + §6 (связи с модулями) + §8 (RBAC).

### Q1 (V-024). ✅ РЕШЕНО: A (финальное)
- В КП-доке сейчас только «отправлено» и «принято». А если деньги пришли до подписания договора (редко, но бывает)?
- **Вариант A:** Запрещено — нужно сначала оформить договор, потом принимать оплату.
- **Вариант B:** Можно — создаётся Договор «задним числом», к которому привязывается уже совершённая оплата.
- **✅ РЕШЕНО: A (финальное, 24.06.2026 — правка B).** Юр. безопасный вариант зафиксирован в §5.1 правило 2 и §6.1. Конвертация КП → Договор из статуса «Оплачено» — **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНА** в v1. Если клиент заплатил без договора — это его риск; система не должна помогать создавать кривые документы «задним числом». Если деньги реально пришли — возврат/зачёт через отдельный платёжный путь (см. МОДУЛЬ-ФИНАНСЫ.md §6 правила возврата и Storno/Refund). Дополнительно см. GAP-011 (откат ошибочного оплачивания).

## 11. Триггер Order = подписание Договора

**При подписании Договора** (`Contract.status='SIGNED'`) автоматически создаётся `Order` в Модуле Финансы в статусе `draft` → авто-переход в `in_progress` (СПОР-5). Связь `Order.contractId` объявлена как **NOT NULL, RESTRICT ON DELETE** — Договор нельзя удалить при наличии привязанного Order (только через закрытие цепочки по refs или новую редакцию Договора через `parentContractId`). Это финальное решение СПОР-5, согласованное PO 24.06.2026 в рамках закрытия `OPEN-QUESTIONS-MASTER.md`.

Подробнее — `МОДУЛЬ-ФИНАНСЫ.md` §4 «Цепочка связей» + Q1 раздела «МОДУЛЬ-ФИНАНСЫ».
