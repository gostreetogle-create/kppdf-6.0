> ⚠️ **STUB.** Создан декомпозицией (см. [MODULE-DECOMPOSITION-PLAN.md §2](../../99_Справочники/MODULE-DECOMPOSITION-PLAN.md)). Наполнение — Аналитик Run 2 (ТЗ-007, правила аналогичны ТЗ-002 КП).

## Назначение

Документирует **инварианты и валидации** для Договора — что должно быть true перед любым действием. Нарушение любого инварианта = ошибка UI + audit log. Источник: `МОДУЛЬ-ДОГОВОР.md` §5.

## 6 групп инвариантов

### 1. Правила созданияДоговора (5 правил)

| # | Правило | Ошибка |
|---|---|---|
| 1.1 | **Один КП = один Договор.** Повторно сконвертировать нельзя | 422 «Уже сконвертирован» |
| 1.2 | **Конвертировать можно ТОЛЬКО из статусов КП «отправлено» и «принято»** | 422 «Неверный статус КП» (СПОР-11 + V-024) |
| 1.3 | **Из «Оплачено» — КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО** в v1 | 422 «Запрещено: сначала договор» |
| 1.4 | **Шаблон выбирается ДО конвертации** (в редакторе КП) | 422 «Шаблон не выбран» |
| 1.5 | **Позиции НЕ редактируются** в Договоре после конвертации | 422 «Создайте новую версию» |

### 2. Правила для Спецификации (4 правила)

| # | Правило | Ошибка |
|---|---|---|
| 2.1 | **По умолчанию** = копируется вместе с ценами, наценками, скидками (для типа «со Спецификацией») | — |
| 2.2 | **В Рамочном Договоре** = копируется только номенклатура + количество, цена = NULL | — |
| 2.3 | **Спецификация живёт ВНУТРИ Договора** (как раздел/приложение А4) | 422 «Не отдельный документ» |
| 2.4 | **Спецификация — snapshot.** Изменение справочника товаров после конвертации НЕ влияет | — (это фича) |

### 3. Правила для статусов (3 правила)

| # | Правило | Ошибка |
|---|---|---|
| 3.1 | **`SIGNED_CLIENT`** — менеджер ставит вручную (нет авто-получения подписи) | — |
| 3.2 | **`SENT → DRAFT`** (отзыв) разрешён, но только до получения подписи | 422 «Уже подписан» |
| 3.3 | **`COMPLETED`** — ставится автоматически когда связанный ProductionOrder в COMPLETED (через 1 день — отложенный триггер) | — |

| 4.2 | Новая версия = новый `Д-XXXX-N` с обновляемыми позициями | — |

### 6. Правила для триггеров (2 правила)

## Принципы наполнения (Аналитик Run 2)

1. **Одно правило = одна ситуация.** Не «всё создание Договора» — разбить на 5+ правил.
2. **Код ошибки** = стабильный (422, 403, 409). Менять опасно — это API contract.
3. **Side-effect** = явный кросс-реф на запускаемый код-модуль (`МОДУЛЬ-ФИНАНСЫ`, `МОДУЛЬ-СКЛАД`).
4. **СПОР/Q** — каждое правило со ссылкой на закрытое решение в [`../../99_Справочники/СПОРНЫЕ-МОМЕНТЫ.md`](../../99_Справочники/СПОРНЫЕ-МОМЕНТЫ.md).
