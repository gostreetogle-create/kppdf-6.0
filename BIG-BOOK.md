## 1. Что такое v6? Бизнес-цель

### 1.1 Что это за CRM

**KPPDF CRM v6** — система управления коммерческим циклом для компании, занимающейся производством и поставкой спортивного оборудования (и смежных металлоизделий). Это НЕ широкая CRM типа Битрикс24, а **специализированный инструмент под конкретный бизнес-цикл «КП → Договор → Производство → Склад → Финансы»**.

### 1.2 Пользователи (6 ролей + 1 viewer)

| Роль | Что делает | Главный экран |
|---|---|---|
| **Менеджер по продажам** | Создаёт КП, отправляет клиентам, конвертирует в Договор, ставит «Оплачено» | `/proposals` |
| **Директор** | Стратегический обзор, видит себестоимость/margin, утверждает крупные списания | `/packages` (Картотека), `/reports` |
| **Бухгалтер** | Регистрирует входящие платежи, выставляет счета, approve мелких списаний, аннулирует ошибочные платежи | `/finance/orders` |
| **Начальник производства** | Ведёт производственные заказы (ЗК), распределяет задачи по цехам | `/production` |
| **Кладовщик** | Принимает приходы, оформляет отгрузки/списания, контролирует остатки | `/warehouse` |
| **Закупщик (manager расширенный)** | Оформляет заявки на закупку, контролирует приходы от поставщиков | `/purchase` |
| **Админ** | Настраивает справочники (товары, клиенты, шаблоны), управляет RBAC | `/admin/*` |

### 1.3 Боль, которую решает

| Боль | Решение через v6 |
|---|---|
| 15 минут на «КП для другого юрлица» | Кнопка «Скопировать для другого юрлица» — 30 секунд + авто-применение наценки/НДС |
| Долгая сборка корзины товаров | Инлайн-поиск в пустой строке таблицы (главная механика v3.0) |
| Расхождения «по бумаге 50 шт., реально 47» | Все операции = документы с автором/датой (StockMovement immutable) |
| «Какие у нас там юрлица и кто подписывает» | Единый справочник Organization с авто-подстановкой в шапку КП |
| «Прибыль по сделке?» | Детальный `OrderClosing` с маржой |
| «Кто что может?» | RBAC по 6 ролям + 1 viewer |

## 3. 5 модулей — краткий обзор

### 3.1 КП — Коммерческое предложение

**Что:** Документ, который менеджер отправляет клиенту с товарами, ценами, условиями. **Первый документ в цепочке продажи.**

**Префикс:** `КП-XXXX`. **6 статусов:** DRAFT → SENT → ACCEPTED → REJECTED → PAID → CONVERTED (финальный).

**Главный экран:** `/proposals/new` — витрина товаров + корзина. `/proposals/[id]` — редактор (3-зонный макет: витрина слева, лист A4 по центру, панель действий справа).

**Главные фичи:**
- **Инлайн-реиск товара** прямо в пустой строке таблицы (главная механика добавления).
- **3-зонный макет «Каркас-Kit»** — глобальный дизайн-каркас, единый для всех документов.
- **Кнопка «Скопировать для другого юрлица»** — смена нашей организации-продавца за 30 сек + авто-применение её НАЦЕНКИ и НДС.
- **Автосохранение черновика**: debounce 5-10 сек + localStorage для защиты от потери.
- **Версионирование КП** через `parentProposalId` (линейная структура версий).

**Ключевое правило:** Конвертация КП «Оплачено» → Договор **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНА**. Юр. логика: КП не считается офертой после оплаты.

↪ Подробнее: [`01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`](01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md) (14 разделов, ~945 строк)

### 3.2 Договор — Contract

**Что:** Юридический документ, подписывается клиентом. Содержит Спецификацию (позиции с ценами) или рамочные условия без цен.

**Префикс:** `Д-XXXX`. **7 статусов:** DRAFT → SENT → SIGNED → IN_PROGRESS → COMPLETED → ARCHIVED → **TERMINATED** (добавлен 24.06.2026 для расторжения после оплаты).

**Главный экран:** `/contracts/[id]` — редактор с шапкой/футером, Спецификацией, ссылками на родительский КП.

**Ключевая фича:** При подписании Договора (`status='signed'`) автоматически создаётся **Order в Финансах** (СПОР-5) — это триггер всей финансовой части цепочки.

**Связь:** 1 Договор = 1 ЗК (в v1; v2 возможно multi-ЗК).

↪ Подробнее: [`02_Договор/МОДУЛЬ-ДОГОВОР.md`](02_Договор/МОДУЛЬ-ДОГОВОР.md) (11 разделов)

### 3.3 Производство — ЗК (ProductionOrder)

**Что:** Производственный заказ, создаваемый **автоматически** при КП `paid`. Распределение задач по цехам, отслеживание прогресса.

**Префикс:** `ЗК-XXXX`. **8 статусов:** CREATED → PLANNING → AWAITING_MATERIALS → IN_PROGRESS → PARTIAL → COMPLETED → CANCELLED → ARCHIVED.

**Главный экран:** `/production` — доска всех ЗК с фильтрами по статусу/цеху/ответственному.

**Ключевые фичи:**
- **Авто-создание при КП `paid`** — менеджер не создаёт ЗК вручную.
- **Тип позиции:** `ITEM` (товар) → авто-приход в Склад при завершении. `SERVICE`/`WORK`/`INSTALLATION` → НЕ приходит в Склад (R1/GAP-009).
- **Прогресс** = `SUM(quantityActual) / SUM(quantityPlanned) × 100%`.

↪ Подробнее: [`03_Производство/МОДУЛЬ-ПРОИЗВОДСТВО.md`](03_Производство/МОДУЛЬ-ПРОИЗВОДСТВО.md) (11 разделов)

### 3.4 Склад — Warehouse

**Что:** Операционная точка контроля движения товара. 4 базовые операции = 4 документа:

| Документ | Префикс | Статусов | Что делает |
|---|---|---|---|
| SupplierDelivery (Приход от поставщика) | `СД-XXXX` | 6 | Приёмка товара → StockMovement: IN |
| Shipment (Отгрузка клиенту) | `ОТК-XXXX` | 6 | Отгрузка → StockMovement: OUT |
| WriteOffAct (Акт списания) | `АС-XXXX` | 5 | Списание (брак/недостача) → StockMovement: OUT, требует approve |
| PurchaseOrder (Заявка на закупку) | `ЗП-XXXX` | 8 | Заказ поставщику → потом приход через СД |

**5 страниц на «Каркас-Kit»:**
- `/warehouse` — главная (остатки + журнал движений).
- `/supplier-deliveries`, `/shipments`, `/purchase`, `/write-off`.

**Ключевые фичи:**
- **availableQty = quantity − reservedQuantity** — главная цифра для менеджера.
- **Immutable StockMovement** — никаких ретро-правок.
- **Авто-предложение** создать Shipment при завершении ЗК (`kind='ITEM'`).

### 3.5 Финансы — Order/Invoice/Payment/Refund

**Что:** Финансовый заказ (Order), счёт (Invoice), входящие платежи (Payment), возвраты (Refund). Создаётся **автоматически** при подписании Договора.

**Префиксы:** `Order-XXXX`, `Inv-XXXX`, `Pay-XXXX`, `Ref-XXXX`.

**5 статусов Order:** DRAFT → IN_PROGRESS → AWAITING_PAYMENT → CLOSED → **CANCELLED** (добавлен 24.06.2026 для отмены после оплаты).

**Ключевая архитектура:**

```
Payment.type = INCOMING  (amount > 0)   — обычное поступление
Payment.type = STORNO    (amount < 0)   — техническая корректировка опечатки бухгалтера
                                       (деньги физически НЕ двигались)
Refund.amount > 0        — РЕАЛЬНЫЙ возврат денег клиенту
```

**Формула net payment для Order:**
```
Order.paidAmount = Σ INCOMING.amount − Σ STORNO.amount − Σ Refund.amount
```

**Ключевая фича:** Маржа доступна только ролям с правом `costPrice:read` (`admin/director/accountant`). **Менеджер НЕ видит себестоимость** (GAP-014).

## 4. Полная цепочка «КП → Деньги»

### 4.1 Happy path (нормальный случай)

```
День 0:   Менеджер рисует КП-0001
День 1:   КП отправлен → клиент получил (SENT)
День 2:   Клиент «Принято» (ACCEPTED)
День 3:   Менеджер конвертирует в Договор-0001 → подписан (SIGNED)
         → АВТО Order-0001 (IN_PROGRESS)
День 5:   Клиент «Оплачено» КП → АВТО ЗК-0001 (CREATED)
День 6:   Цех принял → ЗК-0001 (IN_PROGRESS)
День 7:   Все задачи done → ЗК-0001 (COMPLETED)
         → АВТО StockMovement: IN (только для kind='ITEM')
         → авто-ПРЕДЛОЖЕНИЕ создать Shipment кладовщику
День 8:   Кладовщик упаковал → ОТК-0001 (PACKED → SHIPPED)
         → АВТО StockMovement: OUT
День 9:   Доставлено → DELIVERED → OrderClosing.progress++
День 10:  Бухгалтер выставил Invoice-0001
День 11:  Клиент платит → Payment-0001 (INCOMING)
День 12:  Все Invoice оплачены → Order-0001 (CLOSED)
         → МАРЖА рассчитана ✓
```

### 4.2 Автоматические триггеры (must-know)

| Триггер | Что создаёт | Источник |
|---|---|---|
| `Proposal.status='paid'` | авто `ProductionOrder` (ЗК) | `МОДУЛЬ-ПРОИЗВОДСТВО.md` §5.1 |
| `Contract.status='signed'` | авто `Order` в Финансах (СПОР-5) | `МОДУЛЬ-ФИНАНСЫ.md` §4 |
| `ProductionOrder.status='completed'` (только kind='ITEM') | авто `StockMovement: IN, reason=production` | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §11.8 |
| `ProductionOrder.status='completed'` + есть ITEM | авто-ПРЕДЛОЖЕНИЕ создать Shipment | `МОДУЛЬ-СКЛАД-UI.md` §4.3 |

## 5. Ключевые сущности и связи

### 5.1 Карта всех 32 сущностей (high-level)

```
Справочники:
  User → Organization (multi-role) → OrganizationSigner + OrganizationContact
  Product (kind: ITEM|SERVICE|WORK)
  DocumentTemplate, Workshop, Warehouse, Counter

Сделки:
  Proposal (КП) → Contract (Договор) → ProductionOrder (ЗК)
                                       → Shipment (ОТК)
                                       → SupplierDelivery (СД) ← PurchaseOrder (ЗП)
  
Финансы:
  Order ← Invoice ← Payment (type: INCOMING/STORNO)
                  → Refund (отдельная сущность, НЕ отрицательный Payment)

Связи сделок ↔ финансы:
  Proposal (1) → Contract (1) → Order (1) → [Invoice, Payment, Refund]
  ProductionOrder (1) → Shipment (N)
  ProductionOrder (1) → SupplierDelivery (N)
  StockRecord (Warehouse × Product) ← StockMovement (immutable)
```

### 5.2 Префиксы и счётчики (11 разных)

| Префикс | Сущность | Счётчик |
|---|---|---|
| `КП-XXXX` | Proposal | отдельный |
| `Д-XXXX` | Contract | отдельный |
| `ЗК-XXXX` | ProductionOrder | отдельный |
| `СД-XXXX` | SupplierDelivery | отдельный |
| `ОТК-XXXX` | Shipment | отдельный |
| `АС-XXXX` | WriteOffAct | отдельный |
| `ЗП-XXXX` | PurchaseOrder | отдельный |
| `Order-XXXX` | Order | отдельный |
| `Inv-XXXX` | Invoice | отдельный |
| `Pay-XXXX` | Payment | отдельный |
| `Ref-XXXX` | Refund | отдельный |

**Не синхронизированы между собой!** `КП-0042`, `Д-0042`, `ЗК-0042` могут одновременно существовать — это **нормально**.

**Safe-increment** через `SELECT FOR UPDATE` + serializable isolation (PostgreSQL) защищает от race conditions при параллельном создании.

## 6. Ключевые бизнес-правила (must-know)

1. **Конвертация КП «Оплачено» → Договор КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНА** (правка B). Юр. логика.
2. **Storno ≠ Refund** (2 разные сущности, GAP-023). Storno = техническая корректировка опечатки. Refund = реальный возврат денег.
3. **Приход от ЗК только для `Product.kind='ITEM'`** (R1/GAP-009). Услуги/работы = информационное сообщение, не движение.
4. **Snapshot-поля обязательны** на всех `*Item` (`productSku, productName, productUnit, priceWithoutVat, vatRate`). Защита от ретро-правок в справочниках.
5. **StockMovement immutable** — никогда не редактируется задним числом. Только встречные движения для исправления.
6. **Currency жёстко RUB в v1** — поле `currency` заведено «на вырост» для v2.
7. **availableQty = quantity − reservedQuantity** — главная формула остатка.
8. **Никаких каскадных переходов статусов между модулями** (R4). Каждый модуль управляет своими.
9. **Менеджер НЕ видит `costPrice`** (GAP-014). Себестоимость видна только `admin/director/accountant`.
10. **Картотека сделки = view** (SQL UNION по `packageTag`), НЕ отдельная таблица. Доступ: `admin/director`.
11. **Договор 1-to-1 ЗК в v1** (multi — v2).
12. **Списание требует approve** — двухступенчатый процесс с авто-маршрутизацией по сумме (accountant до 5000 ₽, director выше).
13. **Логотип/печать — только `admin/director` с 2FA** (GAP-016).
14. **`packageTag` — ручной тег сделки**, не auto. Связывает документы одной сделки через виртуальную Картотеку.

↪ Подробнее: `МОДУЛЬ-*.md` §9 «Бизнес-правила» в каждом модуле.

## 7. RBAC — кто что может

**Сводная матрица по 7 ролям × 30+ действиям** — в [`99_Справочники/RBAC-MATRIX.md`](99_Справочники/RBAC-MATRIX.md).

### Краткая сводка по ролям

| Роль | Главное отличие |
|---|---|
| **admin** | Полный доступ. Импорт данных, настройка RBAC. |
| **director** | Видит маржу и себестоимость. Approve крупных списаний. Расторгает договоры. |
| **accountant** | Финансовый контроль. Регистрирует платежи, approve списаний ≤5000 ₽, маржинальные отчёты. |
| **manager** | Работает с КП «своими». Не видит себестоимость. |
| **production** | Ведёт «свои» ЗК. Отгружает по своим ЗК. |
| **storekeeper** | Главный пользователь Склада. Все операции Склада. |
| **viewer** | Только чтение. |

### Критичные RBAC-инварианты

- **Менеджер НЕ видит `costPrice`** — поле скрыто в UI + API (GAP-014).
- **Картотека-comments только для `admin/director`** (GAP-013). Менеджер не видит комментарии директора про свою сделку.
- **Логотип/печать загружается ТОЛЬКО `admin/director`** с 2FA (GAP-016).
- **`manager` ограничен «свой»** — не может править чужие КП/Договоры/ЗК.
- **`storekeeper` главный пользователь Склада**, `accountant` — только approve мелких списаний.
- **`production_master`** (отдельная роль) — отложена в v2.

## 8. Глоссарий ключевых терминов

> Полный словарь из 60+ терминов — в [`99_Справочники/GLOSSARY-MASTER.md`](99_Справочники/GLOSSARY-MASTER.md). Здесь — критичные.

| Термин | Что значит |
|---|---|
| **КП / Proposal** | Коммерческое предложение. Префикс `КП-XXXX`. |
| **Договор / Contract** | Юридический документ. Префикс `Д-XXXX`. |
| **ЗК / ProductionOrder** | Производственный заказ. Префикс `ЗК-XXXX`. |
| **Склад / Warehouse** | Физическое место хранения товара. |
| **availableQty** | `quantity − reservedQuantity`. Что можно отгрузить. |
| **packageTag** | Уникальный тег сделки (вручную). Связывает документы одной сделки. |
| **Snapshot** | Копия данных на момент операции. Защита от ретро-правок. |
| **Картотека сделки (view)** | Виртуальное представление всех документов по сделке. Доступ: admin/director. |
| **Каркас-Kit (Karkas-Kit)** | Глобальный дизайн-каркас. 3-зонный макет: левая витрина / центр / правая панель. |
| **RUB** | Валюта **жёстко RUB** в v1. |
| **Storno** | Техническая корректировка Payment (type='STORNO', amount<0). Деньги не двигались. |
| **Refund** | Реальный возврат денег клиенту (отдельная сущность, amount>0). |
| **RBAC** | Управление правами по 7 ролям. |
| **Single-tenant** | v1 = одна компания. Multi-tenancy — v2. |
| **Soft delete (`isActive`)** | Логическое удаление. `isActive=false` = архив. Физическое удаление в v1 запрещено. |
| **Safe-increment** | `SELECT FOR UPDATE` в `$transaction` для защиты счётчиков от race conditions. |
| **RESTRICT (ON DELETE)** | Запрет удаления родителя при наличии дочерних FK. Применяется к критичным FK. |
| **Item (Snapshot)** | У каждой `*Item` таблицы (ProposalItem, ContractItem, и т.д.) — SKU/name/unit/price/vat и т.д. копируются на момент операции. |
