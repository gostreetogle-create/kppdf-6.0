### Полная цепочка: от заявки до закрытия

```
КП (черновик) → КП (отправлено) → КП (принято) → КП (оплачено) → ЗК (создан) → ЗК (в работе) → ЗК (завершён) → Склад (приход) → Склад (отгрузка) → Финансы (счёт) → Финансы (оплата) → Финансы (закрыт)
                                                                                       ↓
                                                                              КП (конвертировано) → Договор (черновик) → Договор (отправлен) → Договор (подписан) → Договор (в работе) → Договор (завершён)
```

### Найденные разрывы в цепочке:

| # | Разрыв | Описание | Критичность |

| R1 | **КП «оплачено» → ЗК: что если товар уже на складе?** | МОДУЛЬ-ПРОИЗВОДСТВО §3 описывает «всё в ЗК». МОДУЛЬ-СКЛАД §3 альтернативный сценарий: «КП «Со склада» — товар не требует производства». ## 6. Итоговый документ

### Общая оценка готовности к разработке: **5/10**

Модуль КП описан **сильно** (90% готовности). Остальные модули — **каркасы** (50-60% готовности). Модуль Финансы — самый слабый (40%).

### Список блокеров (P0) — без чего нельзя начинать:
1. Product.kind (ITEM/SERVICE/WORK)
2. TemplateBlock.type (конструктор шаблонов)
3. User поля (auth + RBAC)
4. Organization role «поставщик»

### Список важных уточнений (P1):
1. SupplierDelivery, Shipment, WriteOffAct, PurchaseOrder — поля
2. OrganizationSigner.isDefault
3. Физлицо: dateOfBirth обязательность
4. Optimistic locking для автосохранения
5. OrderClosing триггер
6. Распределение платежа по счетам
