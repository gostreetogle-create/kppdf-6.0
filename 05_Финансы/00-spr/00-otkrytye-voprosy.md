## OQ-Ф-001. Распределение платежа при двух счётах (аванс + финальный)

**Context.** `МОДУЛЬ-ФИНАНСЫ.md` §4 SC-06: «Один Order, два счёта (аванс + финальный). Платёж пришёл один на всю сумму.» СПОР-7 решён: ручной выбор в v1. Но: что если бухгалтер НЕ выбрал счёт?

| Variant | Описание |
|---|---|
| A | Платёж привязывается к Order в целом (invoiceId = null), распределение — ручное позже |
| B | Обязательный выбор счёта при регистрации (validation error если invoiceId = null) |
| C | Авто-FIFO (аванс → финальный) — но СПОР-7 отложил это в v2 |

**Рекомендация:** A — привязка к Order без указания Invoice, менеджер распределяет позже. Минимум в v1.

## OQ-Ф-002. Refund при частичном возврате — менять ли статус Order?

**Context.** `МОДУЛЬ-ФИНАНСЫ.md` §6: «Если проведён Refund на всю оставшуюся сумму при отменённом ЗК → `Order.status = 'CANCELLED'`.» Но: что при **частичном** возврате?

| Variant | Описание |
|---|---|
| A | Частичный Refund НЕ меняет статус Order (остаётся `AWAITING_PAYMENT`) |
| B | Частичный Refund меняет статус на `IN_PROGRESS` (требует ревью) |
| C | Отдельный статус `PARTIALLY_REFUNDED` |

**Рекомендация:** A — частичный возврат не влияет на статус. Полный возврат при отменённом ЗК → `CANCELLED`.

## OQ-Ф-003. Просроченный Invoice — уведомление или только UI-метка?

**Context.** `МОДУЛЬ-ФИНАНСЫ.md` §6: «Просроченный счёт считается просроченным на следующий день после `dueDate` при сальдо > 0. В списке заказов выделяется красноватым.» Но: автоматическое уведомление?

| Variant | Описание |
|---|---|
| A | Только UI-метка (красная подсветка), без уведомлений |
| B | UI-метка + email-уведомление менеджеру (через notification service) |
| C | UI-метка + dashboard-виджет «Просроченные счета» |

**Рекомендация:** A в v1 (минимум), B/C — v2 при росте объёмов.

---

## OQ-Ф-004. Закрытие Order с просроченным Invoice — двухступенчато или нет?

**Context.** `МОДУЛЬ-ФИНАНСЫ.md` §8 Q3: «Принудительное закрытие заказа → director + accountant двухступенчато.» Но: детали процедуры?

| Variant | Описание |
|---|---|
| A | Бухгалтер нажимает «Закрыть» → Director подтверждает (2 клика) |
| B | Бухгалтер ставит флаг «Готов к закрытию» → Director закрывает (workflow) |
| C | Только одна роль (бухгалтер) может закрыть без подтверждения |

**Рекомендация:** A — две кнопки в UI («Закрыть» → «Подтвердить закрытие»). Простая двухступенчатая проверка.

---

## OQ-Ф-005. Payment без привязки к Invoice — допустимо ли?

**Context.** `МОДУЛЬ-ФИНАНСЫ.md` §4: «Если клиент прислал платёж на несколько счетов — бухгалтер вводит несколько Payment, по одному на счёт, или один Payment без указания invoiceId (привязка к Order в целом).»

| Variant | Описание |
|---|---|
| A | Допустимо: `Payment.invoiceId` nullable, привязка к Order |
| B | Запрещено: `Payment.invoiceId` NOT NULL (всегда привязан к счёту) |
| C | Опционально: nullable, но UI показывает warning «Привяжите к счёту» |

**Рекомендация:** C — nullable с warning. Гибкость для реальных сценариев (клиент платит общей суммой).
