# LAUNCH-ARCHITECT.md — Package для запуска Архитектора (Phase 0, первый прогон Pipeline v6)

> **Назначение.** Готовый copy-paste пакет для запуска первого агента (Архитектора) в **новой сессии Codebuff**. Подготовлен 2026-06-26 после pre-launch: коммит `c65e666` запушен в `origin/main`.
>
> **Когда использовать.** Сразу ПОСЛЕ прочтения этого файла — открой `codebuff` (новый чат) и скопируй `## Промпт для Codebuff` ниже в первую строку + прикрепи файлы из `## Файлы для attach`.

---

## 0. Что должно произойти

1. **Открыть новую сессию Codebuff** (CLI: `codebuff` → New Chat).
2. **Прикрепить 10 файлов** из секции `## Файлы для attach` ниже.
3. **Вставить copy-paste промпт** из секции `## Промпт для Codebuff` (между ```text ... ```).
4. **Отправить** — Архитектор начнёт работу.
5. **Вернуться в эту сессию** с output'ом Архитектора → записать **PSL-004** + commit + push.
6. **Запустить следующего агента** (Бизнес-аналитик) уже на основе output'а Архитектора.

## 2. Промпт для Codebuff (СКОПИРОВАТЬ ЦЕЛИКОМ)

```text
Ты — Архитектор. Прочитай [FILES_ATTACHED].

Твоя задача: декомпозировать файл МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md
(945 строк) на 5 папок по принципу из AGENT-METHOD.md §3:
  00-spr → 01-shablon → 02-konstruktor-kp → 03-zhiznennyj-cikl → 04-pravila

Вход: 01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md (945 строк).

⛔ КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ — НАРУШЕНИЕ = OOM + дрейф ролей

1. СТРОГО ЗАПРЕЩЕНО писать содержимое самих .md файлов. Только
   мета-уровень: ДЕРЕВО папок (Markdown tree) + имена файлов +
   1-2 предложения комментария про НАЗНАЧЕНИЕ файла (ЧТО будет
   внутри, но НЕ само содержимое). Содержимое — работа других ролей.
   Нарушение → LLM-drift + перерасход контекста.

2. УЧИТЫВАЙ существующие файлы (проверь через list_directory/glob):
   - 01_КП/00-spr/00-otkrytye-voprosy.md УЖЕ СУЩЕСТВУЕТ (5 baseline
     OQ-001..OQ-005). ОБЯЗАТЕЛЬНО включи в дерево 00-spr/ как есть,
     НЕ дублируй и НЕ удаляй.
   - Сам МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md распускается по
     подфайлам или становится 00-spr/00-glossary.md.
   - В 01_КП/{01-shablon,02-konstruktor-kp,03-zhiznennyj-cikl,04-pravila}/
     уже есть STUB-маркеры 00-README.md (37-41 строк). ВКЛЮЧИ их в
     дерево как существующие точки входа для каждой папки.

3. Соблюдай агентную границу. НЕ пиши правила, поля, edge-кейсы
   (это Аналитик). НЕ пиши SQL/Prisma (это Моделировщик). НЕ пиши
   UX-макеты с состояниями кнопок (это UX). Ты = только размещение
   файлов по папкам + 1-2 предложения назначения для каждого.

4. Hard limit размера каждого проектируемого файла — см.
   AGENT-REVIEW.md §1.6 (target 80-250 / hard 400). Если предложенный
   файл будет > 400 строк — предложи разбить на 2-3.

Выход: Markdown-дерево папок + список файлов с обоснованием
(вход/выход), БЕЗ текстового содержимого файлов.

Применяй чек-лист самопроверки AGENT-REVIEW.md перед сдачей.
```

## 3. Ожидаемый формат ответа Архитектора

Архитектор должен вернуть **только это** (Markdown tree + таблица):

```text
## Дерево папок модуля 01_КП/

01_КП/
├── README.md                          # Точка входа модуля (новый)
├── 00-spr/
│   ├── 00-glossary.md                # ← из МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ §13
│   ├── 00-orgs.md                    # ← ...§X
│   ├── 00-otkrytye-voprosy.md        # ← СУЩЕСТВУЕТ (5 baseline OQ)
│   └── ...
├── 01-shablon/
│   ├── 00-README.md                  # ← СУЩЕСТВУЕТ (STUB-маркер)
│   ├── 01-konstruktor.md             # ← из МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ §X
│   └── ...
[... аналогично для 02/03/04 ...]

## Сводная таблица новых файлов

| Путь | Вход (из МОДУЛЬ-КП §X) | Назначение | Размер (rows) | Приоритет |
|---|---|---|---|---|
| 01_КП/00-spr/00-glossary.md | §13 Глоссарий | Термины модуля КП | ~50 строк | 🟢 P2 |
| 01_КП/01-shablon/01-konstruktor.md | §4 Витрина | Конструктор шаблонов | ~100-150 строк | 🔴 P0 |
| ... | ... | ... | ... | ... |

## Существующие файлы (НЕ создаются, только учтены)

| Путь | Текущее состояние |
|---|---|
| 01_КП/00-spr/00-otkrytye-voprosy.md | ✅ 5 baseline OQ-001..005 (НЕ ТРОГАТЬ) |
| 01_КП/01-shablon/00-README.md | ✅ STUB (готов как слот) |
| 01_КП/02-konstruktor-kp/00-README.md | ✅ STUB |
| 01_КП/03-zhiznennyj-cikl/00-README.md | ✅ STUB |
| 01_КП/04-pravila/00-README.md | ✅ STUB |
```

**Если Архитектор пишет содержимое файлов** — СТОП. Это нарушение Ограничения #1. Попросить перегенерировать без содержимого.

## 4. Пост-обработка output'а Архитектора (в ЭТОЙ сессии)

Когда Архитектор вернёт ответ:

### Шаг A: Создать реальные файлы через `mkdir` + `touch` (или `write_file` с STUB-маркером)

```bash
# Создать директории (Архитектор должен был уже назвать каждую)
mkdir -p '01_КП/<sub-folder>/<sub-sub-folder>'

# Touch пустые STUB-файлы (по количеству из output'а Архитектора)
touch '01_КП/00-spr/00-glossary.md'
# ... и т.д.
```

Каждый новый STUB должен содержать:
- Заголовок `# <file-name>.md — <назначение 1 предложение>`
- `> **Назначение.** 1-2 строки.`
- Пустой `## Содержание` блок (заполнят Аналитик/Моделировщик/UX).

### Шаг B: Записать PSL-004 в `PROJECT-STATE-LOG.md`

```markdown
### PSL-004 — Архитектор завершил первый прогон (декомпозиция МОДУЛЬ-КП)

| Поле | Значение |
|---|---|
| Дата | <YYYY-MM-DD> |
| ID | PSL-004 |
| Тип | structure |
| Модуль | КП |
| Автор | PO / Архитектор / Claude Sonnet-4.5 |
| Связанные OQ | OQ-001, OQ-002, OQ-003 (5 baseline OQ созданы в PSL-002) |
| Описание | Архитектор декомпозировал МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md (945 строк) в дерево из N новых .md файлов по 5 папкам (00-spr → 01-shablon → 02-konstruktor-kp → 03-zhiznennyj-cikl → 04-pravila). Сохранены все 5 baseline OQ-001..005 как существующие. Создано N STUB-файлов под бизнес-аналитика. |
| Причина | Завершение первого прогона Pipeline v6. Подготовка к Phase 1 Bootstrap — следующий запуск Бизнес-аналитика для заполнения правил в созданных STUB'ах. |
| Затронутые файлы | - 01_КП/<sub>/<file>.md — N новых STUB<br>- PROJECT-STATE-LOG.md — эта запись |
```

### Шаг C: Коммит + push в `origin/main`

```bash
git add .
git commit -m "chore(docs): Architect first run — KP module decomposition (PSL-004)

- N новых STUB .md файлов по спецификации Архитектора
- PROJECT-STATE-LOG.md: PSL-004 зафиксирована

Ref: PSL-001, PSL-002, PSL-003"
git push origin main
```

### Шаг D: Запустить Бизнес-аналитика

Следующая роль по [`AGENT-ROLES.md` §3 Pipeline loop](../../AGENT-ROLES.md) — **Бизнес-аналитик** (промпт в [`AGENT-PROMPTS.md` §2](../../AGENT-PROMPTS.md)). Аналитик НЕ запускает LLM, а берёт дерево Архитектора и правит STUB-файлы (добавляет `**Правило N.**`...).

## 5. Контроль качества запуска

✅ **CHECK 1: OOM не произошёл.** Если Архитектор вернул обрезанный/пустой ответ — файл МОДУЛЬ-КП слишком большой, split на 2 части или attach меньше (только AGENT-* + МОДУЛЬ-КП + 00-otkrytye).

✅ **CHECK 2: Содержимое НЕ написано.** Если Архитектор начал писать `## Описание поля` или `## Правило 1.` — это нарушение Ограничения #1, попросить перегенерировать.

✅ **CHECK 3: Существующие файлы учтены.** В output'е Архитектора ОБЯЗАТЕЛЬНО упоминаются 5 baseline OQ и 4 STUB README. Если нет — вернуть на доработку.

✅ **CHECK 4: Hard limit соблюдён.** Ни один предложенный файл > 400 строк. Если есть — Архитектор должен сам предложить разбивку.
