> **Назначение.** Готовый copy-paste пакет для запуска следующего агента (Бизнес-аналитика) в **новой сессии Codebuff**. Подготовлен 2026-06-26 после успешного первого прогона Архитектора: коммит `94fc5a6` запушен в `origin/main` (PSL-004, 20 STUB `.md` созданы из исходного 945-строчного `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`, удалённого через `git rm`). **Это Pipeline v6 прогон Аналитика 1/5** (не путать с Phase 1 Bootstrap миграций Prisma — это другая вещь, см. `99_Справочники/OPEN-QUESTIONS-MASTER.md` §5).
>
> **Когда использовать.** Сразу ПОСЛЕ прочтения этого файла — открой `codebuff` (новый чат) и скопируй `## Промпт для Codebuff` ниже в первую строку + прикрепи файлы из `## Файлы для attach`.

## 0. Что должно произойти

1. **Открыть новую сессию Codebuff** (CLI: `codebuff` → New Chat).
2. **Прикрепить 11 файлов** из секции `## Файлы для attach` ниже (порядок важен).
3. **Вставить copy-paste промпт** из секции `## Промпт для Codebuff` (между ```text ... ```).
4. **Отправить** — Бизнес-аналитик начнёт работу.
5. **Вернуться в ЭТУ сессию** с output'ом Аналитика → записать **PSL-006** (Pipeline v6 прогон Аналитика Run 1) + commit + push.
6. **Запустить QA-валидатора** (петля по Правилу 3.1 AGENT-ROLES.md §3 ОБЯЗАТЕЛЬНАЯ) → после QA → **Run 2 Аналитика** (state-машина).

> **Почему именно эта роль сейчас.** Per [`AGENT-ROLES.md` §3 Pipeline](../../AGENT-ROLES.md): «Architect → BusinessAnalyst → DataModeler → UX → TechWriter → QA → Coordinator». Бизнес-аналитик ПЕРЕД Моделировщиком и UX потому что именно он задаёт **пронумерованные правила** (`Правило 1.`, `Правило 2.`) — это контракт, на который Моделировщик пишет поля, а UX рисует состояния.

## 6. Контроль качества запуска

✅ **CHECK 1: OOM не произошёл.** Если Аналитик вернул обрезанный/пустой ответ — split на 2 прогона (только RBAC в Run 1a, только biznes-pravila в Run 1b).

✅ **CHECK 2: Только 3 STUB из Пакета C изменены.** Если в diff попали 04-glossary.md, 02-knopki.md и т.п. — Аналитик вышел за scope Run 1. Попросить вернуться к фокусу Run 1.

✅ **CHECK 3: Нумерация правил по П6.** Каждое правило = `**Правило N.**`. Без `1)` или `№` — анти-паттерн.

✅ **CHECK 4: Hard limit соблюдён.** `04-rbac.md` + `04-biznes-pravila.md` (или split-вариант) — каждый ≤ 400 строк. Если нет — split обязателен.

✅ **CHECK 5: Ссылки на СПОР и OQ присутствуют.** Каждое правило с источником имеет явную ссылку `_Источник: СПОР-N (24.06.2026)_` или `_Источник: OQ-NNN_`. Без источника = правило не проверяемо.

✅ **CHECK 6: ⚠️ STUB-маркер заменён на содержание.** Если в изменённых файлах всё ещё остался маркер `⚠️ **STUB** (создан Архитектором…)` — Аналитик не удалил артефакт статуса. Заменить на обычный текст раздела «0. Контекст».
