# `01_КП/00-spr/00-otkrytye-voprosy.md` — Открытые вопросы модуля КП

> **Назначение.** Локальный реестр **дыр в покрытии** модуля «Коммерческое предложение». Используется агентами всех ролей для фиксации того, что в текущих документах модуля КП отсутствует, противоречит или недостаточно определено.
>
> **Канонический источник формата:** [`AGENT-ENTRYPOINT.md §4.6`](../../AGENT-ENTRYPOINT.md#46-шаблон-фиксации-дыры-для-00-otkrytye-voprosymd-и-open-questions-mastermd). Ниже — копия схемы для самодостаточности модуля.
>
> **Граница:** этот файл — **только для модуля КП** и **только для гранулярных implementation-dыр** (отсутствующее правило / противоречие / undefined behavior / битая ссылка / terminology drift).
>
> **Когда эскалировать в [`99_Справочники/OPEN-QUESTIONS-MASTER.md`](../../99_Справочники/OPEN-QUESTIONS-MASTER.md):** если дыра требует выбора PO из вариантов A/B/C (бизнес-решение). Тогда в MASTER создаётся новый Q-NNN, а здесь OQ помечается `🔺 Переведено в MASTER Q-NNN`. Полная архитектура 3-уровневого реестра — в [`PROJECT-STATE-LOG.md` §2](../../PROJECT-STATE-LOG.md).
>
> Подробнее об этих правилах — в `AGENT-ENTRYPOINT.md §4.4` (критерий «локальная vs проектная» дыра).

---

## 0. Схема записи OQ

Каждая запись **обязана** иметь следующие поля. Формат строго табличный (для машиночитаемости):

| Поле | Описание |
|---|---|
| **ID** | `OQ-XXX` (порядковый номер в пределах файла, монотонно растёт с 001; НЕ пересекается с ID в `OPEN-QUESTIONS-MASTER.md`) |
| **Модуль** | здесь всегда `КП` (файл локальный; для Универсально — другой файл) |
| **Дата обнаружения** | `YYYY-MM-DD` |
| **Обнаружил** | роль агента, обнаружившего дыру: `Архитектор` / `Бизнес-аналитик` / `Моделировщик` / `UX-проектировщик` / `QA-валидатор` / `Тех.писатель` / `Координатор` |
| **Тип** | `missing_rule` / `contradiction` / `undefined_behavior` / `broken_link` / `terminology_drift` |
| **Приоритет** | `🔴 P0` (блокер MVP) / `🟡 P1` (важно) / `🟢 P2` (мелочь) |
| **Что хотел сказать автор** | контекст — почему дыра вообще возникла |
| **Что реально отсутствует/противоречит** | конкретное описание проблемы (с цитатой из файла если применимо) |
| **Минимальное исправление** | предложение как починить (или «открыто для PO») |
| **Статус** | `🔵 OPEN` (активна) / `🟡 DEFERRED` (отложено до следующей итерации) / `🟢 RESOLVED` (закрыто — укажи дату и PSL-номер из PROJECT-STATE-LOG) |

**Правила ведения:**
- ID монотонный, никогда не переиспользовать.
- Новые записи — сверху.
- При закрытии добавляется строка «Закрыто: PSL-NNN от YYYY-MM-DD» в поле «Минимальное исправление».
- Если дыра оказалась межмодульной — переносится в `OPEN-QUESTIONS-MASTER.md` со ссылкой сюда.

### OQ-002 — Что произойдёт если менеджер нажал «Создать КП» без выбранного шаблона

| Поле | Значение |
|---|---|
| **ID** | OQ-002 |
| **Модуль** | КП |
| **Дата обнаружения** | 2026-06-26 |
| **Обнаружил** | QA-валидатор |
| **Тип** | `undefined_behavior` |
| **Приоритет** | 🔴 P0 |
| **Что хотел сказать автор** | Кнопка «Создать КП» в витрине — но не описано обязательность выбора шаблона. По бизнес-логике шаблон обязателен (иначе документ не имеет макета). |
| **Что реально отсутствует** | В `01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md §4` (Витрина и Корзина) не сказано: (а) что кнопка «Создать КП» disabled пока шаблон не выбран, (б) какой fallback если единственный шаблон удалён, (в) может ли быть «черновик без шаблона» (авто-выбор шаблона по умолчанию). |
| **Минимальное исправление** | Добавить правило в `01_КП/02-konstruktor-kp/02-knopki.md`: **Правило N.** Кнопка «Создать КП» доступна только если `proposal.templateId != null`. Дополнительно: правило «если в системе 1 шаблон — он preselected автоматически», правило «если 0 шаблонов — все кнопки скрыты и показано сообщение-инструкция админу». |
| **Статус** | 🔵 OPEN |

### OQ-003 — Формула «итого» в таблице позиций описана частично

| Поле | Значение |
|---|---|
| **ID** | OQ-003 |
| **Модуль** | КП |

| **Тип** | `contradiction` |
| **Приоритет** | 🟡 P1 |

Явно описать округление (`Math.round копейки`), границы (markup ≥ -100%, discount ∈ [0, 100%]). |

### OQ-004 — Нет правила удаления КП

| Поле | Значение |
|---|---|
| **ID** | OQ-004 |
| **Модуль** | КП |
| **Дата обнаружения** | 2026-06-26 |
| **Обнаружил** | Бизнес-аналитик |
| **Тип** | `missing_rule` |
| **Приоритет** | 🟡 P1 |
| **Что хотел сказать автор** | У КП есть статусы «Черновик / Новая версия / Отправлено / Принято / Оплачено / Конвертировано / Архив / Отменено». Но непонятно: могут ли все статусы быть удалены, или только черновики, или вообще удаление запрещено (вместо него — перевод в Архив)? |
| **Что реально отсутствует** | В `01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md §2` (жизненный цикл) про удаление ни слова. В RBAC (`99_Справочники/RBAC-MATRIX.md`) действие `deleteProposal` присутствует, но без привязки к статусу. |
| **Минимальное исправление** | Добавить в `01_КП/03-zhiznennyj-cikl/03-statusy.md` правило: **Правило N.** Удаление КП = soft-delete (флаг `isDeleted`). Допустимо только для статусов `DRAFT`, `CANCELLED`, `EXPIRED`. Для остальных — кнопка скрыта, для перевода в Архив — отдельное действие. |
| **Статус** | 🔵 OPEN |

| **Минимальное исправление** | Добавить в `99_Справочники/GLOSSARY-MASTER.md` раздел: «Картотека сделки — UX-контейнер: клиент + все документы его цикла (КП → Договор → ЗК → Склад → Финансы) в одной вкладке. Доступна всем ролям с фильтром по своим/чужим через RBAC». |
