## 2. «Первая критика» — что было сделано до нас

### 2.1 Архитектурная история v3 → v4 → v5

| Версия | Название | Стек | Итог |
|---|---|---|---|
| **v3.0** | KPPDF 3.0 | Angular 21 + Express.js + MongoDB | Сильный редактор КП с А4 на экране + drag-and-drop. **Отклонено**: Angular менее актуален, MongoDB без ACID для финансов. |
| **v4.0** | KPPDF 4.0 «СпортИН-Юг» | Angular 21 + Express 5 + MongoDB 8 | Универсальное ядро, конструктор документов, 443 теста, Docker Compose. **Отклонено**: тот же стек, не повторяем. |
| **v5.0** | **KPPDF CRM v5.0** | **Next.js 16.2.9 + React 19 + Prisma 7.8 + PostgreSQL 16** | Актуальный baseline. Полный цикл: Товары → КП → Договор → Производство → Склад → Финансы. 47 моделей, 89 API, 88 unit-тестов, Gantt, RBAC на 6 ролей. **Это фундамент для v6.** |

### 2.2 Что подтвердилось в v5 (берём в v6 без изменений)

- **Модульная структура БД (47 моделей Prisma)** — Proposal → Contract → ProductionOrder с автоконвертацией при `status='paid'`. Это основной architectural pattern.
- **89 API endpoints** — покрывают весь бизнес-цикл, проверены 88 Vitest unit-тестами.
- **JWT через HttpOnly cookies** — рабочая схема без тяжёлых библиотек (`src/lib/jwt.ts` после cycle 39 refactor).
- **`env.ts` как единый источник правды** (cycle 40) — `isProd / isDev / baseUrl` экспортируются оттуда, ESLint блокирует прямые обращения к `process.env.NODE_ENV` в коде.
- **Disk + Server API Routes** в одном проекте (Next.js 16 App Router) — даёт серверные компоненты и Server Actions без отдельного backend-сервиса.

### 2.3 Что НЕ подтвердилось в v5 (пересматриваем в v6)

## 4. Мои виденья (что добавляем поверх baseline v5)

### 4.1 State management: Zustand + TanStack Query (двухслойная модель)

> **Рекомендация: A (принять двухслойную модель).**

- **Zustand (как в v5)** — для **local UI state**: режим руководителя (правка D), toggle предпросмотра А4, открытые dropdown-ы, фокусы в форме. Уже используется в v5 — `persist` middleware добавим **для autosave КП (правка C)**.
- **TanStack Query (React Query) v5** — для **server state**: кэш списков (КП, Договоры, ЗК), синхронизация данных, автоинвалидация после мутаций, обработка loading/error states. В v5 такого нет — данные подгружаются напрямую через `fetch` + manual cache. Это утомительно для 89 API и ухудшает UX.
- **Альтернатива (отклонена)**: только Zustand + прямые fetch (как в v5). Проще на старте, но нет кэша → каждый переход на `/proposals` рефетчит все КП. Для менеджера с 100+ КП в работе — медленно.

**Где:**
- TanStack Query: для всех API списков и patch/mutation в Next.js Client Components.
- Zustand: для локальных UI toggle-ов и persist (autosave черновика КП в localStorage).
- Zustand + persist для autosave: middleware пишем сами (30 строк кода), не берём готовую библиотеку поверх.

### 4.2 ORM: Prisma 7 vs Drizzle

> **Рекомендация: A (Prisma 7, как в v5).**

- **Prisma 7 (как в v5)** — проверенный, **47 моделей уже написаны**, **88 тестов** уже проходят, ERRATA про PrismaPg driver adapter уже учтена. Миграции по правкам A-G — это **3-5 schema-изменений**, не переписывание.
- **Drizzle ORM** — ближе к SQL, type-safe, быстрее на холодных стартах. Но: **придётся переписать 47 моделей** + 89 API handlers + 88 тестов. Это **+2-3 месяца работы** или полный отказ от baseline v5.
- **Вердикт**: преимущества Drizzle не окупают стоимости миграции для v6. Рассматриваем Drizzle только если v6 окажется провальным и придётся делать v7 с нуля на другом ORM.

### 4.3 API: Next.js Route Handlers + Zod 4 vs tRPC

> **Рекомендация: A (Next.js Route Handlers + Zod 4, как в v5).**

- **Next.js API Routes + Zod (как в v5)** — проверенная схема. Zod-схемы уже в `src/lib/validations/`, 89 endpoint-ов. RBAC проверки в самих route.ts.
- **tRPC** — type-safe, минимум boilerplate, автогенерация клиентских хуков. **Но**: tRPC ≠ REST. Если завтра появится мобильный клиент или интеграция с DaData/1С/внешним ERP — tRPC не справится, нужно будет дублировать в REST. Это **архитектурный долг**.
- **Альтернатива (отклонена)**: tRPC + параллельный REST gateway. Делать то и то — удвоение работы.
- **Вердикт**: Next.js Route Handlers + Zod остаётся. Для type-safe клиентских хуков используем **TanStack Query + typed fetch wrapper** (10 строк кода).

### 4.4 Forms: react-hook-form + Zod resolver vs native React 19

> **Рекомендация: B (react-hook-form + Zod resolver).**

- **react-hook-form** — минимум ре-рендеров, удобный API для больших форм (КП-редактор — это 50+ полей в одной форме!), валидация через `@hookform/resolvers/zod` (одни и те же Zod-схемы на клиенте и сервере).
- **Native React 19 useActionState + form actions** — современно, но не оптимально для **сложной клиентской интерактивности** (инлайн-редактирование ячеек таблицы, динамический preview А4). React 19 actions хороши для простых форм (логин, создание), но не для главного экрана КП.
- **Вердикт**: react-hook-form для КП-редактора, native React 19 actions для простых форм (логин, search).

### 4.5 File / Image processing: sharp

> **Рекомендация: A (sharp в API Route).**

- **sharp** — стандарт для Node.js, нативно обрабатывает JPEG/PNG/WebP, серверный CPU-only ресайз. Размер файла сжимается с 5MB до 200KB без видимой потери качества.
- **Альтернатива (отклонена)**: client-side кроппер (как в МОДУЛЬ-доке v3 раздел 10 — фото-редактор с кроппером). По правке G **отменён** в v1 как scope creep: 2-3 дня разработки, баги на мобилах, проблемы с памятью на больших фото, низкая бизнес-ценность. Если в v2 потребуется — отдельная задача, готовые компоненты (react-image-crop) — без велосипеда.
- **Вердикт**: только `sharp` для серверного сжатия. Client-side кроппер — v2 при реальном спросе. (Также тяжёлый Puppeteer для PDF-генерации отложен на v2, сжатие и генерация в v1 остаются максимально легковесными — `sharp` + `jsPDF`).

### 4.6 Package manager: pnpm vs npm vs bun

> **Рекомендация: B (pnpm).**

- **npm (как в v5)** — проверен, но медленный: установка зависимостей для нового разработчика — 5 минут. `node_modules` дублирует пакеты между проектами.
- **pnpm** — в 2-3 раза быстрее **установка** (content-addressable storage), экономия диска (symlinks вместо копий), monorepo-friendly. Совместим с Next.js 16 и всеми зависимостями v5.
- **bun** — самый быстрый по бенчмаркам, но: **нестабилен** для production с Next.js + Prisma (issue с Postgres driver в некоторых конфигурациях), маленькое сообщество.
- **Вердикт**: pnpm. Миграция с npm → pnpm = смена `package-lock.json` на `pnpm-lock.yaml` + `npm install` → `pnpm install`. Занимает 1 день. Команда привыкает за неделю. Совместимость с v5: package.json остаётся, lockfile другой — не нужно трогать код.

### 4.7 Тесты: Vitest + Playwright (как в v5)

> **Рекомендация: A (оставить как есть).**

- **Vitest 4** (88 unit-тестов в v5) — быстро, ESM-native, совместим с TypeScript strict.
- **Playwright** (e2e smoke в v5) — для критических путей (login → создать КП → принять → оплатить → проверить ЗК).
- **Вердикт**: расширяем существующие тесты, не вводим новые фреймворки. Миграция на Vitest 5 или Bun test — **отложено до v2**.

### 4.8 Pre-commit quality gates: Husky + lint-staged + commitlint

> **Рекомендация: A (добавить).**

- **Husky** — git hooks в `package.json`. Pre-commit: `pnpm tsc --noEmit` + `pnpm eslint` + `pnpm prettier --check`. Pre-push: `pnpm vitest run`.
- **lint-staged** — запускает check только на staged файлах (быстро, ~5-10 сек).
- **commitlint** — conventional commits (`feat:`, `fix:`, `docs:`) — даёт автогенерацию CHANGELOG.
- **Вердикт**: добавляем с Phase 1. Без этого легко закоммитить `tsc` error в main.

### 4.9 Живой предпросмотр А4 = HTML-first подход

> **Рекомендация: A (строгий HTML/CSS в центральной зоне).**

- **Идея**: Не генерировать PDF на сервере при каждом изменении. Использовать строгий HTML/CSS в точном размере A4 (210×297mm) в центральной зоне редактора.
- **Стили**: Единый CSS для экрана и для печатной формы (`@media print`, `@page { margin: 0; size: A4; }`).
- **Редактирование**: Для инлайн-редактирования таблицы используем `contenteditable` или кастомный React-компонент с `onBlur` (чтобы клик в ячейку делал её input-ом прямо поверх «листа»).
- **Генерация PDF**: При нажатии «Печать/PDF» можно отправлять этот же HTML на серверный рендерер (Puppeteer в v2) или использовать jsPDF/браузерную печать (в v1). Это обеспечивает 100% совпадение экрана и PDF — то, что критик называет «идеальным случаем».

## 6. Альтернативы, отклонённые с обоснованием

### 6.1 Backend stacks

| Альтернатива | Почему отклонена |
|---|---|
| **T3 Stack (tRPC + NextAuth + Prisma + Tailwind)** | tRPC переусложняет API для внешних систем (мобильный клиент, DaData/1С). NextAuth избыточен для простой JWT-cookies схемы. |
| **Remix / React Router** | Команда наработала baseline на Next.js. Миграция = выбросить v5 целиком. Remix интересный, но дороже. |
| **SvelteKit** | Команда не работала с Svelte. Экосистема Prisma для Svelte слабее. Неоправданный риск для v1. |
| **Bun + Elysia** | Bun ещё ловит баги с Next.js + Prisma на prod (issue с native bindings). Elysia не тёстирован в проде для нашего масштаба. Bun оставляем как потенциал v2. |
| **NestJS (standalone backend)** + Express/Fastify | Теряем Server Components и RSC из Next.js. Нужен отдельный CORS-слой, отдельный dev-server. Не окупается для v1. |

### 6.2 ORM

| Альтернатива | Почему отклонена |
|---|---|
| **Drizzle ORM** | Придётся переписать 47 моделей с нуля + 89 API + 88 тестов. +2-3 месяца работы. Преимущества (ближе к SQL, легче миграции) не окупаются. |
| **TypeORM** | Зрелая но декоратор-heavy, медленнее Prisma, хуже для greenfield проекта. |
| **Native SQL + Knex.js** | Слишком много boilerplate, особенно для typed relations (FK, joins, soft-delete). Prisma schema-as-code выразительнее. |
| **PostgreSQL + plpgsql + прямые SQL хендлеры** | Потеряем type-safety на клиенте, автогенерацию клиентских типов. Подходит для high-performance data pipelines, но не для CRUD приложения. |

### 6.3 Database

| Альтернатива | Почему отклонена |
|---|---|
| **MongoDB** | Отсутствие полноценного ACID для модуля Финансы (брак архитектуры v3/v4). Сложнее с FK constraints, rollback. |
| **MySQL / MariaDB** | Работают, но PostgreSQL даёт мощные extensions (PostGIS при будущей опции «карта сделок»), JSONB, advanced indexes (GIN/GIST). |
| **Supabase / Firebase (managed)** | Vendor lock-in, нет миграций в привычном смысле, сложнее self-host. Не оправдано для self-hosted Synology DSM. |

### 6.4 Frontend / forms / state

| Альтернатива | Почему отклонена |
|---|---|
| **Redux Toolkit** | Слишком heavy для 47-модельного CRUD. Zustand + TanStack Query проще и не уступает. |
| **MobX** | Команда не работала; observable паттерн требует переобучения. Zustand и так есть в v5. |
| **Recoil / Jotai** | Атомарный подход хорош для гранулярного UI state, но для CRUD приложения Zustand проще. |
| **Formik** | Старее react-hook-form, больше boilerplate. RHF де-факто стандарт 2026. |
| **Native React 19 useActionState повсеместно** | Хорош для простых форм, но для КП-редактора с 50+ полей и инлайн-редактированием таблицы — неудобно. |

### 6.5 Auth

| Альтернатива | Почему отклонена |
|---|---|
| **NextAuth.js / Auth.js** | Переусложнён для нашей простой JWT-cookies схемы. Свой `jose` (как v5 cycle 39) — 50 строк кода, без зависимости на чужую ORM-specific auth. |
| **Clerk / Auth0** | Vendor lock-in, monthly fees. Для self-hosted Synology DSM — лишнее. |
| **Session cookies вместо JWT** | Сессии требуют хранения на сервере (Redis или БД) + sticky sessions на multi-pod. JWT stateless и достаточен для нашего масштаба. |

### 6.6 PDF

| Альтернатива | Почему отклонена |
|---|---|
| **Puppeteer (Chromium headless)** | **Отклонён для v1 / в планах на v2.** Идеальная точность для длинных/сложных КП, но 200MB+ бинарник Chromium → тяжёлый Docker image. В v1 обходимся HTML-печатью / jsPDF (см. OQ-8). |
| **react-pdf / @react-pdf/renderer** | Server-side React-компоненты для PDF. Интересно, но v5 уже использует jsPDF и это работает — переключение = 2-3 дня + риск regressions. |
| **PDFKit** | Слишком low-level: ручное позиционирование текста. jsPDF поверх него даёт удобный API. |

### 6.7 Прочие отклонения

| Альтернатива | Почему отклонена |
|---|---|
| **GraphQL** | Не окупается для 89 простых CRUD endpoints. REST + Zod + TanStack Query проще и гибче. |
| **Webpack (custom config)** вместо Turbopack из Next.js | Next.js 16 поддерживает и Turbopack, и Webpack. Turbopack быстрее в dev — оставляем дефолт. |
| **Storybook для дизайн-системы компонентов** | Полезно, но отложено до v2 (сейчас компонентов немного, разработка «по месту» достаточна — см. OQ-2). |

### 6.8 Что отклонили по итогам критики техлида

- **Node.js + NestJS как отдельный backend** — Next.js 16 (App Router + Route Handlers + Server Actions) уже содержит встроенный backend, функционально аналог NestJS. Выделять NestJS как отдельный сервис — это CORS, дублирование типов и +2-3 недели для команды 1-3 разработчиков. Критик не учел нашу базу v5.
- **Vue / Svelte как замена React** — Команда уже наработала baseline v5 на React. Переход сбросит 16 циклов разработки.
- **Отказ от Next.js** — критик советует «без Next.js всё behind auth». Мы используем Next.js не столько ради SSR, сколько как интегрированный React-фреймворк (Server Actions + API Routes + RSC + zero-config deploy) — это снижает общую сложность, не увеличивает её.
- **Puppeteer в v1** (см. таблицу выше и раздел 6.6) — тяжёл для MVP, отложен.

### 6.9 Что подтвердил техлид (<<принято без изменений>>)

- **Монолит с модульной структурой внутри** — у нас уже так (5 модулей внутри одного Next.js приложения).
- **PostgreSQL** — baseline ✅.
- **React + TypeScript** — baseline ✅.
- **Docker + Docker Compose** — уже в предписании.
- **Что НЕ нужно в v1** совпадает с нашим: ❌ GraphQL / ❌ Redis / ❌ Elasticsearch / ❌ Kubernetes / ❌ WebSocket (мы уже заменили на История комментариев по правке F).

### 6.10 Что было бы хорошо, но мы не выбрали в v1

- **Express + tRPC (standalone)** — типобезопасный API, но + абстракция для не-Next клиентов (мобильный, DaData/1С). Next.js Route Handlers + Zod проще.
- **Свой CDN для статических файлов** — пока Synology NAS + nginx достаточно.
- **Embed-версия КП для отправки клиенту через web-link** — отдельная задача v2.
- **Multi-tenancy (несколько компаний в одной БД)** — пока single-tenant.
