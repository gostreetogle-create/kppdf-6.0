# RBAC-MATRIX.md — Сводная матрица прав доступа v6

> **Дата:** 2026-06-24.
> **Назначение:** Единая матрица «кто-что-может» по всем модулям проекта. Синтезирована из RBAC-таблиц, разбросанных по каждому МОДУЛЬ-доку.
>
> **Структура:** 7 ролей × ~30 действий, сгруппированных по модулям.
>
> **Условные обозначения:**
> - ✅ = разрешено без ограничений
> - ❌ = запрещено
> - ⚠️ = разрешено **с ограничением** (пояснение в скобках)
> - 🔒 = разрешено, но требует **двухфакторного подтверждения** (SMS / email-код)
> - 🔵 = только просмотр (read-only)

---

## 0. TL;DR — кто что может (кратко)

| Роль | Главное |
|---|---|
| **admin** | Полный доступ ко всему. Настройка RBAC, управление справочниками, импорт данных. |
| **director** | Стратегический обзор: видит всё включая себестоимость/margin, утверждает крупные списания и договоры, расторгает контракты. |
| **accountant** | Финансовый контроль: видит `costPrice`, регистрирует платежи, утверждает списания до 5000 ₽, маржинальные отчёты. Не работает с КП напрямую. |
| **manager** | Клиентская работа: создаёт КП, конвертирует в Договор, ставит «Оплачено». **НЕ видит себестоимость** в КП-редакторе. |
| **production** | Производство: ведёт ЗК, распределяет задачи по цехам, отгружает по своим ЗК. Не работает с КП/Финансами. |
| **storekeeper** | Склад: принимает приходы, оформляет отгрузки/списания, контролирует остатки. Главный пользователь `/warehouse`. |
| **viewer** | Только просмотр (аудиторы, наблюдатели). |

## 2. Пояснения по критическим ограничениям

### 2.1 Ограничения «свой»

Касается ролей `manager`, `production`. «Свой» = документ, где:
- `createdById = user` (менеджер создал КП)
- `responsibleUserId = user` (он ответственный за ЗК)
- `assigneesProductionOrder ∈ [user]` (он в списке ответственных)
- `КП/Договор/ЗК` по которым менеджер был инициатором.

**Это защищает от ситуации: «Менеджер Петров правит КП менеджера Сидорова»** — без явного ограничения.

### 2.2 Ограничения по сумме/порогу

`accountant` ↔ `WriteOffAct.approve`: до 5000 ₽ — accountant, выше — director. **Это защищает бухгалтера от крупных финансовых потерь.**

### 2.3 Двухфакторное подтверждение (🔒)

Где применяется:
- Загрузка логотипа/печати организации (`admin`, `director`) — через SMS / email-код на bound-аккаунт.

**Это защита фирменного стиля**: менеджер не может случайно подменить логотип клиента или нашу печать.

### 2.4 Только просмотр (🔵)

Где применяется:
- `accountant` редактирует финансовые поля товаров: только для целей отчётности (см. `организацию финансовых полей`).

### 2.5 RBAC в многоступенчатых процессах

**Annul payment** (аннулирование платежа) — **двухступенчатое**:
1. `accountant` готовит (вводит причину).
2. `director` одобряет + добавляет обоснование в `notes`.

Это защита от одностороннего списания денег.

## 3. Инварианты прав доступа (бизнес-правила)

### 3.1 Менеджер НЕ видит себестоимость

Это **GAP-014**, согласованный через сводный трекер:
- В редакторе КП поле `costPrice` **не отображается** для роли `manager` (ни в витрине товара, ни в Пакетe документов).
- В печатной форме (PDF для клиента) себестоимость **никогда не выводится** — это правило жёсткое независимо от роли.
- В режиме анализа маржи (отдельный экран) менеджер **не имеет доступа**.

### 3.2 История комментариев Картотеки — только director + admin

Это **GAP-013** (сводный):
- Роль `manager` (и потенциальные будущие роли `senior_manager` / `junior_manager`) **НЕ имеют доступа к комментариям Картотеки сделки** в v1.
- Комментарии Картотеки — внутренняя переписка **для director + admin**.
- Менеджеры могут комментировать свой КП (отдельная сущность `Comment` с прямым FK на КП — НЕ Картотека).

### 3.3 Финансово-критичные поля справочников — отдельная категория

Это **GAP-015**:
- **Финансовые поля** (`ИНН`, `КПП`, `ОГРН`, банк, БИК, р/с, `basePrice`, `costPrice`) — ТОЛЬКО `admin`/`director`/`accountant`.
- **Нефинансовые поля** (ФИО контакта, адрес, телефон, email, описание, фото, категория товара, тип позиции) — ВСЕ менеджеры (`admin`, `director`, `accountant`, `senior`, `manager`, `junior`, `viewer` — но viewer read-only).

### 3.4 Защита печати и логотипа

Это **GAP-016**:
- Загрузка/замена файлов «Логотип» и «Печать» — ТОЛЬКО `admin`/`director` с двухфакторным подтверждением.
- Менеджер **видит** уже загруженные файлы (автоподстановка в шапку КП), но НЕ может залить новый.

## 5. Граничные кейсы

### 5.1 Один пользователь = несколько ролей

В v1 **один пользователь может иметь одну роль**. Это упрощение. Multi-role-per-user запланировано в v2 (например, `Ivanov = director + admin`).

### 5.2 Новые роли в v2

Согласовано (но отложено до v2):
- `senior_manager`, `junior_manager` — подвиды manager с разными правами.
- `purchase_manager` — отдельная роль для закупщика.
- `accountant_senior` — расширенный бухгалтер с правом approve крупных списаний.
- `production_master` — мастер цеха (подвид `production`).

### 5.3 Multi-tenancy

**v1 = single-tenant** (одна компания). Multi-tenancy изолированных БД — v2.
