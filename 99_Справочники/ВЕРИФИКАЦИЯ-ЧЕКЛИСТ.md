> **Назначение.** Этот документ — **каркас для «лёгкой теоретической симуляции»** будущей системы, которая сделан из обычных .md-файлов и ИИ-агента. Цель — найти логические дыры на стыках модулей **ДО** начала кодинга, чтобы не получилось: «доделали модуль КП → дошли до Производства → поняли, что в Товарах нужно было ещё 5 полей».
>
> **Статус.** ✅ Чеклист готов к прогону. Все 73 пункта проверки сформулированы, шаблоны промптов подготовлены, формат логирования gap-карточек задан.
>
> **Как пользоваться.** Открой новый чат с ИИ (или Obsidian + локальную LLM), загрузи **все** .md-документы проекта (МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md, OPEN-QUESTIONS-LEGACY.md, README, archive/ если нужно), дай ему роль «системы» по промпту из раздела 7 этого файла. Потом шаг за шагом прогоняй пункты V-001…V-073 из разделов 4–6. Каждый найденный пробел оформляй как gap-карточку (раздел 8). Когда будет набрано достаточно — отдай gap-карточки «аудитору» (раздел 7, второй промпт) и/или сразу продакт-оунеру для решения.
>
> **Этот файл — не модуль-документ.** Не описывает поведение системы. Это рабочий инструмент для **теоретической верификации** архитектуры.

## 2. Оси проверки (5 категорий стыков)

Каждый проверочный пункт ниже лежит в одной из этих осей:

| Ось | Что ищем | Почему важно |
|---|---|---|
| **A. Общие сущности** | Поля и типы, которые несколько модулей читают/пишут (Product, Organization, Template) | Один неверный тип — ломает 2-3 модуля сразу |
| **B. События и переходы** | Что происходит на границе статусов: KP paid → PO created; KP converted → Contract | Не описано = дыра в процессе |
| **C. Роли и права (RBAC)** | Кто что видит, кто что делает, кто что подписывает | Размытые права = инциденты |
| **D. UI и навигация** | Кросс-модульная навигация: из КП → в Договор → в Производство | Плохая связность = менеджер теряется |
| **E. Данные и edge cases** | Нестандартные сценарии: валюта, истёкший срок, удалённые сущности | Их ВСЕГДА забывают |

**Нумерация пунктов:** `V-AAB`，где AA = номер категории (01–05), B = порядковый номер в категории. Примеры: `V-010` (сущность №10), `V-205` (событие №05 в категории 2).

## 3. Общие правила для всех проверок и терминология

### 3.1 Терминология

- **Симулятор** → ИИ в роли «системы» (см. промпт в разделе 7).
- **Аудитор** → Второй ИИ / другая модель (см. промпт в разделе 7).
- **Пробел (Gap)** → То, что не определено ни в одном документе. Оформляется как gap-карточка (раздел 8).
- **Противоречие** → Два места в документах говорят разное.
- **`[ПРОБЕЛ]`** → Маркер, который симулятор ОБЯЗАН использовать в ответе, когда не может ответить строго по документам.

### 3.2 Правила для симулятора (зашиты в его промпт)

1. Отвечай **только** по тому, что написано в загруженных .md-документах. Не придумывай «как обычно делают в 1С» / «как принято вообще».
2. Если в загруженных документах поведение не описано или противоречит само себе → отвечай строго: `[ПРОБЕЛ: <краткое описание, что неясно>]` и **не продолжай** без этого маркера.
3. Указывай **источник** — название раздела/пункта КП-дока, на который опираешься (например `[источник: МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md, раздел 7, кнопка "Отметить как оплаченное"]`).
4. **Не выдумывай** поведение. Если документа нет — это gap, не придумывай «логичное» решение.

### 3.3 Правила для проверяющего (ты / продакт-оунер)

1. Проходи пункты V-001…V-073 **строго по одному**. Не перескакивай.
2. На каждый пункт дай симулятору сценарий-вопрос (можно дословно из пункта, можно перефразировать с дополнительным контекстом).
3. Прочитай ответ симулятора. Если `[ПРОБЕЛ]` — это **подтверждение gap-а** → заведи gap-карточку в журнале (раздел 9).
4. Если ответ «логичный» — проверь, есть ли действительно описание в `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`. Если нет — это gap, даже если ответ звучал разумно.
5. Подключай ИИ-аудитора **раз в неделю** или когда накопилось >10 gap-карточек (см. промпт в разделе 7).

### 3.4 Ожидаемый результат прогона

- Часть пунктов даст ответ «описано в КП-доке, раздел X» — отмечай как ✅ Verified.
- Часть даст `[ПРОБЕЛ]` — это реальные gap-карточки. Они и есть цель прогона.
- Часть даст противоречие — фиксируй как 🔀 Contradiction.

## 4. V-001…V-018 — Стыки через ОБЩИЕ СУЩНОСТИ (Ось A)

Сущности, которые **несколько модулей** одновременно читают или пишут. Несовпадение полей или типов → каскад ошибок.

### V-001: Поля сущности «Товар» (Product)
- **Где:** Product ↔ KP, Contract, Production, Warehouse
- **Вопрос:** Какие поля у товара? Перечисли всё, что упоминается во всех модулях (КП-док + archive/).
- **Опорное поведение:** В КП-доке упоминаются: SKU (артикул), название, фото, тип (Товар/Услуга/Работа), категория, единица измерения (шт/комплект/м²/м.п/кг/час), базовая цена, себестоимость, признак «архивный». В archive/ANALYSIS-KP-priority.md дополнительно: `productType` (purchased/manufactured), `defaultMarkupPercent`, `costPrice`, `basePrice`, `kind` (ITEM/SERVICE/WORK). В archive/project_for_discussion.md: `modules` (BOM), `categories`.
- **Где должен быть ответ:** Модуль «Справочник товаров» (которого нет в v6).

### V-002: Поля сущности «Организация» (Organization)
- **Где:** Organization ↔ KP (как продавец и клиент), Contract, Production (заказчик), Finance
- **Вопрос:** Какие поля у организации? Перечисли с разбивкой на категории (юрлица / ИП / физлица).
- **Опорное поведение:** В КП-доке раздел 3 (таблица «Что должно подставляться автоматически»): полное наименование, краткое, ИНН, КПП, ОГРН/ОГРНИП, юр. адрес, почтовый адрес, банк, БИК, расч. счёт, ФИО подписанта (несколько!), должность подписанта, ставка НДС по умолчанию, срок оплаты в днях, логотип, печать, контактное лицо (несколько!), базовая наценка (10%/12%/15%), роль «продавец», фильтр-чекбоксы по категориям Юрлица/ИП/Физлица.
