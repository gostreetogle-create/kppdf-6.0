## 5. ПРОБЕЛЫ В ДОКУМЕНТАЦИИ — что отсутствует

### 5.1 ПРОБЕЛ-ЗОНА #1: Админка (Справочники)

**Что:** Не описано, как админ заводит Склад, Цех, Поставщика, Шаблон документа. RBAC-таблицы закрывают «кто может», но не «как выглядит экран».

**Критичность:** 🟡 P1 — нужны макеты UI для админки (даже схематичные ASCII).

### 5.2 ПРОБЕЛ-ЗОНА #2: Тестовые бизнес-сценарии (не V-чеклист)

**Что:** V-чеклист проверяет поля и права. Но нет сценариев, покрывающих «что если».

**Примеры отсутствующих сценариев:**
- «Клиент оплатил 50%, произвели 100%, отгрузили 50%, клиент вернул 10%, списали 5% брака. Как сойдётся маржа?»
- «Два менеджера одновременно правят КП одного клиента — что показывает система?»
- «Что видит бухгалтер, если банковская выписка не загружена, но директор ручную ставит „Оплачено" менеджер?»

**Рекомендация:** Создать документ с такими сценариями. Сделан `USER-JOURNEYS.md` в этом аудите как стартовый набор.

### 5.3 ПРОБЕЛ-ЗОНА #3: Safe-increment счётчиков

**Что:** PostgreSQL concurrency не описан. SELECT FOR UPDATE не упомянут. Atomic Counter Table не упомянута.

**Рекомендация:** Описать в `SCHEMA-CONSOLIDATED.md` (сделан в этом аудите).

### 5.4 ПРОБЕЛ-ЗОНА #4: Multi-tenancy (если планируется)

**Что:** Проект явно single-tenant (одна компания). Но если в v2 нужно несколько — нет описания, как изолировать данные.

**Рекомендация:** Зафиксировать «single-tenant» как финальное решение v1 в README.

### 5.5 ПРОБЕЛ-ЗОНА #5: RRULE / Recurring задачи / Авто-нумерация

**Что:** Counter-table есть, но не описаны правила (reset? archive?).

**Рекомендация:** Описать в отдельной секции `SCHEMA-CONSOLIDATED.md`.

### 5.6 ПРОБЕЛ-ЗОНА #6: Валидация для переводов на двуязычие

**Что:** Сказано «отложено», но нет интернационализации в коде моделей.

## 6. БИЗНЕС-ЛОГИКА — общая проверка связности цепочки

### 6.1 Полная цепочка «КП → Деньги»

```
ЦЕПОЧКА: КП → Договор → ЗК → Склад-Приход → Склад-Отгрузка → Финансы → Маржа
```

#### Шаг 1: КП в «Оплачено»

```
Менеджер нажимает «Отметить как оплаченное»
  → Proposal.status = 'paid'
  → Авто-создаётся ProductionOrder (ЗК-XXXX)
  → Proposal.sentAt остаётся прежним
```

**Проверка документов:** `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §7, `МОДУЛЬ-ПРОИЗВОДСТВО.md` §5.1 — **полностью описано**. ✅

#### Шаг 2: Конвертация в Договор

```
Менеджер нажимает «Конвертировать в договор»
  → Доступно ТОЛЬКО из «отправлено» или «принято» (не из «оплачено»!)
  → Копируются позиции в ContractItem (с ценами snapshot по умолчанию, NULL для рамочного)
  → Proposal.status = 'converted'
  → Contract.status = 'draft'
```

**Проверка документов:** `МОДУЛЬ-ДОГОВОР.md` §3 — описано. Но **противоречие #1** — статус «оплачено» нельзя конвертировать. ✅

#### Шаг 3: Подписание Договора

```
Клиент подписывает
  → Manager ставит Contract.status = 'signed'
  → Триггер СПОР-5: создаётся Order в Финансах
```

**Проверка документов:** `МОДУЛЬ-ДОГОВОР.md` §2, `МОДУЛЬ-ФИНАНСЫ.md` §4 — описано. ✅

#### Шаг 4: Производство

```
Начальник распределяет задачи по цехам
  → ProductionOrder.status = 'in_progress'
  → По факту выполнения → ProductionTask.status = 'done'
  → Все задачи done → ProductionOrder.status = 'completed'
  → Авто-создаётся StockMovement: Приход (только для типа «Товар»)
```

**Проверка документов:** `МОДУЛЬ-ПРОИЗВОДСТВО.md` §3, §6 — описано. ✅ НО: **R1 / GAP-009** «ЗК без товаров» — что видит кладовщик? Решение в `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §11.8. ✅

#### Шаг 5: Отгрузка

```
Кладовщик формирует Shipment
  → Shipment.status = 'shipped'
  → Авто-создаётся StockMovement: Расход
  → При 'delivered' → OrderClosing.progress обновляется (в v1 ручной бухгалтером!)
```

**Проверка документов:** `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §6.4 — описано. ✅ **ПРОБЛЕМА:** в `МОДУЛЬ-ФИНАНСЫ.md` `OrderClosing.trigger` явно не описан. R3.

#### Шаг 6: Финансы

```
Бухгалтер регистрирует Payment
  → Invoice.paidAmount += amount
  → Invoice.balance -= amount
  → Если balance = 0 → Invoice.status = 'paid'
  → Если все Invoice.paid → Order.status = 'closed'
  → Маржа = totalAmount - costOfGoodsSold
```

**Проверка документов:** `МОДУЛЬ-ФИНАНСЫ.md` §6 — описано в общих чертах. **НО: 8 открытых Q-вопросов остались.** 🔴
