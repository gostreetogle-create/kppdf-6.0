## 1. Ключевой фактор: ИИ как единственный разработчик

### 1.1 Что важно для ИИ-разработчика

| Фактор | Почему важно | Лучший стек |
|---|---|---|
| **Объём обучающих данных** | ИИ лучше генерирует код для стеков с большим количеством примеров в обучении | React/Next.js >> Angular |

## 2. Сравнение стеков

### 1.1 Frontend Framework

| Критерий | Next.js 16 (React) | Angular 19+ |
|---|---|---|
| **Производительность (SSR)** | Server Components, RSC, Turbopack — быстрый HMR | Hydration, Zone.js — медленнее на cold start |
| **Размер bundle** | 45-80 KB ( React + Next runtime) | 120-150 KB (полный фреймворк) |
| **Скорость разработки** | Высокая ( Server Actions, zero-config) | Средняя (много boilerplate, decorators) |
| **Экосистема UI** | Mantine, Tailwind, Radix, shadcn/ui | Angular Material, PrimeNG |
| **Типизация** | TypeScript strict (из коробки) | TypeScript strict (из коробки) |
| **Управление состоянием** | Zustand + TanStack Query | NgRx / NGXS / Services |
| **Документация** | Отличная, большое сообщество | Хорошая, но меньше community |
| **Для команды 5-10** | ✅ Проще для junior/sредних dev | ⚠️ Требует опытных Angular dev |

**Правило О.С.1 (Быстродействие Frontend).** Next.js быстрее на 20-30% по bundle size и cold start. Для 5-10 пользователей это не критично, но влияет на UX.

### 1.2 Backend / ORM

| Критерий | Prisma 7 + PostgreSQL | Mongoose + MongoDB |
|---|---|---|
| **ACID транзакции** | ✅ Полноценный ACID | ⚠️ Multi-document ACID (с 4.0), но ограниченный |
| **Финансовые операции** | ✅ Идеально (Payment, Invoice, Refund) | ❌ Рискованно (нет строгих FK, rollback) |
| **Foreign Keys** | ✅ Нативные FK + ON DELETE | ❌ Нет FK (программная проверка) |
| **Миграции** | Prisma Migrate (versioned, safe) | Manual scripts или mongoose-migrate |
| **Сложные запросы** | SQL JOINs, CTEs, window functions | Aggregation pipeline (сложнее) |
| **Производительность (5-10 users)** | ✅ Достаточно | ✅ Достаточно |
| **Производительность (1000+ users)** | ✅ Масштабируемо | ⚠️ Требует sharding |

**Правило О.С.2 (Быстродействие Backend).** Для 5-10 пользователей оба варианта работают. Но PostgreSQL + Prisma дают лучшую гарантию целостности данных (критично для финансов).

## 2. Приоритет "Лёгкость обновлений"

> ⚠️ **Критический приоритет:** Обновления должны проходить БЕЗ БОЛИ и БЕЗ ручного вмешательства.

### 2.1 Что делает обновление "лёгким"

| Фактор | Как проверить | Next.js + Prisma | Angular + MongoDB |
|---|---|---|---|
| **Автоматические миграции** | Есть ли CLI для обновления схемы? | ✅ `prisma migrate dev` | ❌ Ручные скрипты |
| **Обратная совместимость** | Ломает ли обновление старые данные? | ✅ Prisma сохраняет данные | ⚠️ MongoDB не гарантирует |
| **Rollback** | Можно ли откатить обновление? | ✅ Prisma supports rollback | ❌ Ручной откат |
| **LTS версии** | Есть ли long-term support? | ✅ Next.js LTS, PostgreSQL LTS | ⚠️ Angular LTS, MongoDB Community |
| **Документация обновлений** | Есть ли чёткий changelog? | ✅ Подробный changelog | ⚠️ Менее подробный |
| **Тесты миграций** | Можно ли проверить миграцию на dev? | ✅ `prisma migrate deploy --dry-run` | ❌ Нет стандартного способа |

### 2.2 Сценарий обновления (шаг за шагом)

**Next.js + Prisma (рекомендуется):**
```bash
# 1. Обновляем зависимости
pnpm update

# 2. Генерируем новые типы
pnpm prisma generate

# 3. Применяем миграции (автоматически)
pnpm prisma migrate deploy

# 4. Проверяем
pnpm vitest run
```
**Время:** 5-10 минут. **Ручное вмешательство:** 0.

**Angular + MongoDB:**
```bash
# 1. Обновляем зависимости
npm update

# 2. Ручные скрипты миграции (если есть)
node scripts/migrate-v2.js

# 3. Ручная проверка схемы
# 4. Ручное тестирование
```
**Время:** 30-60 минут. **Ручное вмешательство:** Обязательно.

### 2.3 Риски обновлений

| Риск | Next.js + Prisma | Angular + MongoDB |
|---|---|---|
| **Потеря данных при миграции** | Низкий (Prisma проверяет) | Высокий (нет FK) |
| **Breaking changes в API** | Низкий (стабильный API) | Средний (Angular обновления) |
| **Необходимость ручного вмешательства** | Нет | Да |
| **Время простоя при обновлении** | 5-10 мин | 30-60 мин |

## 3. Анализ по приоритетам заказчика

### 2.1 Быстродействие

**Для 5-10 пользователей:**
- Оба варианта (Next.js и Angular) обеспечивают достаточное быстродействие
- Next.js выигрывает по bundle size (45-80 KB vs 120-150 KB)
- PostgreSQL быстрее на сложных запросах (JOINs, агрегации)
- MongoDB быстрее на простых read/write операциях (document-based)

**Правило О.С.3 (Подытог по быстродействию).** Next.js + PostgreSQL для общего быстродействия, но разница минимальна для 5-10 users.

### 2.2 Качество

**Качество кода:**
- Prisma дает type-safe запросы (меньше багов)
- PostgreSQL гарантирует целостность данных (FK, constraints)
- Angular имеет строгую архитектуру (modules, services, dependency injection)
- React более гибкий, но требует дисциплины

**Качество данных:**
- PostgreSQL: ACID, FK, constraints, triggers — идеально для финансов
- MongoDB: гибкая схема, но нет строгих ограничений — риск битых данных

**Правило О.С.4 (Подытог по качеству).** PostgreSQL для качества данных, Angular для качества кода (при опытной команде).

### 2.3 Поддерживаемость

**Для команды 5-10:**
- Next.js проще для junior/middle разработчиков
- Angular требует более опытных dev (decorators, RxJS, modules)
- Prisma проще в поддержке чем Mongoose (миграции, типы)
- MongoDB проще в настройке, но сложнее в долгосрочной поддержке (нет FK)

**Правило О.С.5 (Подытог по поддерживаемости).** Next.js + Prisma для поддерживаемости.
