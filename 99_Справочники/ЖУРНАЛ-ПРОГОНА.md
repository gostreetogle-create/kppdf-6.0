### 8.2 Ключевые находки (Ось D)

**1. Конструктор шаблонов = самый старший P1 gap для оси D.** Без UI админки админ не может создать шаблон «Договор поставки 2026» — что означает **все шаблоны жёстко закодированы в коде** на старте v1 (только 4 стандартных: Стандартный/Компактный/Развёрнутый/Розничная). Это не катастрофа (для v1 хватает), но снижает гибкость для нестандартных сделок. **Рекомендация**: описать минимальный UI для v2 (левая колонка = меню блоков, центр = редактор Markdown-блоков, правая = preview А4). Подробнее — GAP-017.

**2. packageTag как string упрощает картотеку, но ограничивает cross-deal scenarios.** СПОР-10 зафиксировал `packageTag: String` (строго 1:1). Если клиент ведёт две параллельные сделки с нами (например, «спортивное оборудование» и «ремонт зала») — менеджер **не может создать общий сводный документ** в одной Картотеке. Это OK для 90% случаев. Если продакт-оунере понадобится гибкость — переход на `packageTags: String[]`. Зафиксировано как GAP-020 (P2).

**3. Кросс-модульные гиперссылки — UX-критично (GAP-019).** Из Картотеки сделки директор должен одним кликом перейти в любой КП/Договор/ЗК/Отгрузку/OrderClosing. Сейчас это сделать можно (есть `parentProposalId`, `parentContractId`, `productionOrderId` в соответствующих модулях), но **визуальное представление номеров документов в виде кликабельных ссылок — стандарт UI, не описан ни в одном МОДУЛЬ-доке**. Дублирование правки потребуется в **всех 5 модулях** одновременно (КП, Договор, Производство, Склад, Финансы). Зафиксировано как GAP-019 (P1) — критично для UX.

**4. Карточка клиента без вкладки «Документы»**. Базовый use case: менеджер открывает «ООО Спорткомплект» → видит все КП, Договоры, ЗК, отгрузки клиента. Сейчас приходится переключаться в `/proposals` с фильтром по `customerId` или `/contracts`/`/production` аналогично. Это 3 клика вместо 1. Зафиксировано как GAP-018 (P2) — мелочь, но ежедневная боль.

**5. Открытые вопросы в `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §8** (по автосохранению) — `V-057` отметил их как «открытый вопрос, можно (отдельный в ОТКРЫТЫЕ-ВОПРОСЫ.md)». **Рекомендация для v1**: автосохранение работает для КП-редактора + корзины (уже описано). Для формы товара/клиента — отложено в v2.

**6. Сценарий «Кладовщик ищет ЗК через Картотеку»** (R1 + GAP-009): неявно покрыт. Картотека сделки (по `packageTag`) содержит ВСЕ документы, включая ЗК/СД/ОТК клиента. Кладовщик заходит в `/packages?tag=X`, видит строку «ЗК-0023 от 14.05.2026 — Статус: Завершён», клик открывает `/production/0023`. **Это закрывает часть R1 + GAP-009**, но явная UX-инструкция для кладовщика в `МОДУЛЬ-СКЛАД-UI.md` §6.3 (где описаны связи со Складом) **уже сделана** — sample TX ✅.

### 9.2 Ключевые находки Блока 2e

1. **Multi-currency = заложить архитектурно СЕЙЧАС, даже если UI не показывает.** СПОР-14 фиксирует RUB-only в v1, но Prisma 6 MIGRATION‑PLAN §8А уже требует `currency: String @default(\"RUB\")` во всех 4 Warehouse‑доках. Без этого v2 с включением USD/EUR сломает БД. **Рекомендация**: добавить поле в БД сейчас (даже если UI скрывает) — закрывает GAP-021. Пересечение с **СПОР-14**.

2. **Частичная оплата ≠ binary «Оплачено».** В `МОДУЛЬ-ФИНАНСЫ.md` Invoice имеет статус «Частично оплачен» (RFC), но в КП статус бинарный. Менеджер вручную нажимает «Оплачено» по факту получения аванса. Синхронизация КП↔Invoice не описана. **Рекомендация v1**: бинарный КП-статус + бухгалтерский комментарий в Invoice ; v2 — синхронизация. Закрывает GAP-022.

3. **Refund без корректировочного Invoice ломает `OrderClosing.balance`.** СПОР-12 зафиксировал Refund через Модуль Финансы, но как физически оформить отрицательный `Payment.amount` ? Модель данных в `МОДУЛЬ-ФИНАНСЫ.md` ещё не детализирована (каркас статусов есть, поля `Payment` не описаны). **Рекомендация**: расширить `МОДУЛЬ-ФИНАНСЫ.md` до полной схемы Payment/Invoice/OrderClosing в Phase 0‑prep. Закрывает GAP-023.

4. **Консорциумы (multi‑org в одной сделке) = разные КП+Договоры с общим `packageTag`.** Подтверждено, что для v1 поддерживается только последовательность: «2 КП + 2 Договора → связь через `packageTag` (СПОР‑10)». Это дороже для пользователя, но решимо. v2 : дополнительные FK `consigneeId` для грузополучателя. Закрывает GAP-024.

5. **validUntil = `Date` (без времени), граница включительна.** КП действует до конца дня `validUntil` (включительно) в часовом поясе пользователя. Хранение в БД как `Date` (Prisma 6), не `DateTime`. Предупреждение показывается с **00:00** следующего дня. Закрывает GAP-025.
