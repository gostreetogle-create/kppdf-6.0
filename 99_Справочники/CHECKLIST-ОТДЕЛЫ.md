# ЧЕК-ЛИСТ ПРОГОНА — «инженеры космического корабля»

> **Метафора:** Перед стартом ракеты-носителя каждый отдел-инженер по очереди говорит «ГОТОВ». Если хоть один говорит «НЕ ГОТОВ» или молчит — старт отменяется. Этот чек-лист — **сводный опросник**, через который пропускается наша CRM-система перед каждым крупным релизом (Phase 0-prep → Phase 1 Bootstrap → Phase 2 Bootstrap → v1 MVP).
>
> **Источники:**
> - Пример пользователя (5 базовых блоков — Сущности/Процессы/Критерии/Алгоритм/Итог)
> - Дополнения Буфера: разделы для CRM-системы v6 (жизненные циклы, edge-кейсы, миграции, observability, compliance, UX/a11y)
> - Web-research: NASA/FAA/финтех-практики readiness checklist, dark/grey zones, state machine coverage
> - Архитектура проекта v6: `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`, `МОДУЛЬ-ДОГОВОР.md`, `МОДУЛЬ-ПРОИЗВОДСТВО.md`, `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md`, `МОДУЛЬ-ФИНАНСЫ.md`, `RBAC-MATRIX.md`, `РИТМ-FLOW-MAP.md`
>
> **Как использовать:**
> 1. **Перед релизом** — пройтись по каждому блоку, по каждому пункту проставить ✅/❌/⚠️
> 2. **Накопление дыр** — все ❌ записать в таблицу «Найденные дыры» (последний раздел)
> 3. **Финал** — последний раздел «ИТОГОВАЯ ПРОВЕРКА»: каждый блок должен иметь подпись владельца «ГОТОВ» или «НЕ ГОТОВ (компенсируется X)»
> 4. **Результат** — «ОТДЕЛ ГОТОВ = 100% ✅» = сигнал к релизу
>
> **Периодичность:** Полный прогон — перед каждой смены фазы (Phase 0→1, 1→2, 2→MVP). Выборочный (только изменённые блоки) — перед каждым minor-релизом.

> Добавлено мной. Каждый домен v6 имеет диаграмму переходов. Чек-лист покрытия — стандартный паттерн QA (state coverage + valid/invalid transitions + dead states).

| № | Сущность | Статусы | Где диаграмма |
|---|---|---|---|
| 1 | `Proposal` (КП) | DRAFT, SENT, ACCEPTED, REJECTED, CONVERTED, PAID, ARCHIVED | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §2 |
| 2 | `Contract` (Договор) | DRAFT, SIGNED, ACTIVE, COMPLETED, ARCHIVED, **TERMINATED** (добавлен 24.06.2026) | `МОДУЛЬ-ДОГОВОР.md` §5 |
| 3 | `ProductionOrder` (ЗК) | CREATED, IN_PROGRESS, COMPLETED, ARCHIVED, **CANCELLED** | `МОДУЛЬ-ПРОИЗВОДСТВО.md` §3 |
| 4 | `Order` (финансы) | DRAFT, IN_PROGRESS, **CANCELLED**, CLOSED | `МОДУЛЬ-ФИНАНСЫ.md` §4 |
| 5 | `SupplierDelivery` (СД) | 6 статусов | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §5 |
| 6 | `Shipment` (ОТК) | 6 статусов | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §6 |

| 8 | `PurchaseOrder` (ЗП) | 8 статусов | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §8 |

### Критерии покрытия по state machines:

- [ ] **State coverage:** Каждый статус достижим хотя бы одним переходом. Пройти через все статусы в smoke-тесте.
- [ ] **Valid transitions:** Каждый документированный переход (A → B) работает согласно диаграмме.
- [ ] **Invalid transitions (negative):** Попытка нелегального перехода (A → C минуя B) — ошибка с понятным сообщением. Никаких 500-ок, никаких silent successes.
- [ ] **Dead states:** Есть ли тупиковые статусы (откуда нет выхода)? В v6 такие: `REJECTED`, `ARCHIVED` — допустимо, но нужно подтверждение через audit log.
- [ ] **Unreachable states:** Не упомянуты ли в enum статусы, до которых невозможно добраться (это code smell). Пример: `Order.status='CANCELLED'` появился недавно — все ли знают, как туда попасть?
- [ ] **Missing transitions:** Нет ли «зависающих» документов (статус без кнопки «куда дальше»). Пример: ошибка оплаты → Proposal = ??? (нужно принять решение).
- [ ] **Side-effects:** Каждый переход имеет привязанные триггеры? `Contract→SIGNED` → создание Order. `ProductionOrder→COMPLETED` → StockMovement (только ITEM). Все триггеры прописаны в `FLOW-MAP.md`.

### Визуальный чек (косвенный):

- Если диаграмма статусов напоминает «спагетти» (> 12 переходов, ромбовидные развилки) — это архитектурный smell. До запуска упростить.

## Часть V. КРИТЕРИИ ПОКРЫТИЯ RBAC (dark zones в правах)

> Расширение базового блока «Критерии покрытия» из пользовательского примера — конкретно под RBAC v6.

- [ ] **Каждый ресурс имеет владельца роли:** нет ресурса без явно назначенных разрешений. Если в матрице пустая ячейка — это либо явно «запрещено», либо «не применимо» с комментарием почему.
- [ ] **Каждое действие над ресурсом разрешено хотя бы одной роли:** иначе feature существует, но никто не может его использовать → мёртвый код.
- [ ] **Нет «сиротских» разрешений:** каждое `Permission` связано с существующей `Role`. Удаление роли → каскад на permissions.
- [ ] **Матрица полна:** все пары (роль × ресурс × действие) задокументированы. Сейчас: 6 ролей × ~30 действий = **180 ячеек**. Каждая ячейка ✅/❌/N/A.
- [ ] **Нет горизонтальной эскалации:** пользователь с ролью `manager` не может, добавив себе ещё `accountant`, получить `admin`-права. Union приводит к union, не к escalation.
- [ ] **Минимальные привилегии:** у каждой роли нет разрешений «про запас». Каждое право имеет business justification.
- [ ] **SoD (разделение обязанностей):** одна роль не может:
  - Создать `Payment` И подтвердить его (спек: manager+accountant по разным ролям требуют 2 разных людей)
  - Зарегистрировать `Order` и одобрить `OrderClosing`
  - Создать `WriteOffAct` и подписать его (2-stage approve обеспечивает это)
  - Подписать `Contract` и сделать `Refund` по нему
- [ ] **Гостевой доступ:** пользователь без ролей → только `/login`, `/profile`, `/forgot-password`. Никаких утечек через API.
- [ ] **Множественные роли:** если у пользователя 2+ роли — результирующие разрешения объединяются (или пересекаются по правилу приоритета запрета). Явно зафиксировано в коде.
- [ ] **Переходы ролей:** повышение/понижение — теряет старые права, получает новые. Не происходит задержки или «призрачных» прав.

### Как искать dark zones в правах (методика):

1. Пройти по **FLOW-MAP.md** — каждый шаг journey. Для каждого шага спросить: «Кто может это сделать?» Если ответ «никто» — дыра.
2. Взять каждое `Permission` из `RBAC-MATRIX.md` — спросить: «Кто использует?» Если ответ «никто» — удалить (overengineering).
3. Тест «Случайный пользователь»: создать User без `manager` — попытаться создать КП. Должно быть 403.

## Часть VI. БЕЗОПАСНОСТЬ

> Добавлено мной. RBAC — лишь один из слоёв безопасности.

| № | Категория | Пункты проверки |
|---|---|---|
| 1 | **Transport** | TLS/HTTPS везде, HSTS заголовок, редирект HTTP→HTTPS, корректные сертификаты (не self-signed в проде). |
| 2 | **Auth** | Пароли: bcrypt/argon2 (не MD5/SHA). JWT короткоживущие + refresh. Logout инвалидирует токены. Lockout после N попыток. |
| 3 | **Secrets** | Нет секретов в коде/Git. `.env` в `.gitignore`. Production secrets через Vault/AWS Secrets Manager. |
| 4 | **Input validation** | Все эндпоинты валидируют input (Zod-схемы уже есть в `src/lib/validations/`). Server-side, не только client. |
| 5 | **XSS / CSRF / SQL inj** | Output encoding, CSRF-токены для state-changing операций, parameterized queries (Prisma уже спасает). |
| 6 | **CORS / Same-origin** | Строгий whitelist origins. Cookies: `SameSite=Lax`/`Strict`, `HttpOnly`. |
| 7 | **Rate limiting** | На критичных эндпоинтах (login, payment). На каждый IP. Degrade gracefully при избытке. |
| 8 | **PII / GDPR** | Минимизация: храним только нужное. Право на удаление: soft-delete. Логи не содержат PII. |
| 9 | **Audit log** | Все auth-события (login/logout/failed), все permissions changes. Append-only, immutable. |
| 10 | **Dependency check** | `npm audit` без critical. Зависимости обновлены до non-vulnerable версий. SCA (Software Composition Analysis) в CI. |

## Часть VII. ЦЕЛОСТНОСТЬ ДАННЫХ

> Добавлено мной. CRM-система особенно страдает от «битых ссылок» между модулями.

- [ ] **FK constraints:** каждая связь в `SCHEMA-CONSOLIDATED.md` имеет явный `ON DELETE` (RESTRICT / SET NULL / CASCADE). Нет `ON DELETE NO ACTION` (undeclared).
- [ ] **Транзакции:** многошаговые операции атомарны (пример: Конвертация КП → Договор → смена статусов — всё в одной транзакции).
- [ ] **Snapshot integrity:** поля `*AtPriceSnapshot`, `*AtQtySnapshot` заполняются при создании записи `ProposalItem` и **не пересчитываются** при изменении `Product.price`. Тест: меняем price после создания КП → сумма КП неизменна.
- [ ] **Cascade behavior:** тест: удаление `Customer` с активными КП — должно либо блокироваться (RESTRICT), либо переводить в `ARCHIVED` (soft). Нет silent orphans.
- [ ] **Dead records:** регулярный отчёт «записи без внешних ссылок». В v1: ручной или SQL-запрос.
- [ ] **Immutability:** `StockMovement`, `Payment`, `Comment` — append-only. Тест: попытка UPDATE → ошибка.
- [ ] **Currency consistency:** все финансовые суммы в одной currency (RUB) в v1. `currency` поле заведено «на вырост» для v2 multi-currency. Тест: попытка создать Payment в USD → 400.
- [ ] **Total calculations:** `Order.totalAmount`, `Proposal.totalAmount` = сумма активных items. Тест: добавил/удалил item → totals пересчитаны атомарно.

## Часть VIII. ОТКАЗОУСТОЙЧИВОСТЬ (resilience)

> Добавлено мной. Что если что-то сломалось ПОСРЕДИ операции?

| № | Сценарий отказа | Ожидаемое поведение | Где проверяется |
|---|---|---|---|
| 1 | Сеть отвалилась при сохранении КП | UI показывает «сохранено локально, отправится при восстановлении» (через localStorage + retry). Idempotent по `updatedAt`. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §11 |
| 2 | Внешний API банка вернул timeout при подтверждении Payment | Payment остаётся в `PENDING`, scheduler retries через 5/30/120 мин, потом → `FAILED` с уведомлением бухгалтеру. НЕ записываем в Order без подтверждения. | `МОДУЛЬ-ФИНАНСЫ.md` §6 |
| 3 | DB connection pool исчерпан | Endpoint возвращает 503 через 5 секунд. Текущие транзакции не теряют данные (rollback). | `FLOW-MAP.md` §3 |
| 4 | Conflicting concurrent edits | Optimistic locking через `updatedAt`. Пользователь видит «документ изменён, обновить?». НЕ silent overwrite. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §11 (правка «АНАЛИЗ-П1») |
| 5 | Power loss на сервере посреди транзакции | Postgres гарантирует атомарность (WAL). Все partial commits откатываются. | `SCHEMA-CONSOLIDATED.md` (тех.раздел) |
| 6 | Disk full при backup | Backup падает → immediate alert + запрет новых операций до решения. Не silent data loss. | Ops checklist (отдел Ops) |
| 7 | Попытка откатить уже подписанный Договор | Заблокировано (RBAC + business rule). Вместо этого — новая редакция через `parentContractId`. | `МОДУЛЬ-ДОГОВОР.md` §9 (Q3) |

### Fault Injection Testing (обязательно иногда):

- [ ] Симуляция обрыва сети (Chrome DevTools → Offline)
- [ ] Симуляция медленного DB (например, `pg_sleep(30)` в тестах)
- [ ] Симуляция отмены посреди транзакции (kill -9 server)

Записать результаты в «Матрицу рисков» (dark/grey zones, см. Часть XVIII).

## Часть IX. КОНКУРЕНТНОСТЬ

> Добавлено мной. Менеджеров 6+, они правят КП параллельно.

- [ ] **NoLostUpdates:** два менеджера правят один КП, оба сохраняют — один получает «устаревшие данные» alert, не silent overwrite.

## Часть XI. ПОЛЬЗОВАТЕЛЬСКИЕ ПУТИ (smoke / integration)

> Добавлено мной. Все journeys из `USER-JOURNEYS.md` (7+1) должны проходить end-to-end в реалистичных средах.

### Список сквозных journeys для smoke-прогона:

1. **Менеджер создаёт КП** → отправляет → принимают → конвертирует → менеджер-создатель становится владельцем Договора
2. **Менеджер + бухгалтер** на одной сделке: КП оплачено → Order создан → Payment подтверждён → ЗК стартует
3. **Производственник** + **склад**: ЗК завершён → авто StockMovement (только ITEM) → авто-предложение Shipment → кладовщик создаёт Shipment → отгружает
4. **Кладовщик + бухгалтер**: Shipment delivered → costOfGoodsSold фиксируется → OrderClosing возможен → маржа рассчитана
5. **Менеджер по закупкам + кладовщик**: StockRecord.quantity < minStock → авто PurchaseRequest → ЗП создан → СД принят → авто StockMovement IN
6. **Директор** + Любая роль: открывает Картотеку сделки по `packageTag` → видит все 5+ документов в одном view → может читать комментарии (GAP-013)
7. **Админ**: создаёт User с `manager` → новый менеджер входит → создаёт КП. Тест полного RBAC cycle.
8. **Возврат/списание**: бракованное кольцо → клиент вернул → WriteOffAct с reason=DEFECT → StockMovement OUT → Order correction.

Каждый journey = **1 integration test**. Минимум 1 happy path + 1 sad path (отказ/edge).
