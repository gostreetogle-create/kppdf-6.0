## §1. Mission и принципы

### 1.1 Главный вопрос методологиста

> **«Каждый из 15 СПОР-ов и 38 Q имеет явный 1-5 score по 5 критериям, и существует шаблон для будущих СПОР-ов?»**

Если ответ «да» — задание сдано.
Если ответ «нет, и вот почему» — фиксируем gaps в Post-action и отдаём PO на решение.

### 1.2 Три принципа методологиста

1. **APPEND-only.** Никогда не удалять / переписывать существующий текст СПОРА/Q. Только добавлять блок `## Scoring` в КАЖДУЮ запись.
2. **CONSISTENCY выше индивидуальности.** Один и тот же критерий оценивается по одной шкале для всех 53 артефактов (см. §5).
3. **TRACEABILITY.** Каждый scoring-блок содержит дату оценки + рецензента (ИИ-агент).

### 1.3 Что НЕ входит в миссию

| За пределами | Почему |

| Изменение решения по СПОР/Q | Запрещено (CHECKLIST §9 «не ломай сделанного») |

## §5. Phase 1 — Scoring rubric (канонические 5 критериев × 1-5)

### 5.1 5 критериев (канонизировано PSL-007)

| # | Критерий | Что оценивает | Шкала |
|---|---|---|---|
| **K1** | **Бизнес-ценность** | Влияние на прибыль / UX / процессы | 1 = косметика / 5 = критическая функция |
| **K2** | **Сложность** | Трудоёмкость реализации | 1 = тривиально / 5 = требует месяцев |
| **K3** | **Риски** | Риск от неправильного решения | 1 = минимальный / 5 = катастрофический |
| **K4** | **Совместимость** | Соответствие стеку/архитектуре | 1 = требует рефактора / 5 = нативно для стека |
| **K5** | **Время** | Срочность | 1 = может подождать / 5 = горит |

### 5.2 Шкала 1-5 (подробно)

| Балл | K1 Бизнес-ценность | K2 Сложность | K3 Риски | K4 Совместимость | K5 Время |
|---|---|---|---|---|---|
| **1** | Косметика / UX-эстетика | Тривиальная правка | Минимальный (макс. локальный файла) | Требует серьёзного рефактора | Отложить триместр |
| **2** | Полезно, но не блокирующее | Часть модуля | Локальный (затрагивает <3 сущности) | Требует частичной адаптации | Можно в след. release |
| **3** | Важно для 1-2 ролей | Целый модуль | Средний (затрагивает 1 модуль целиком) | Совместимо с минимальными изменениями | В текущем квартале |
| **4** | Существенно для большинства ролей | Связанные модули | Высокий (cross-module эффект) | Хорошо ложится на стек | В текущей итерации |
| **5** | Критическая для продукта | Архитектурный уровень | Катастрофический (может остановить проект) | Нативно для стека | Горит, делать сегодня |

### 5.3 Формат scoring-блока (canonical)

```markdown

## Scoring

**Дата оценки:** YYYY-MM-DD
**Рецензент:** METHODOLOGIST / <модель>

| Критерий | Балл | Обоснование |
|---|---|---|
| K1 Бизнес-ценность | N | <1-2 строки почему> |
| K2 Сложность | N | <1-2 строки почему> |
| K3 Риски | N | <1-2 строки почему> |
| K4 Совместимость | N | <1-2 строки почему> |
| K5 Время | N | <1-2 строки почему> |
| **Σ** | <N сумма> | average: <среднее> |

**Verdict (на основе scoring):**
- (Σ ≥ 20) — критический приоритет, делать сегодня
- (15 ≤ Σ < 20) — высокий приоритет, текущая итерация
- (10 ≤ Σ < 15) — средний приоритет, текущий квартал
- (5 ≤ Σ < 10) — низкий приоритет, отложить
- (Σ < 5) — косметика, можно отменить

**Cohort:** <категория спора, например «Бизнес-логика КП»>
```

### 5.4 Sanity-check правил (обязательно для каждого блока)

Агент ОБЯЗАН проверить:
- [ ] Все 5 критериев имеют балл (не 0 и не >5).
- [ ] Баллы — числа 1-5 (не 0, не 6, не «высокий», не три звезды).
- [ ] Каждый балл имеет обоснование (1-2 строки, не пусто).
- [ ] Σ + average тоже есть (для удобства сортировки).
- [ ] Verdict рассчитан корректно по порогам.
- [ ] Cohort назначен (нужно для группировки).

## §6. Phase 2 — Apply scoring к 15 СПОР (ретроактивно)

### 6.1 Алгоритм для каждого СПОР-N

1. Прочитай `СПОРНЫЕ-МОМЕНТЫ.md`.
2. Найди запись `### СПОР-N — <Название>` (или `## СПОР-N`).
3. Перейди в конец этой записи (до следующего СПОР-N+1 или EOF).
4. APPEND блок `## Scoring` по формату §5.3.
5. Оцени по 5 критериям согласно содержанию СПОР-N.

### 6.2 Шпаргалка оценки для типовых СПОР

| Тип СПОР | Типичные scores |
|---|---|
| Стратегический (направление проекта) | K1=4-5, K2=3-5, K3=3-4, K4=3-4, K5=2-3 |
| Тактический (конкретный модуль) | K1=3-4, K2=2-3, K3=2-3, K4=4-5, K5=3-4 |
| UI/UX | K1=4-5, K2=2-4, K3=1-2, K4=4-5, K5=3-5 |
| Архитектурный | K1=3-5, K2=4-5, K3=4-5, K4=3-4, K5=2-3 |
| Процессный (методология) | K1=3-4, K2=2-3, K3=2-3, K4=4-5, K5=1-3 |

### 6.3 Пример (СПОР-1 условный)

```markdown

## Scoring

**Дата оценки:** 2026-06-27
**Рецензент:** METHODOLOGIST / Claude Sonnet-4.5

| Критерий | Балл | Обоснование |
|---|---|---|
| K1 Бизнес-ценность | 4 | Существенно для всех ролей КП-модуля |
| K2 Сложность | 3 | Один модуль с cross-ref на 3 других |
| K3 Риски | 2 | Локальный (только этот модуль) |
| K4 Совместимость | 4 | Ложится на стек без рефактора |
| K5 Время | 3 | В текущей итерации Phase 1 Bootstrap |
| **Σ** | **16** | average: 3.2 |

**Verdict:** 15 ≤ Σ < 20 — высокий приоритет, текущая итерация

**Cohort:** Бизнес-логика КП
```

## §9. Pass/Fail gates

### 9.1 Verdict logic для ТЗ-006

```
IF coverage == 100% (54/54) AND SANITY_OK AND TEMPLATE_OK:
    VERDICT = ✅ PASS
ELIF coverage >= 95% (51/54) AND SANITY_OK:
    VERDICT = ⚠️ PASS-WITH-WARNINGS
ELIF coverage < 95%:
    VERDICT = ❌ BLOCKED
```

Где:
- `coverage = (scored_spor + scored_q) / 53`
- `SANITY_OK = (все scoring-блоки прошли §5.4) И НЕТ дубликатов ID`
- `TEMPLATE_OK = (шаблон создан в §0 СПОРНЫЕ-МОМЕНТЫ.md И все 9 полей присутствуют)`

### 9.2 Что означает каждый verdict

| Verdict | Что делать |
|---|---|
| ✅ PASS | Задание сдано, методология канонизирована |
| ⚠️ PASS-WITH-WARNINGS | Сдать PO для review warnings |
| ❌ BLOCKED | Не сдано; переделать с акцентом на недостающие артефакты |

## §10. Anti-patterns методологиста

### A1. Не перезаписывать существующий текст СПОР/Q

APPEND-only. Трогать можно ТОЛЬКО добавляя блок `## Scoring`.

### A2. Не выдумывать оценки

Score должен иметь обоснование. Если непонятно какой балл → пиши обоснование «insufficient data, оставлено для review PO».

### A3. Не выходить за рамки 53 артефактов

Если видишь незарегистрированный СПОР или Q → flag в Post-action, не создавай новый.

### A4. Не изменять scoring-критерии

5 критериев 1-5 канонизированы в §5. Изменение критериев = новый PSL-NNN.

### A5. Не смешивать RU/EN терминологию

Использовать GLOSSARY-MASTER для всех терминов.

### A6. Не повторять ID

Σ и verdict рассчитаны по формуле одинаково для всех 53 артефактов — никаких отклонений.

### A7. Не оставлять полу-заполненные блоки

Каждый блок должен быть либо полным, либо отсутствующим (без N/A без обоснования).
