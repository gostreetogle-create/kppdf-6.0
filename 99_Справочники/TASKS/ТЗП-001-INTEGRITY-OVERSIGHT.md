# ТЗП-001 — INTEGRITY-OVERSIGHT (Техническое задание на проверку целостности)

> **Назначение.** ТЗП-001 — **надзорное** задание (отличается от ТЗ = «работать»). Цель: проверить ЦЕЛОСТНОСТЬ всего, что сделано в проекте KPPDF CRM v6 к моменту завершения 3-5 параллельных агентов (ТЗ-001/002/003/004/005). Агент-надзор проверяет cross-ref / дубли / hard limits / spec compliance / numbers convergence / специфичные правила по 5 модулям, и **имеет право БЛОКИРОВАТЬ** Phase 2 (Mantine UI) и Phase 1 Bootstrap Prisma при обнаружении критических нарушений. Это руководящая роль — выше любой из 5 рабочих ролей.
>
> **Жанр.** ТЗП (техзадание на проверку) = параллельный жанр к ТЗ (техзадание на работу). В отличие от ТЗ, ТЗП **не создаёт нового контента** — только верифицирует существующий и фиксирует результат проверки. Если в ходе проверки обнаружены нарушения — пишет GAP-LIST, но **НЕ правит нарушения сам** (это прерогатива Аналитика / Архитектора).
>
> **Объём:** ~1100 строк. Hard limit ≤ 1500.
>
> **Hard pre-action:** см. §0 ниже.

## §1. Mission и принципы

### 1.1 Главный вопрос аудита

> **«Можно ли стартовать Phase 1 Bootstrap Prisma (код) и Phase 2 Mantine UI, опираясь на текущую документацию, без риска глобальных переделок?»**

Если ответ «да с оговорками» — выдать ⚠️ PASS-WITH-WARNINGS + список оговорок.
Если ответ «нет, и вот почему» — ❌ BLOCKED + причины блокировки.

### 1.2 Три принципа надзора

1. **ЦЕЛОСТНОСТЬ выше полноты.** Лучше проверить 80% глубоко, чем 100% поверхностно. Глубокий аудит = anti-drift защита.
2. **ДОКАЗАТЕЛЬНОСТЬ выше мнения.** Каждое утверждение в отчёте = ссылка на конкретный файл/строку. Без «по-моему, тут плохо» — только «файл X, строка Y, правило Z нарушено».
3. **БЛОКИРОВКА выше вежливости.** Не делать вид, что всё хорошо, если это не так. PASS-WITH-WARNINGS на 🔴 P0 = фактически BLOCKED.

### 1.3 Что НЕ входит в миссию аудитора

| За пределами аудита | Почему |
|---|---|
| Написание нового контента | Прерогатива Архитектора / Аналитика |
| Правка обнаруженных нарушений | Прерогатива Аналитика / Архитектора |
| Решение бизнес-вопросов | Прерогатива PO |
| Код-ревью кода | Phase 3 задача (TechWriter + QA роль) |
| Пересмотр закрытых PSL/Q/СПОР | Запрещено (см. CHECKLIST.md §9 «не ломай сделанного») |

## §5. Phase 1 — Inventory (построить полный реестр)

### 5.1 Алгоритм

1. Открой `PROJECT-STATE-LOG.md`.
2. Прочитай все PSL-NNN за текущий месяц.
3. Составь список `files_in_scope = []`.
4. Для каждого PSL: добавь в `files_in_scope` все файлы из поля «Затронутые файлы».
5. Удали дубли (sort uniq).
6. Добавь meta-файлы из §2.3 (ВСЕГДА в scope).
7. Финал: `inventory.txt` с columns: file_path, lines, last_modified, psl_source.

### 5.2 Формат inventory.txt

```
file_path	lines	last_modified	psl_source
AGENT-METHOD.md	500	2026-06-26	PSL-008,PSL-009
CHECKLIST.md	350	2026-06-27	PSL-006,PSL-010
...
```

### 5.3 Что делать с файлами вне inventory

Файлы вне inventory НЕ проверяются (anti-pattern: scope creep). Если аудитор считает что какой-то файл тоже нужно проверить → добавляет в §5 AUDIT-REPORT «Что НЕ проверил».

## §6. Phase 2 — Static checks (12 категорий mechanical проверок)

Каждая категория = 1 binary check (PASS / FAIL + детали). Категории сгруппированы по приоритету (P0/P1/P2).

### 6.1 🔴 P0 BLOCKING категории (старт Phase 2 заблокирован при FAIL)

| # | Категория | Что проверяется | Метод |
|---|---|---|---|
| **C1** | Hard limits × meta-файлы | CHECKLIST.md ≤400 строк / AGENT-METHOD.md ≤500 / PROJECT-STATE-LOG.md без overflow | `wc -l` + сравнение |
| **C2** | Cross-references × broken | Все `[link](path)` ведут на существующие файлы | regex extract + file_exists check |
| **C3** | ID conflicts × PSL/Q/СПОР | Нет конфликтующих IDs в PROJECT-STATE-LOG (монотонный рост PSL-NNN), OQ-XXX, СПОР-N | grep uniqueness |
| **C4** | SPEC compliance × Phase 1 Bootstrap | Если ТЗ-004 запущен — код соответствует СТЕК-ПРЕДПИСАНИЕ.md (правильные версии пакетов) | text search + manual review |
| **C5** | Schema-doc alignment | Если `schema.prisma` создан — все 32 сущности из SCHEMA-CONSOLIDATED.md присутствуют | entity_name grep |

### 6.2 🟡 P1 IMPORTANT категории (warning, не блокировка)

| # | Категория | Что проверяется | Метод |
|---|---|---|---|
| **C6** | Format compliance × A1-A11 | Все .md следуют AGENT-FORMAT (явные нумерации правил, цифры вместо слов, нет дублирования) | heuristic + manual |
| **C7** | Rule numbering consistency | В ТЗ-001 REGISTRY — каждый RULE-NNN уникален, формат `RULE-{MODULE}-{TYPE}-{NNN}` | regex + count |
| **C8** | Cross-module rule coherence | Правила из 5 модулей не противоречат друг другу (особенно status-машины) | semantic cross-check §7 |
| **C9** | Numbers convergence | 38 Q все ПРИНЯТО, 15 СПОР все закрыты, 73 V-проверки все ✅ | grep counts vs CHECKLIST §7 |
| **C10** | UX-принципы соблюдены | В LAUNCH-UX.md (если создан) — присутствуют 4 принципа из AGENT-METHOD §5.4 | text search + structural |

### 6.3 🟢 P2 NICE-TO-HAVE категории

| # | Категория | Что проверяется | Метод |
|---|---|---|---|
| **C11** | Style consistency | Нет смешения стилей (RU/EN терминология из GLOSSARY-MASTER соблюдена) | manual spot-check |
| **C12** | Anti-patterns check | Нет анти-паттернов A1-A11 из AGENT-FORMAT (длинные вводные, копии вместо cross-ref, etc.) | heuristic grep + manual |

### 6.4 Сводка

Всего 12 категорий: 5 P0 + 5 P1 + 2 P2. PASS/FAIL/⚠️ по каждой.

## §7. Phase 3 — Semantic cross-checks (7 типов)

Эти проверки сложнее механических. Требуют **ручного** анализа с cross-file контекстом.

### 7.1 Тип S1 — Status-машина consistency между модулями

| Что | Где |
|---|---|
| Модуль КП имеет 8 статусов (DRAFT → SENT → ...) | `03-statusy.md` |
| Модуль Договор имеет свои N статусов | `МОДУЛЬ-ДОГОВОР.md` |
| Проверка: правило «КП→Договор только если КП.PAID» соблюдено в обоих | semantic cross-ref |

**Метод:** Grep переходов PAID→{DRAFT,SIGNED} во всех МОДУЛЬ-доках + ручная проверка покрытия.

### 7.2 Тип S2 — RBAC vs Операции консистентность

| Проверка | Файл |
|---|---|
| RBAC-MATRIX 7 ролей × 30 действий → соответствует ли операциям в МОДУЛЬ-доках? | RBAC-MATRIX.md + каждый МОДУЛЬ |
| Нет операций «в воздухе» (есть в 1 файле, нет в RBAC) | grep + manual |

### 7.3 Тип S3 — Field-уровневая видимость

| Проверка | Файл |
|---|---|
| Поле `price` в каждой сущности — упомянуто в RBAC-MATRIX для всех ролей? | МОДУЛЬ-доки + RBAC-MATRIX + новый КП STUB |
| Если нет — GAP в RBAC | |

### 7.4 Тип S4 — Суммы и расчёты

| Проверка | Где |
|---|---|
| Формула суммы КП = Σ позиций * qty * price | КП + REGISTRY |
| Формула НДС = 20% (или другая ставка) | КП + Финансы |
| Совпадают ли формулы через модули? | semantic check |

### 7.5 Тип S5 — Workflow transitions vs State machines

| Проверка | Где |
|---|---|
| Из `03-statusy.md` КП: разрешённые переходы | КП STUB |
| Из `03-perehody.md` КП: переходы с правилами | КП STUB |
| Консистентность: то что разрешено в одном — разрешено в другом? | manual |

### 7.6 Тип S6 — Money Math precision

| Проверка | Где |
|---|---|
| Хранение денег: decimal vs float? | SCHEMA-CONSOLIDATED + каждая МОДУЛЬ |
| Округление: round-half-even / round-up? | МОДУЛЬ-доки |
| Совместимость? | manual |

### 7.7 Тип S7 — Time logic (сроки, даты)

| Проверка | Где |
|---|---|
| Срок действия КП (если есть): фиксирован / auto-renew? | КП + REGISTRY |
| Дедлайны по модулям не противоречат? | manual |

## §8. Phase 4 — Coverage report

### 8.1 Расчёт покрытия

`coverage_pct = (categories_passed + 0.5 * categories_warn) / total_categories * 100%`

Targets:
- ✅ ≥90% — можно стартовать Phase 2
- ⚠️ 75-89% — pass with warnings
- ❌ <75% — заблокировано

### 8.2 Пример Coverage matrix

```
P0: 4/5 passed, 1 warn = 90%
P1: 4/5 passed, 0 warn, 1 fail = 80%
P2: 2/2 passed = 100%
TOTAL: 10.5/12 = 87.5% → ⚠️ PASS-WITH-WARNINGS
```

### 8.3 Risk zones (топ-5)

Из GAP-LIST отсортировать по severity × uncertainty. Вывести топ-5 наиболее рискованных пунктов.

### 8.4 OK zones (топ-5)

Что прошло идеально — записывается для PR в финальном отчёте.

## §9. Pass / Fail / Block gates

### 9.1 Verdict logic

```
IF (C1..C5 — все 5 P0 = PASS) AND (coverage ≥90%):
    VERDICT = ✅ PASS
ELIF (C1..C5 — все 5 P0 = PASS) AND (coverage ≥75%):
    VERDICT = ⚠️ PASS-WITH-WARNINGS
ELIF (любая из C1..C5 = FAIL):
    VERDICT = ❌ BLOCKED
ELIF (coverage <75%):
    VERDICT = ❌ BLOCKED
```

### 9.2 Что означает каждый verdict

| Verdict | Значение | Что делать PO |
|---|---|---|
| ✅ PASS | Всё идеально, можно стартовать Phase 2 | Запустить Phase 2 |
| ⚠️ PASS-WITH-WARNINGS | Можно стартовать, НО с оговорками | Прочитать GAP-LIST, решить какие warnings принять |
| ❌ BLOCKED | Критические нарушения, Phase 2 ЗАБЛОКИРОВАН | Создать fix-ТЗ, отправить соответствующему рабочему агенту |

### 9.3 Authority аудитора

- Вердикт ❌ **ОБЯЗАТЕЛЕН к исполнению** Буфером-архистратором.
- ⚠️ — может быть overridden PO через PSL в PROJECT-STATE-LOG.
- ✅ — фиксируется и автоматически разблокирует Phase 2.

## §10. Anti-patterns аудитора

### A1. Не править найденные нарушения

Аудитор НЕ редактирует файлы в scope. Только фиксирует в GAP-LIST.

Если нарушение тривиальное (опечатка) → упоминает в GAP-LIST с severity 🟢 P2.

### A2. Не пересматривать закрытые PSL/Q/СПОР

См. CHECKLIST.md §9 «не ломай сделанного». Закрытые IDs остаются закрытыми.

Если аудитор обнаруживает что закрытая Q фактически не решена → ❌ BLOCKED с пометкой «требует эскалации к PO через новый OQ».

### A3. Не выходить за рамки ТЗП-001

Если аудитор считает что нужно проверить ещё что-то → добавляет в §5 «Что НЕ проверил», НЕ расширяет scope молча.

### A4. Не писать "по-моему тут плохо"

Каждое утверждение = конкретный файл + конкретная строка + конкретное правило, которое нарушено.

Format: `❌ CHECK FAIL | file: 01_КП/04-rbac.md:15 | rule: C7 (Rule numbering consistency) | detail: RULE-КП-RBAC-007 duplicate of RULE-КП-RBAC-003`.

### A5. Не пропускать категории "потому что лень"

Каждая из 12 категорий ОБЯЗАНА быть проверена. Если пропущена → §5 «Что НЕ проверил» с пометкой почему.

### A6. Не блокировать "по приколу"

Если coverage 98% — это ✅ PASS, не ❌ BLOCKED. Блокировка только за реальные 🔴 P0 нарушения.

### A7. Не использовать шаблонные фразы

> КАЖДЫЙ gap = уникальное описание проблемы.
>
> ❌ Плохо: «Найдены проблемы в форматировании».
> ✅ Хорошо: «AGENT-METHOD.md §5.6 имеет нумерацию 5.6.1-5.6.6, но §5.4 имеет нумерацию 5.4.1-5.4.5 без 5.4.6 → неконсистентность».

## §12. Acceptance criteria и handoff

### 12.1 Acceptance criteria (аудит считается сданным)

- [ ] Inventory собран полностью (§5)
- [ ] Все 12 категорий static checks выполнены (§6)
- [ ] Все 7 типов semantic checks выполнены (§7)
- [ ] Все 4 deliverables созданы (§3)
- [ ] Verdict выставлен и обоснован
- [ ] Если verdict = ❌ BLOCKED → создан fix-ТЗ на устранение P0 gaps
- [ ] Если verdict = ⚠️ PASS-WITH-WARNINGS → PO проинформирован
- [ ] Если verdict = ✅ PASS → Буфер-архистратор может стартовать Phase 2

### 12.2 Handoff

**Кому:** PO + Буфер-архистратор + (если есть правки) соответствующий рабочий агент (Аналитик / Архитектор).

**Что передать:**
- AUDIT-REPORT.md (verdict)
- GAP-LIST.md (если есть gaps)
- BLOCKING-ISSUES.md (только если есть)
- 02-REPORT-CHANGES.md (машиночитаемый diff)

**Способ:** PSL-NNN в PROJECT-STATE-LOG.md + ссылка из CHECKLIST.md §3.

### 12.3 Когда запускать аудит повторно

- После исправления каждого 🔴 P0 gap
- После завершения Phase 1 Bootstrap Prisma (новый аудит кода)
- После Phase 2 UI (новый аудит UI-spec vs UI-impl)

## §13. Глоссарий ТЗП-специфичных терминов

| Термин | Определение |
|---|---|
| **Audit** | Полная проверка всех артефактов проекта на целостность |
| **Gap** | Конкретное нарушение конкретного правила в конкретном файле:строка |
| **Coverage** | Процент категорий проверок, прошедших успешно |
| **Verdict** | Финальное суждение аудитора (PASS/WARN/BLOCKED) |
| **Inventory** | Полный перечень файлов в scope аудита |
| **Static check** | Механическая проверка (легко автоматизируется) |
| **Semantic check** | Ручная проверка с пониманием cross-file контекста |
| **🔴 P0** | Severity = блокирующая (Phase 2 невозможна) |
| **🟡 P1** | Severity = важная (желательно исправить в ближайшее время) |
| **🟢 P2** | Severity = nice-to-have (можно отложить) |
| **AUDITOR** | Роль агента-надзора (новая роль в Pipeline v6, выше 5 рабочих) |
| **Handoff** | Передача результата аудита следующему звену (PO / рабочий агент) |
