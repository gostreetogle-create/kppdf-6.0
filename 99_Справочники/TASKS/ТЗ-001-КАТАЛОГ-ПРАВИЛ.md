## 0. Контекст и условия задачи

### 0.1 Зачем это ТЗ

Проект KPPDF CRM v6 находится в Phase 0 Decomposition (завершено для КП) → Phase 1 Bootstrap прогонов Аналитика Run 1/5 (в работе). Бизнес-правила модуля КП разнесены по 20 STUB-файлам + 2 сводным справочникам (`99_Справочники/RBAC-MATRIX.md`, `99_Справочники/SCHEMA-CONSOLIDATED.md`). Для остальных 4 модулей (Договор / Производство / Склад / Финансы) правила живут внутри одного большого `МОДУЛЬ-X.md` файла, без извлечения в табличную форму.

**Проблема:** единой машиночитаемой таблицы правил **по всем 5 модулям** не существует. Это означает, что при переходе к коду (Phase 1 Bootstrap Prisma или Phase 2 Mantine UI) каждый разработчик и каждый аналитик будут пересобирать правила с нуля → drift → переделки.

**Решение:** создать **канонический Каталог правил** — единый документ `99_Справочники/REGISTRY-OF-RULES.md`, в котором ВСЕ правила (RBAC, state-machines, валидации, инварианты, cross-module flow, edge cases, нумерация, snapshot-логика, approval-workflows) сведены в таблицы. Этот документ становится **источником истины** для последующей генерации Zod-схем, RBAC-проверок, Vitest-тестов, и UI-rules в Phase 2.

### 0.2 Чего это ТЗ НЕ делает

| Что НЕ делает | Почему |
|---|---|
| Не создаёт новых бизнес-правил | Чтобы не было drift. Если правило не существует в исходных МОДУЛЬ-доках — оно **не попадает** в Каталог. Если правило противоречит 2 документам — фиксируется в `09-AMBIGUITIES.md` для решения PO. |
| Не пишет код | Это документация. Zod-схемы, RBAC-middleware, тесты — задача Phase 1+ Bootstrap, делается по Каталогу позже. |
| Не создаёт новые сущности БД | Не трогает `SCHEMA-CONSOLIDATED.md`. Каталог **описывает** правила над существующей схемой. |
| Не заменяет существующие справочники | Каталог **дополняет** RBAC-MATRIX.md (покрывает в нём только матрицу 7×30; остальные правила — новый материал). SCHEMA-CONSOLIDATED.md остаётся единственным источником схемы. |
| Не редактирует 99_Справочники/OPEN-QUESTIONS-MASTER.md / СПОРНЫЕ-МОМЕНТЫ.md | Это завершённые Q и СПОРЫ (38 ✅ и 15 ✅ — не трогать). |

## 1. Миссия

> **Одной фразой:** Извлечь ВСЕ правила из существующих 5 МОДУЛЬ-документов и сводных справочников в единый машиночитаемый Каталог (`99_Справочники/REGISTRY-OF-RULES.md`), чтобы Каталог стал **источником истины** для Phase 1 Bootstrap и Phase 2 UI-разработки без необходимости повторного изучения правил каждым агентом.

**Декомпозиция миссии (что агент делает по шагам):**

1. **Прочитать** все МОДУЛЬ-доки + сводные справочники (полный список — §5).
2. **Извлечь** правила в 12 категорий (§7 настоящего ТЗ).
3. **Структурировать** каждое правило в табличной форме с ID, формулировкой, источником, следствием при нарушении.
4. **Сверить** между модулями: найти противоречия и пробелы → отдельный файл `09-AMBIGUITIES.md`.
5. **Задокументировать** работу в `01-LOG.md` (для audit trail).
6. **Сдать** отчёт `02-REPORT.md` с метриками качества.

**Гарантия качества:** Каталог покрывает **100%** правил из исходных документов. Каждое правило имеет **машиночитаемый ID** в формате `RULE-{MODULE}-{TYPE}-{NNN}` (например `RULE-КП-INV-001`).

## 2. Контекст проекта (что Агент должен понять перед работой)

### 2.1 Что такое KPPDF CRM v6

KPPDF CRM v6 — система управления коммерческими предложениями (КП), договорами, производственными заказами (ЗК), складом и финансами для малого/среднего производственного бизнеса. Заменяет Excel + email. Целевая аудитория: менеджер по продажам, бухгалтер, директор, производство, кладовщик.

**Стек:** Next.js 16 + React 19 + Mantine 8 + Tailwind 4 + Prisma 7 + PostgreSQL 16 + Zustand + TanStack Query + react-hook-form + Zod 4 + sharp + pnpm + Husky. Полный стек согласован 24.06.2026 в [`99_Справочники/СТЕК-ПРЕДПИСАНИЕ.md`](../СТЕК-ПРЕДПИСАНИЕ.md). Baseline — `kppdf-5.0` (47 моделей, 89 API, 88 unit-тестов).

**5 модулей:**
1. **КП** (Коммерческое предложение) — `01_КП/` — 20 STUB + 1 README. Самый детальный модуль.
2. **Договор** — `02_Договор/` — 1 МОДУЛЬ-док ~402 строки.
3. **Производство (ЗК)** — `03_Производство/` — 1 МОДУЛЬ-док ~405 строк.
4. **Склад** — `04_Склад/` — 3 МОДУЛЬ-дока (обзорный / подробный / UI).
5. **Финансы** — `05_Финансы/` — 1 МОДУЛЬ-док ~280 строк.

## 6. Methodology — как Агент извлекает правила

### 6.1 Общий алгоритм работы

```
1. Прочитать §5.1 приоритетный список (Шаг A → Шаг B → Шаг C).
2. Для каждого МОДУЛЬ-дока пройти структуру:
   2.1. §0 контекст / термины → Занести в REGISTRY §12 Glossary Mapping.
   2.2. §1 §2 ... §N → Найти каждое утверждение вида:
        - «X может / не может Y»       → RBAC Rules (§1)
        - «статус: ... → ...»           → State Machines (§2)
        - «обязательное поле»           → Validation Rules (§3)
        - «нельзя X» (инвариант)        → Business Invariants (§4)
        - «при X → авто-Y в другом модуле» → Cross-Module Triggers (§5)
        - «если Z, то ...» (граничный)  → Edge Cases (§6)
        - «формат номера ...»           → Numbering Schemes (§7)
        - «при создании X копируется ...» → Snapshot Rules (§8)
        - «approver: ...» двухшаговый   → Approval Workflows (§9)
        - «цена = basePrice × qty × (1 - discount) × (1 + margin)...»  → Money Math (§10)
        - «срок действия ...»           → Time & Date Logic (§11)
   2.3. Каждую находку занести в REGISTRY с ID + источник.
3. После каждого модуля — самопроверка по §11 настоящего ТЗ.
4. Все найденные **противоречия** между модулями → `09-AMBIGUITIES.md`.
5. Создать `02-REPORT.md` со сводкой.
```

### 6.2 Один проход = одна порция правил

**Правило «100 правил за проход»:** за один проход через МОДУЛЬ-док Агент должен извлечь **не менее 100 правил**. Если меньше — значит, или документ бедный, или Агент поверхностный. В обоих случаях — повторный проход.

**Правило «каждое правило проверяется через 3 теста»:**
1. Можно ли нарушить это правило через прямой API call? Если да — правило нужно.
2. Видно ли нарушение пользователю в UI? Если да — формулировка UI-поведения обязательна.
3. Есть ли в коде сейчас проверка? Если нет — правило становится TODO для Phase 1 Bootstrap.

### 6.3 Что НЕ считать правилом

| Это НЕ правило | Пример |
|---|---|
| Описательная фраза без ограничения | «Система позволяет создавать КП» — это не правило, это факт. |
| Терминология | «КП = Коммерческое предложение» — не правило, идёт в §12 Glossary. |
| Описание UI без запрета/требования | «На экране КП есть три зоны» — не правило (это layout). |
| Ссылка на другой документ | «См. SCHEMA-CONSOLIDATED.md §3» — не правило, а навигация. |

**Правило = конкретное действие, которое система ДОЛЖНА / НЕ ДОЛЖНА выполнять**, плюс проверка нарушения.

## 11. AMBIGUITIES — как обрабатывать противоречия

### 11.1 Когда создавать `09-AMBIGUITIES.md`

**Только если** найдено реальное противоречие между **двумя и более** источниками. Примеры:

- Документ A говорит «X может Y», документ B говорит «X не может Y».
- Документ A не упоминает правило Z, документ B явно говорит «правила Z нет».
- Документ A: «если X, то Y», документ B: «если X, то Z» (где Y ≠ Z).
- Статус определён 5 раз по-разному в 5 источниках.

### 11.2 Формат каждой записи

```markdown
## PROT-NNN — <краткое название>

| Поле | Значение |
|---|---|
| **ID** | PROT-NNN |
| **Источник A** | `path/to/A.md`, § <раздел>, "<цитата из A>" |
| **Источник B** | `path/to/B.md`, § <раздел>, "<цитата из B>" |
| **Конфликт** | что именно расходится |
| **Предположение** | какая сторона права (если можно предположить) |
| **Рекомендация Агента** | как разрешить (НЕ применять, только рекомендация) |
| **Серьёзность** | 🔴 P0 (контракт меняется) / 🟡 P1 (важно) / 🟢 P2 (терминология) |
```

### 11.3 Что НЕ считать противоречием

| Это НЕ противоречие | Почему |
|---|---|
| Документ A описывает X про КП, документ B описывает X про Договор | Разные модули — нет конфликта. |
| Документ старше документа B | Версия != конфликт (если A был superseded by B). |
| Терминологическое расхождение в одном файле | Дрейф термина внутри файла — баг формата, не правила. |

## 13. Edge cases / known ambiguities (для Агента)

### 13.1 Что делать, если в МОДУЛЬ-доке нет правил

- Это означает, что правил нет в v1, и в REGISTRY они не попадают.
- Если это критический модуль (КП) — добавить в AMBIGUITIES как `PROT-MISS-NNN: в X не описаны правила Y, нужно ли их изобретать?`.

### 13.2 Что делать, если МОДУЛЬ-док противоречит СПОР-решению

- Если 99_СПОР-Ы уже согласовали правило (закрытый СПОР-NNN), а МОДУЛЬ-док говорит иначе → приоритет СПОР-решения.
- Зафиксировать в AMBIGUITIES.

### 13.3 Что делать, если в МОДУЛЬ-доке правило расплывчато

- Привести к машиночитаемой форме (формулировка должна подходить под Zod test).
- Если невозможно → AMBIGUITIES с описанием.

### 13.4 Что делать, если Агент не уверен в категории правила (§1 vs §3)

- **Принцип отбора:** если правило ограничивает ДЕЙСТВИЕ пользователя → §1 RBAC; если правило ограничивает ДАННЫЕ (значение поля) → §3 Validation; если правило = «всегда должно быть true» → §4 Invariant.

### 13.5 Что НЕ делать

- Не придумывать правила «от себя» — только извлечение.
- Не решать за PO — только рекомендации в REPORT.
- Не редактировать SOURSE-файлы — только извлекать.
- Не создавать `.md`-файлы вне указанных в §4.
