## 0. Контекст

### 0.1 Что такое Karkas-Kit и зачем он сейчас

Karkas-Kit = единый UI-патерн проекта KPPDF CRM v6. Название от слова «каркас» (frame/skeleton) — набор стандартных UI-компонентов и поведений, единых для ВСЕГО продукта.

PO (via [`AGENT-METHOD.md` §5.4](../../AGENT-METHOD.md#54)) сформулировал **4 канонических UX-принципа**:
1. **Единообразие** (consistency) — паттерн как в Windows/macOS.
2. **Интеллектуальные подсказки** (smart suggestions) — автокомплит, suggestions, WYSIWYG.
3. **Логичная последовательность** (intuitive flow) — прогрессивное раскрытие, breadcrumbs.
4. **Минимизация трения** (low friction) — авто-сохранение, умные дефолты, shortcuts.

**Без Karkas-Kit каждый разработчик будет изобретать свои dropdown / button / modal** → drift, плохой UX, нарушение принципа 1.

### 0.2 Что в этом ТЗ, чего нет

В этом ТЗ: **только модуль КП** + **Karkas-Kit как общий UI-каркас**.
НЕ в этом ТЗ: код/Mantine реализация (Phase 2 позже).
НЕ в этом ТЗ: компоненты для модулей Договор / ЗК / Склад / Финансы — они построятся по тому же Karkas-Kit позже.

### 0.3 Кто работает по этому ТЗ

**Агент** — параллельный ИИ, роль = **UX-дизайнер** (новая роль, launch-пакет создаётся этим ТЗ).

## 5. Methodology

### 5.1 Алгоритм работы

```
1. Прочитать §4 inputs в порядке 1-11.
2. Для каждого типа компонента (раздел §6):
   2.1. Определить 4 UX-принципа, которые он реализует.
   2.2. Описать canonical форму (стиль, размер, состояния).
   2.3. Перечислить варианты для КП (где применяется, какой action).
   2.4. Указать RBAC-условия видимости (по RBAC-MATRIX.md).
   2.5. Указать state-machine-условия доступности (по 03-statusy.md).
   2.6. Специфицировать hover / focus / active / disabled / loading состояния.
   2.7. Специфицировать hot-keys (shortcuts) если применимо.
3. Привязать к STUB модуля КП (какие STUB используют компонент).
4. Self-verify по §8.
```

```markdown
## 4. Modal — диалог подтверждения действия

### 4.1. Каноническая форма

| Атрибут | Значение | Принцип |
|---|---|---|
| Trigger | destructive action (Удалить / Расторгнуть / Архивировать) | — |
| Header | Заголовок действия ("Удалить КП?") + крестик справа (Esc) | (1) |

| Confirm button | красный danger ("Удалить"/"Расторгнуть"), Enter-shortcut | (1) |
| Backdrop | затемнение 50% + click-outside закрывает ТОЛЬКО если не destructive | (2) |
| Анимация | fade-in 150ms, scale 0.95 → 1.0 | (3) |
| Фокус | autofocus на Cancel (НЕ на Confirm — защита от случайного клика) | (4) |

### 4.2. Где применяется в КП

- Подтверждение удаления позиции из корзины.
- Подтверждение расторжения КП.
- Подтверждение «Конвертировать в Договор» (НЕ destructive, но требует явного подтверждения).
- Подтверждение «Отметить оплачено».

### 4.3. RBAC

Cancel — доступен всем, кто видит trigger. Confirm — по RBAC-таблице действия (по RBAC-КП-A-NNN).
```

### 6.5 Компонент: Toast (notification)

```markdown
## 5. Toast — короткие уведомления

### 5.1. Каноническая форма

| Тип | Цвет | Иконка | Пример | Auto-close |
|---|---|---|---|---|
| success | зелёный | ✓ | "КП сохранён" | 3 сек |
| info | синий | ℹ️ | "Черновик восстановлен из локального хранилища" | 5 сек |
| warning | жёлтый | ⚠️ | "Срок действия КП истекает через 3 дня" | НЕ auto-close (важно) |
| danger / error | красный | ✕ | "Ошибка сохранения. Попробуйте ещё раз." | НЕ auto-close (action required) |

### 5.2. Положение

Всегда внизу справа. НЕ блокирует UI. Можно закрыть кликом.

### 5.3. Поведение
```
