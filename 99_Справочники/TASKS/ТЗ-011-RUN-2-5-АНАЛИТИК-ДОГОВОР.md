## 5. Methodology — как извлекать правила

### 5.1 Алгоритм работы

```
1. Прочитать §4 inputs в порядке 1-21.
2. Для каждого из 4 STUB — пройти структуру 2.2:
   2.1. 04-rbac.md:
        - Видимость ДЕЙСТВИЙ (НЕ полей — их у Договор нет в форме) по 7 ролям.
        - ID формата RBAC-ДОГ-{TYPE}-{NNN} где TYPE ∈ {A (action), OW (ownership), V (visibility), C (conditional)}.
        - Mirror структуру КП RBAC-КП-TYPE-NNN (consistency!).
   2.2. 04-biznes-pravila.md:
        - 9 групп инвариантов Договор.
        - ID формата INV-ДОГ-{GROUP}-{NNN} где GROUP ∈ {PARTIES (стороны), TYPES (типы), SIGN (подписание), EXEC (исполнение), SPEC (спецификация), ⛔⛔ DEPRECATED: use CHAIN-КП per ID-ALIAS-MAP PSL-043 ⛔⛔ CHAIN-KP (связь с КП), CHAIN-PRODUCTION (связь с ЗК/Склад/Финансы), TERM (расторжение), SOFT (архив)}.
        - Hard-link на КП-правила для ⛔⛔ DEPRECATED: use CHAIN-КП per ID-ALIAS-MAP PSL-043 ⛔⛔ CHAIN-KP группы.
        - Cross-ref на МОДУЛЬ-ПРОИЗВОДСТВО для CHAIN-PRODUCTION.
        - Cross-ref на МОДУЛЬ-ФИНАНСЫ для авто-создания Order.
   2.3. 03-statusy.md:
        - 7 статусов Договор + точные определения (не синонимы с КП!) + англ. имя + цвет UI + RBAC.
        - ID формата SM-ДОГ-{NNN}.
        - Явная таблица соответствия КП ↔ Договор статус (если есть mapping).
   2.4. 03-perehody.md:
        - ~10-12 переходов между статусами + триггер + RBAC + preconditions + side-effects.
        - ID формата SM-ДОГ-T-{NNN}.
        - Mermaid stateDiagram-v2 (полная, включая TERMINATED).
        - Negative-rules: запрещённые переходы.
3. Каждое правило: ID + Источник (МОДУЛЬ §NN или КП-RULE-ID-NNN) + Следствие при нарушении.
4. Перепроверить по §8 self-check + §3.3 cross-module consistency.
5. Передать в 11-02-REPORT.md → handoff.
```

### 5.2 Что считать правилом

**Правило = конкретное действие / ограничение / требование**, которое проверяется в коде:

| Тип                      | Пример Договор                                                   |
| ------------------------ | ---------------------------------------------------------------- |
| RBAC (action visibility) | «director может терминировать, manager — НЕ может»               |
| Validation (данные)      | «Contractor.inn обязателен (10 или 12 цифр)»                     |
| Invariant (всегда true)  | «Contract → Архив только после COMPLETED + 90 дней»              |
| State machine (переход)  | «SENT → SIGNED_CLIENT разрешён, SENT → ARCHIVED запрещён»        |
| Side-effect trigger      | «SIGNED_CLIENT → ORDER создаётся в Модуле Финансы автоматически» |

**Не правило:**

- Описательная фраза без ограничения («Договор — это юридический документ»).
- Терминология.
- Layout UI без запрета/требования.

- **Не дублировать правила Модуль-Производство** для CHAIN-PRODUCTION — ссылаться на `МОДУЛЬ-ПРОИЗВОДСТВО.md`.

### 5.4 Cross-module: КП (NEW for ТЗ-011)

Конвертация КП → Договор — это **boundary event** между модулями. Правила, которые её описывают, **уже есть** в КП (INV-КП-CONV-*). В Договор создаются **mirror-правила**, которые ссылаются на оригинал:

| ID КП (canonical)                                    | ID Договор (mirror)                                                                | Что делает                                                                                                                                                          |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `INV-КП-CONV-001` PAID КП нельзя конвертировать      | `INV-ДОГ-⛔⛔ DEPRECATED: use CHAIN-КП per ID-ALIAS-MAP PSL-043 ⛔⛔ CHAIN-KP-001` | Hard-link Source column на `INV-КП-CONV-001` (МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ §14). Дублировать ВСЁ правило НЕ нужно — только специфичные для Договор side-effects. |
| `INV-КП-CONV-002` Снапшот ContractItem.priceSnapshot | `INV-ДОГ-⛔⛔ DEPRECATED: use CHAIN-КП per ID-ALIAS-MAP PSL-043 ⛔⛔ CHAIN-KP-002` | Тот же подход.                                                                                                                                                      |
| `INV-КП-CONV-003` parentProposalId NOT NULL          | `INV-ДОГ-⛔⛔ DEPRECATED: use CHAIN-КП per ID-ALIAS-MAP PSL-043 ⛔⛔ CHAIN-KP-003` | Тот же подход.                                                                                                                                                      |

**Почему mirror вместо full-duplicate:** Если правило изменится в КП, нужна синхронизация обоих мест → риск drift. Hard-link снимает эту проблему.

### 5.5 Cross-module: Модуль-Производство + Модуль-Финансы

Side-effect правила Договор → ProductionOrder + Order живут в других модулях. Правила Договор должны **фиксировать side-effect триггер** (например, «при SIGNED_CLIENT создаётся Order»), но **НЕ детализировать** логику Order (это Модуль-Финансы):

| Договор правило                                                            | Hard-link на                               |
| -------------------------------------------------------------------------- | ------------------------------------------ |
