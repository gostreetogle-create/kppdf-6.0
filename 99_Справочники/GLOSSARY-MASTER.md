# GLOSSARY-MASTER.md — Единый словарь терминов проекта v6

> **Дата:** 2026-06-24.
> **Назначение:** Единый словарь всех терминов, используемых в проекте. Синтезирован из глоссариев 5 модульных документов и сводных справочников.
>
> **Категории:** 📦 Сущность | 🔄 Процесс | 💼 Бизнес-термин | 🛠 Технический | 📊 Метрика

---

## 0. TL;DR — критические термины (must-know)

| Термин | Что значит |
|---|---|
| **КП (Proposal)** | Коммерческое предложение. Префикс `КП-XXXX`. Документ менеджера для клиента. 6 статусов. |
| **Договор (Contract)** | Юридический документ. Префикс `Д-XXXX`. 7 статусов (включая `TERMINATED`). |
| **ЗК (ProductionOrder)** | Производственный заказ. Префикс `ЗК-XXXX`. 8 статусов. Авто-создаётся при «Оплачено» КП. |
| **Order (Финансы)** | Финансовый заказ. Префикс `Order-XXXX`. 5 статусов. Авто-создаётся при подписании Договора. |
| **packageTag** | Уникальный тег сделки (вводится менеджером вручную). Связывает все документы одной сделки через виртуальную «Картотеку». |
| **Snapshot** | Копия значений (цена, наименование, ед.) на момент операции. Защита от ретро-правок. |
| **RUB** | Валюта **жёстко RUB** в v1. Multi-currency — v2. |
| **SBOM (вид Каркас-Kit)** | 3-зонный дизайн-каркас: левая витрина / центральный документ / правая панель. |
| **RBAC** | Управление правами. 7 ролей (admin/director/manager/accountant/production/storekeeper/viewer). |
| **Storno** | Техническая корректировка опечатки (amount < 0, type='STORNO', correctsPaymentId). Деньги не двигались. |
| **Refund** | Реальный возврат денег клиенту (отдельная сущность, amount > 0). |

## 2. Процессы (операции и состояния)

| Термин | Категория | Определение | Где определено |
|---|---|---|---|
| **Конвертация** | 🔄 | Превращение КП в Договор. Кнопка «Конвертировать в договор» в редакторе КП. **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО из статуса `paid`**. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §0 |
| **Оплата КП → создание ЗК** | 🔄 | При `Proposal.status='paid'` система **автоматически** создаёт `ProductionOrder` (ЗК-XXXX). | `МОДУЛЬ-ПРОИЗВОДСТВО.md` §5.1 |
| **Подписание Договора → создание Order** | 🔄 | При `Contract.status='signed'` (СПОР-5) **автоматически** создаётся `Order` в Финансах в статусе `draft` → авто-`in_progress`. | `МОДУЛЬ-ФИНАНСЫ.md` §4 |
| **Завершение ЗК → приход на Склад** | 🔄 | При `ProductionOrder.status='completed'` **автоматически** создаётся `StockMovement: IN, reason=PRODUCTION`. **Только для `Product.kind='ITEM'`** (R1/GAP-009). | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §11.8 |
| **Отгрузка (`Shipment.shipped`)** | 🔄 | При ручном переходе `packed → shipped` **автоматически** создаётся `StockMovement: OUT, reason=SALE`. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §6.4 |
| **Доставка (`Shipment.delivered`)** | 🔄 | При ручном переходе → `OrderClosing.progress += quantityActual` (в v1 ручной, в v2 авто). | `МОДУЛЬ-ФИНАНСЫ.md` §4 |
| **Storno (Payment.type='STORNO')** | 🔄 | Техническая корректировка опечатки бухгалтера. `amount<0`, `correctsPaymentId` FK. Деньги физически не двигались. | GAP-023 ✓, `МОДУЛЬ-ФИНАНСЫ.md` §6 |
| **Refund (возврат денег клиенту)** | 🔄 | Реальный возврат. Создаётся сущность `Refund` с `amount > 0`. Вычитается из `Order.paidAmount` математически. | GAP-023 ✓, `МОДУЛЬ-ФИНАНСЫ.md` §6+§10 |
| **OrderClosing (Закрытие заказа)** | 🔄 | Процедура финальной сверки: все позиции отгружены (`progress`), все счета оплачены, `balance = 0`. При выполнении → `Order.status = CLOSED`. | `МОДУЛЬ-ФИНАНСЫ.md` §4, `05_Финансы/00-spr/00-glossary.md` |
| **Списание (`WriteOff`)** | 🔄 | Двухступенчатый процесс: создатель → approver → авто `StockMovement: OUT, reason=WRITE_OFF`. Авто-маршрутизация по сумме. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §7.4 |
| **Резервирование** | 🔄 | При КП `SENT`/`ACCEPTED` создаётся `Reservation`. `availableQty` уменьшается на sum резервов. При `REJECTED`/`ARCHIVED` — резерв снимается. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §5 + `SCHEMA-CONSOLIDATED.md` §5 |
| **Инвентаризация** | 🔄 | Периодическая сверка. Выявленные расхождения → `WriteOffAct`, reason=`inventory_shortage`/`inventory_surplus`. Требует комиссии (3+ человека). | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §2 сценарий 5 |

## 3. Бизнес-термины

| Термин | Категория | Определение | Где определено |
|---|---|---|---|
| **packageTag** | 💼 | **Уникальный тег сделки, задаваемый вручную менеджером.** Не равен `customerId`. Один клиент может иметь несколько сделок с разными тегами. Связывает все документы одной сделки через виртуальную «Картотеку». | СПОР-10, `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §0+§1.1, `GLOSSARY-MASTER.md` |

| **Версия КП** | 💼 | Новая редакция КП через кнопку «Создать новую версию». Всегда отдельный документ в статусе `DRAFT`. Линейная структура версий (не дерево). | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §0, §7 |
| **Копия для другого юрлица** | 💼 | Пересчитанная копия КП для смены организации-продавца. Авто-применяются НАЦЕНКА и СТАВКА НДС нового юрлица. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §3 |
| **Рамочный договор** | 💼 | Договор без Спецификации (только номенклатура + количество, цены согласовываются позже по факту). `priceWithoutVat = NULL` в ContractItem. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §2 Шаг 5, `МОДУЛЬ-ДОГОВОР.md` |
| **Договор поставки со Спецификацией** | 💼 | Тип Договора по умолчанию. К нему автоматически формируется Спецификация с ценами из КП (snapshot). Отличие от рамочного: `priceWithoutVat != NULL`. | `МОДУЛЬ-ДОГОВОР.md` §7, `02_Договор/00-spr/00-glossary.md` |
| **Шаблон Договора** | 📦 | Визуальный макет А4 документа Договора (аналог шаблона КП). Управляется администратором. | `МОДУЛЬ-ДОГОВОР.md`, `02_Договор/00-spr/00-glossary.md` |
| **Неустойка** | 💼 | Фиксированный процент за просрочку (0.05/0.1/0.15/0.2% в день), настраивается per Договор. | `МОДУЛЬ-ДОГОВОР.md`, `02_Договор/00-spr/00-glossary.md` |
| **Parent Договор (parentContractId)** | 🛠 | Ссылка на старую версию Договора (при создании новой версии). Может быть `NULL` для первой версии. | `02_Договор/00-spr/00-glossary.md`, `03_Производство/00-spr/00-glossary.md` |
| **Дополнительное соглашение** | 💼 | Отложено в v2 (отдельная сущность). В v1: создаётся новая версия Договора (`parentContractId`). | `МОДУЛЬ-ДОГОВОР.md` §10 Q3 |
| **Спецификация** | 💼 | Приложение к Договору с позициями и ценами. Хранится в `ContractItem` (snapshot, read-only в UI). | `МОДУЛЬ-ДОГОВОР.md` §7 |
| **Доступное количество (availableQty)** | 💼 | `quantity − reservedQuantity`. То, что можно физически отгрузить. Главная цифра для менеджера по продажам. | `SCHEMA-CONSOLIDATED.md` §5 |
| **Резерв (Reservation)** | 💼 | «Заморозка» товара под активные КП. Не реальное уменьшение остатка, но `availableQty` уже не включает его. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §4.4 |
| **minStock** | 💼 | Минимальный желаемый остаток на `Product`. При `availableQty < minStock` авто-триггер на `PurchaseRequest`. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §4.5 |
| **Каркас-Kit (Karkas-Kit)** | 🛠 | Глобальный дизайн-каркас приложения. 3-зонный макет. Сначала глобальные компоненты, потом локальная специфика документа. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §1.1 |
| **Открытый вопрос (Q)** | 💼 | Вопрос, требующий решения продакт-оунера. Все 38 Q закрыты по состоянию на 24.06.2026. | `OPEN-QUESTIONS-MASTER.md` |
| **Спор (СПОР)** | 💼 | Зафиксированный в прошлом спор между модулями с явным резолюцией. 15 СПОР-ов закрыты. | `СПОРНЫЕ-МОМЕНТЫ.md` |

## 4. Технические термины (важные для понимания архитектуры)

| Термин | Категория | Определение | Где определено |
|---|---|---|---|
| **RBAC** | 🛠 | Role-Based Access Control. Управление правами по 7 ролям. | `RBAC-MATRIX.md` |
| **Snapshot (фиксация данных)** | 🛠 | Копия значений на момент операции. Применяется ко всем `*Item` через snapshot-поля (`productSku, productName, productUnit, priceWithoutVat, vatRate, и т.д.`). Защита от ретро-правок в справочниках. | `SCHEMA-CONSOLIDATED.md` §0 принцип 3 |
| **Soft delete (`isActive`)** | 🛠 | Логическое удаление (не физическое). Все бизнес-сущности имеют `isActive: Boolean @default(true)`. `isActive=false` = архив. | `SCHEMA-CONSOLIDATED.md` §0 принцип 2 |
| **RESTRICT (ON DELETE)** | 🛠 | Запрет удаления родительской записи, если есть дочерние FK. Применяется к критичным FK (КП→Договор, КП→ЗК, Product→StockRecord, и т.д.). | `SCHEMA-CONSOLIDATED.md` §2 |
| **SET NULL (ON DELETE)** | 🛠 | При удалении родителя FK устанавливается в NULL. Применяется к опциональным FK (StockMovement→Proposal, approvedById в WriteOffAct). | `SCHEMA-CONSOLIDATED.md` §2 |
| **CASCADE (ON DELETE)** | 🛠 | При удалении родителя дочерние удаляются тоже. Применяется к «дочерним» таблицам (`*Item` → родитель). | `SCHEMA-CONSOLIDATED.md` §2 |
| **Safe-increment счётчика** | 🛠 | Защита от race conditions в параллельных транзакциях через `SELECT FOR UPDATE` в `$transaction`. Код форматирует в `КП-0001`. | `SCHEMA-CONSOLIDATED.md` §4.2 |
| **Optimistic Locking** | 🛠 | Через поле `updatedAt`. При сохранении проверяется, не изменился ли `updatedAt` с момента загрузки → если да, предупреждение. | `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` §8 + §. 12 |
| **Self-contained-самостаяние** | 🛠 | BIG-BOOK.md «читая этот файл не нужно лезть в папки за базовой информацией (но ссылки на детали есть)». | `BIG-BOOK.md` |
| **Single-tenant** | 🛠 | v1 архитектура: одна компания в одной БД. Multi-tenancy v2. | `МАСТЕР-АУДИТ-V6.md` §5.4 |

| Термин | Категория | Определение | Где определено |
|---|---|---|---|
| **Маржа (margin)** | 📊 | `totalAmount − costOfGoodsSold`. Доступна только ролям с правом `costPrice:read` (admin/director/accountant). | `МОДУЛЬ-ФИНАНСЫ.md` §6 |
| **Себестоимость (costOfGoodsSold)** | 📊/📦 | **Как метрика:** суммарная себестоимость позиций из Склада. **Как сущность:** поле `Shipment.costOfGoodsSold` — фиксируется при отгрузке. Для услуг/работ `costPrice=NULL`, маржа не считается. | `МОДУЛЬ-ФИНАНСЫ.md` §6, `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §6 |
| **reservedQuantity = SUM** | 📊 | `Σ Reservation.quantity WHERE productId = X AND proposalId IN (active KPs)`. | `SCHEMA-CONSOLIDATED.md` §5 |
| **availableQty = quantity − reservedQuantity** | 📊 | Главная формула остатка для отображения менеджеру. | Везде |
| **Σ Refund.amount ≤ originalPayment.amount** | 📊 | Инвариант: по каждому `originalPaymentId` сумма возвратов не может превысить сумму платежа. | `SCHEMA-CONSOLIDATED.md` §6.2 |
| **paidAmount = Σ INCOMING − Σ STORNO − Σ Refund** | 📊 | Математическая формула net payment для Order. | `SCHEMA-CONSOLIDATED.md` §6.2 |
| **Balance (Сальдо Order)** | 📊 | `totalAmount − paidAmount`. Отрицательное = клиент должен. Положительное = переплата. `balance = 0` — условие закрытия Order. | `МОДУЛЬ-ФИНАНСЫ.md` §4, `05_Финансы/00-spr/00-glossary.md` |
| **Дебиторка** | 📊 | Сумма отрицательных `Order.balance` по всем активным (`IN_PROGRESS` + `AWAITING_PAYMENT`) Order. Контрольная метрика для руководителя. | `МОДУЛЬ-ФИНАНСЫ.md` §6, `05_Финансы/00-spr/00-glossary.md` |
| **Пороги approve WriteOffAct** | 📊 | `< 5000 ₽ → accountant`; `5000-50000₽ → director`; `≥ 50000₽ → director + журнал`. Inventory всегда `director + комиссия 3+`. | `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §7.4 |
