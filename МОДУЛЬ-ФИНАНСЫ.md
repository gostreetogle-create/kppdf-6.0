# Модуль «Финансы / Закрытие заказа» (OrderClosing)

> Этот документ — каркас описания модуля «Финансы / Закрытие заказа». Описывает финальный этап цепочки: после производства и отгрузки — фиксация оплаты, сверка счётов и закрытие сделки. Документ намеренно короткий, чтобы задать контуры — детализация будет позже, по приоритету, как и для всего проекта.
>
> Согласовано с `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`: статус КП «оплачено» → переход в финансовый учёт. Согласовано с `МОДУЛЬ-ДОГОВОР.md`: после подписания договора контроль оплаты переходит в финансы. Согласовано с `МОДУЛЬ-ПРОИЗВОДСТВО.md`: факт отгрузки запускает сверку. Согласовано с `МОДУЛЬ-СКЛАД.md`: списание со склада = себестоимость для финансов.
>
> **Что НЕ делаем в первой версии (границы):** см. раздел 8 в конце. Топ-3 ограничения: нет интеграции с банком/1С, нет частичных оплат по этапам, нет авто-перевыставления счетов.

## 0. Глоссарий

| Термин | Что значит |
|---|---|
| **Заказ (Order)** | Сквозная сущность «от КП до оплаты». Один Order = один КП (или одно КП → один Договор → один/несколько ЗП → одна Отгрузка → один/несколько Платёж). Создаётся автоматически при конвертации КП в договор. |
| **Счёт (Invoice)** | Документ выставления оплаты клиенту. Содержит позиции, сумму к оплате, срок. Может быть авансовым или финальным. |
| **Платёж (Payment)** | Факт поступления денег от клиента. Привязан к Счёту (один Счёт → один или несколько Платежей — частичная оплата). |
| **Закрытие заказа (OrderClosing)** | Процедура финальной сверки: все позиции отгружены, все счета оплачены, остаток = 0. После закрытия заказ уходит в архив для аналитики. |
| **Сальдо (Balance)** | Сумма разницы между выставленным (по счетам) и полученным (по платежам). Отрицательное = клиент должен. Положительное = переплата. |
| **Себестоимость заказа** | Суммарная себестоимость всех позиций, рассчитанная модулем Склад при отгрузке. Используется для маржинальности. |

## 1. Для чего нужен этот модуль

**Финансы / Закрытие заказа** — это финальный этап жизненного цикла сделки. Модуль решает четыре задачи:

1. **Фиксация оплаты** — при поступлении денег от клиента менеджер (или бухгалтер, в зависимости от роли) регистрирует платёж.
2. **Контроль дебиторки** — на любой момент времени видно, сколько каждый клиент должен и по каким счетам.
3. **Расчёт маржинальности** — сравнение цены продажи (из КП/Договора) и себестоимости (из Склада). Без этого непонятно, сколько мы реально заработали.
4. **Закрытие заказа** — формальная процедура, которая гарантирует: всё отгружено, всё оплачено, заказ можно передать в архив.

Без этого модуля мы получим ситуацию: КП сделано, договор подписан, производство выполнено, товар отгружен — а бухгалтерия не знает, пришли ли деньги, и сколько мы заработали.

## 2. Для кого он работает

| Роль | Что делает в модуле |
|---|---|
| **Менеджер по продажам** | Видит свой пайплайн: какие счета ждут оплаты, какие КП в архиве. Не имеет права редактировать платежи. |
| **Бухгалтер** | Регистрирует входящие платежи, сверяет с банковской выпиской (вручную, в первой версии), формирует итоговые отчёты. |
| **Руководитель / директор** | Видит сводную аналитику: маржинальность по сделкам, дебиторка по клиентам, динамика выручки. |
| **Администратор** | Настраивает справочники (ставки НДС, валюты, типы счетов), если такие сущности понадобятся в первой версии. |

## 3. Жизненный цикл

### Статусы Заказа (Order)

```
        ┌─────────────────┐
        │ Черновик        │  ← авто-создаётся при конвертации КП в Договор
        └────────┬────────┘
                 │ (договор подписан → Договор в статусе «Подписан»)
                 ▼
        ┌─────────────────┐
        │ В работе        │  ← идёт производство / ожидаем отгрузку
        └────────┬────────┘
                 │ (отгрузка выполнена → Склад: все позиции shipped)
                 ▼
        ┌─────────────────┐
        │ Ожидает оплаты  │  ← выставлен финальный счёт
        └────────┬────────┘
                 │ (сальдо = 0)
                 ▼
        ┌─────────────────┐
        │ Закрыт          │  ← всё оплачено, заказ уходит в архив
        └─────────────────┘
```

### Таблица переходов

| Статус | Следующий | Триггер | Кто может перевести |
|---|---|---|---|
| Черновик | В работе | Договор подписан клиентом | Автоматически (при изменении статуса Договора) |
| В работе | Ожидает оплаты | Склад пометил все позиции как «отгружено» | Автоматически |
| Ожидает оплаты | Закрыт | Сальдо = 0 | Бухгалтер (или автоматически при заведении платежа) |
| Закрыт | — | (архив) | — |

### Статусы Счёта (Invoice)

| Статус | Описание | Когда |
|---|---|---|
| **Черновик** | Счёт создан, но не выставлен | Менеджер/бухгалтер готовит счёт |
| **Выставлен** | Счёт отправлен клиенту | После нажатия «Отправить счёт» |
| **Частично оплачен** | Поступила часть суммы | Один или несколько платежей < суммы счёта |
| **Полностью оплачен** | Сальдо = 0 | Все платежи поступили |
| **Просрочен** | Срок оплаты прошёл, сальдо > 0 | Платёж не поступил до dueDate |

## 4. Ключевой сценарий: «От подписания договора до закрытия заказа»

1. **Договор подписан** (Договор переходит в статус «Подписан»).
   - Автоматически создаётся **Order** в статусе «В работе».
   - Order связан с Договором (`contractId`), с КП (`proposalId`), с будущими Счетами и Платежами.
2. **Бухгалтер выставляет счёт.**
   - Первичный авансовый счёт (например, 50% от суммы договора).
   - Или один счёт на всю сумму — в зависимости от условий договора (`paymentTermDays`).
3. **Менеджер отправляет счёт клиенту** (email / PDF).
   - Счёт переходит в статус «Выставлен».
4. **Бухгалтер регистрирует входящий платёж.**
   - Сальдо пересчитывается. Если = 0 → счёт «Полностью оплачен».
   - Если в Order все счета оплачены → Order переходит в «Ожидает оплаты» → «Закрыт».
5. **Руководитель смотрит аналитику.**
   - Маржа = (сумма продажи по КП) − (себестоимость всех позиций из Склада).
   - Отчёт по дебиторке = сумма отрицательных сальдо по всем активным Order.

### Связь с предыдущими модулями

```
[МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ]   → статус «оплачено» → триггер для финансов
[МОДУЛЬ-ДОГОВОР]                     → подписан → авто-создание Order
[МОДУЛЬ-ПРОИЗВОДСТВО]                → все позиции выполнены → триггер для финансов
[МОДУЛЬ-СКЛАД]                       → все позиции shipped → передача себестоимости в финансы
[МОДУЛЬ-ФИНАНСЫ (этот)]              → фиксация оплаты → закрытие Order → маржа → архив
```

**⚠️ Уточнение (АНАЛИЗ-П1):** Триггер создания Order — **подписание Договора** (когда Contract.status = 'signed'). Order создаётся в статусе 'draft' и автоматически переходит в 'in_progress'. Счёт (Invoice) создаётся отдельно бухгалтером, уже привязанный к Order.

## 6. Бизнес-правила

### RBAC (роли и права)

| Действие | Менеджер | Бухгалтер | Директор | Админ |
|---|---|---|---|---|
| Видеть свой список Order | ✅ | ✅ (всех) | ✅ (всех) | ✅ (всех) |
| Создавать/редактировать счёт | ❌ | ✅ | ✅ | ✅ |
| Регистрировать платёж | ❌ | ✅ | ✅ | ✅ |
| Закрывать заказ | ❌ | ✅ | ✅ | ✅ |
| Видеть маржинальность | ✅ (свою) | ✅ | ✅ (всех) | ✅ |
| Видеть дебиторку по клиенту | ❌ | ✅ | ✅ | ✅ |

### Правила расчёта

- **Сальдо = Сумма счёта − Σ Платежей по этому счёту.** Пересчитывается при каждом новом платеже.
- **Маржа заказа = Σ (цена продажи позиции × кол-во) − Σ себестоимости позиций из Склада.** Себестоимость доступна только когда позиция прошла через Склад (`shipped`).
- **Просроченный счёт** считается просроченным на следующий день после `dueDate` при сальдо > 0. В списке заказов выделяется красноватым.

### Правила целостности

- **Заказ нельзя закрыть, если хотя бы один счёт имеет сальдо > 0.** Кнопка «Закрыть заказ» заблокирована, рядом — список незакрытых счетов.
- **Платёж нельзя удалить**, только **сторнировать** (создать компенсирующий платёж с отрицательной суммой). Это сохраняет аудит-трейл.

> **✅ СПОР-7 (24.06.2026, фиксация):** При регистрации платежа бухгалтер **вручную выбирает, к какому счёту (Invoice) привязан платёж** через dropdown «Привязать к счёту». Если клиент прислал платёж на несколько счетов — бухгалтер вводит несколько Payment, по одному на счёт, или один Payment без указания invoiceId (привязка к Order в целом). Авто-FIFO (аванс → финальный) отложено в v2. Связь 1 Payment ↔ 1 Invoice базовая; 1 Payment ↔ N Invoice — для частичных оплат.

### Связь с предыдущими модулями

- **Order ↔ КП:** при закрытии Order — КП остаётся в статусе «оплачено». КП **не возвращается в «принято»**.
- **Order ↔ Договор:** если Договор расторгнут, Order переходит в **спецстатус «Расторгнут»** (не «Закрыт»). Частично оплаченные счета отмечаются отдельно для разбирательства.
- **Order ↔ Склад:** себестоимость берётся только для **фактически отгруженных** позиций. Если в КП была услуга (монтаж) — отдельная строка в марже, но без себестоимости.
- **Order ↔ Финансы:** если в Order попала позиция со **сложной составной себестоимостью** (сборный товар с модулями) — берётся **итоговая себестоимость** из модуля Склад (уже агрегированная после производства).

## 7. Что в первой версии, а что нет

### ✅ Делаем в v1

- Список заказов с фильтрами и поиском.
- Карточка Order с таблицами счетов и платежей.
- Создание счёта вручную (бухгалтер/директор).
- Регистрация платежа вручную (без банковской интеграции).
- Автоматический пересчёт сальдо.
- Расчёт маржинальности по отгруженным позициям.
- Закрытие заказа с проверкой сальдо.
- Базовый отчёт: дебиторка по клиентам, маржа по менеджерам.

### ❌ Не делаем в v1

- **Интеграция с банком / 1С / банк-клиентом.** Все платежи — ручной ввод по выписке.
- **Частичные оплаты по этапам** (предоплата 30% → производство → доплата 70% → отгрузка). В v1 только один финальный счёт или простой аванс.
- **Авто-перевыставление счетов** при просрочке. Только красная подсветка, менеджер сам решает.
- **Финансовая аналитика** (NPV, графики помесячно, прогноз поступлений). Только списки и простые сводки.
- **Курсы валют.** В v1 работаем только в одной валюте (₽). Мультивалюта — отдельная задача.

> **✅ СПОР-14 (24.06.2026, фиксация):** Поле `Order.currency` и `Invoice.currency` жёстко `'RUB'` в v1 (default в Prisma-схеме). UI не показывает dropdown выбора валюты. Мультивалютность (USD/EUR/KZT) отложена в v2 — потребует доработку Payment.amount (валюта) и финансовых отчётов (конвертация по курсу).
- **Возврат денег (refund).** Только сторнирование (отрицательный платёж).

## 8. Открытые вопросы (требуют решения продукта)

> Эти вопросы нужно решить ДО начала кодинга модуля. Каждый из них влияет на структуру базы данных.

1. **Какие типы счетов поддерживать?** Только «оплата по договору»? Или отдельно «аванс», «доплата», «компенсация возврата»?
2. **Как учитывать НДС в платежах?** Платёж приходит одной суммой — система сама разделяет на «в т.ч. НДС» и «без НДС» по ставке из договора? Или бухгалтер указывает явно?
3. **Допускаются ли отрицательные позиции в счёте** (корректировочный счёт после возврата)? Или это всегда отдельный документ?
4. **Нужна ли связь Order ↔ банковская выписка** в первой версии (хотя бы файл-вложение PDF выписки)? Или только ручной ввод?
5. **Кто имеет право «закрыть заказ» принудительно** — например, если счёт просрочен на 6 месяцев и клиент не выходит на связь (списание долга)? Директор? Бухгалтер? Или новая роль «Финансовый контролёр»?
6. **Маржинальность: видно ли менеджеру свою маржу** — или это считается коммерческой тайной и доступно только директору?
7. **Что считать «датой оплаты»** для учёта в периоде — дата поступления в банк, дата регистрации платежа в системе, или дата, указанная менеджером?
8. **Сценарий: клиент оплатил, но позже потребовал возврат.** Есть ли в v1 процедура «refund» (отрицательный платёж + корректировочный счёт + возврат товара на склад)? Или это за рамками v1?

## 11. Сценарии, которые нужно проверить в ИИ-симуляции

> Этот раздел — подсказка для прогона по `ВЕРИФИКАЦИЯ-ЧЕКЛИСТ.md`. По этим сценариям симулятор должен найти конкретные пробелы.

- **SC-01:** Клиент оплатил 80% суммы, остаток обещал через неделю. Можно ли закрыть заказ? *(должно быть «нет»).*
- **SC-02:** Бухгалтер ошибся и внёс платёж с неправильной суммой (забыл ноль). Как исправить? *(должна быть процедура сторнирования).*
- **SC-03:** Договор расторгнут, но клиент уже заплатил аванс. Что с Order? *(должен перейти в спецстатус «Расторгнут», не «Закрыт»).*
- **SC-04:** КП содержало составной товар (из 5 модулей), себестоимость которого собирается только после производства. Как до этого момента считается маржа? *(должна быть `null` или «не рассчитана» — не додумывать).*
- **SC-05:** В Order попала услуга «Монтаж» (нет себестоимости). Маржа считается только по товарам или по всему заказу? *(по товарам — услуги вычитать не из чего).*
- **SC-06:** Один Order, два счёта (аванс + финальный). Платёж пришёл один на всю сумму. Как система распределит — на первый счёт или второй? *(требует правила).*
- **SC-07:** Менеджер уволился. Что происходит с его активными Order? *(переназначение — ответственность директора).*
