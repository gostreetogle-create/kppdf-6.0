# Модуль «Производственный заказ»

> Это документ модуля «Производственный заказ» в шестой версии проекта. Описан так же, как `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`: простым языком для нового менеджера. Если попалось непонятное место — задайте вопрос в файле `ОТКРЫТЫЕ-ВОПРОСЫ.md`.
>
> → **Связанные модули:** модуль КП (откуда приходит сигнал «Оплачено»), модуль Договор (parent contractId для справки), модуль Склад (приём при завершении), модуль Финансы (OrderClosing по завершённому заказу). Границы MVP — раздел 9.

## 0. Глоссарий терминов модуля

| Термин | Что значит |
|---|---|
| **Производственный заказ (ProductionOrder)** | Внутренний документ на изготовление. Уникальный номер **ЗК-XXXX**. Создаётся автоматически при оплате КП. |
| **Задача (ProductionTask)** | Атомарная часть производственного заказа: «выплавить 100 деталей», «собрать корпус», «выполнить монтаж у клиента». Одна ЗК → несколько Задач. |
| **Цех (Workshop)** | Место, где выполняется задача. В первой версии фиксированный список (например: «Цех металлоконструкций», «Сборочный цех», «Выездная бригада»). |
| **Ответственный** | Сотрудник (User с ролью `production`), назначенный на задачу. |
| **Частичная готовность** | Задача может быть выполнена «частично» (например, 50 из 100 деталей). Документируется в фактКоличество. |
| **Parent КП (parentProposalId)** | Ссылка на КП, при оплате которого создан Производственный заказ. Строгий FK, не NULL. |
| **Parent Договор (parentContractId)** | Ссылка на Договор (если КП был сконвертирован). Не строгий FK, может быть NULL (если КП ещё не сконвертирован, но уже оплачено). |

## 1. Для чего нужен этот модуль

**Производственный заказ** — это «мостик» между Клиентом (деньги заплатил) и Производством (пора делать). До модуля Производство менеджер по продажам сам координировал работу с цехом: ходил, звонил, спрашивал «готово ли?». Это отнимало время и приводило к потере информации.

### Для кого он работает

| Роль | Что делает в модуле |
|---|---|
| **Менеджер по продажам** *(в части КП)* | Не работает с модулем напрямую — только создаёт триггер, отмечая КП «Оплачено» |
| **Начальник производства** | Видит очередь заказов, распределяет по цехам, ставит ответственных, контролирует сроки |
| **Мастер цеха** | Берёт задачу, отмечает «В работе», фиксирует частичное/полное выполнение |
| **Администратор системы** | Настраивает справочник цехов, роли `production`, шаблоны задач |

### Главная боль, которую решает

Прошлая версия (5.0) уже умела делать авто-создание заказа при оплате КП. Боль была в **видимости и контроле**: начальник производства не видел очереди заказов в одном месте, не знал, кто и что сейчас делает. Модуль решает это через единую доску задач с цветовой разметкой по статусам и дедлайнам.

## 2. Жизненный цикл Производственного заказа

Статусов ЗК — восемь. Начальник производства следит за переходами.

### Полная карта статусов

| Статус | Что значит | Кто двигает | Условие перехода |
|---|---|---|---|
| **Создан** | Только что создан (триггер от КП «Оплачено»). Задачи ещё не распределены по цехам | авто-система | автоматически при «Оплачено» КП |
| **Планируется** | Начальник производства смотрит, думает какие цеха и сроки | начальник производства | вручную после «Создан» |
| **Ожидает материалов** | Материалов/комплектующих не хватает на складе | кладовщик / начальник | если StockRecord < required |
| **В работе** | Задачи распределены, мастера выполняют | мастер | вручную |
| **Часть готова** | Часть позиций/задач выполнена (>= 50%), но не все | мастер | автоматически по фактКоличество |
| **Завершён** | Все позиции выполнены, изделия готовы | начальник | автоматически когда все фактКоличество = quantity |
| **Отменён** | Заказ отменён (например, клиент отказался после оплаты, или брак) | начальник производства / директор | вручную |
| **Архив** | Заказ завершён давно (или отменён), переведён в архив | авто или ручной админ | вручную или через 90 дней |

**Диаграмма переходов (ASCII):**

```
         ┌─────────────┐
         │  [Создан]   │ ← автоматически из КП «Оплачено»
         └──────┬──────┘
                │ начальник берёт в работу
                ▼
         ┌─────────────┐
         │[Планируется]│
         └──────┬──────┘
                │ материалы ОК / мастер взял
                ▼
         ┌─────────────┐
         │ [В работе]  │ ◀──── если материалов не хватает
         └──────┬──────┘      → [Ожидает материалов]
                │ ≥50% готово
                ▼
         ┌─────────────┐
         │[Часть готова]│
         └──────┬──────┘
                │ всё готово
                ▼
         ┌─────────────┐
         │ [Завершён]  │ ← готово → автоматический приход на Склад
         └──────┬──────┘
                │ через 90 дней ИЛИ архивировать вручную
                ▼
            [Архив]

  В любом статусе (кроме «Создан» и «Архив»): можно перевести в «Отменён»
```

### Важные правила переходов

- **«Завершён» → автоматический приход на Склад**: при переходе ЗК в «Завершён» система автоматически создаёт `StockMovement: Приход` для каждого изделия в ЗК → идёт на Склад.
- **«Отменён» → что с КП?** — открытый вопрос V-027. См. раздел 8.
- **«Архив» — мягкое удаление**, через `isActive = false` (а не `archivedAt` — единообразно с КП/Договором).

## 3. КЛЮЧЕВОЙ СЦЕНАРИЙ: Жизнь одного производственного заказа

**Ситуация.** Менеджер Иванов оформил КП-0042 «Баскетбольное кольцо 1200мм, оцинкованное, 15 шт. с монтажом» для ООО «Спорткомплект». КП отправлен → клиент ответил «да, согласны» → КП принят → Менеджер оформил Договор Д-0042 → Клиент перевёл оплату на р/с.

**Что происходит дальше:**

1. Менеджер Иванов открывает КП-0042, в правой панели нажимает **«Отметить как оплаченное»**.
2. Подтверждение: «Подтверждаете оплату от клиента? Будет создан производственный заказ» → Да.
3. **Система автоматически:**
   - Меняет статус КП-0042 на **«Оплачено»**.
   - Создаёт новый документ **ЗК-0023** (нумерация по порядку, может НЕ совпадать с КП-0042 — это разные счётчики).
   - Статус ЗК-0023 = **«Создан»**.
   - Копирует позиции из КП (по 1 строке на каждый «товар» и 1 строке на «монтаж»). В монтаже появится тип `Installation`.
4. Начальник производства Петров заходит в `/production`, видит новый ЗК-0023 на доске.
5. Нажимает «Взять в работу», распределяет:
   - Изготовление колец (15 шт.) → цех «Металлоконструкции», мастер Сидоров, план на 5 дней.
   - Монтаж у клиента (1 выезд) → цех «Выездная бригада», дата через неделю после готовности.
6. Статус меняется на **«В работе»**.
7. Через 5 дней Сидоров отмечает: «Готово 15 из 15».
8. Бригада монтажа выезжает, через 2 дня отмечает «Монтаж выполнен».
9. Начальник производства ставит **«Завершён»**.
10. Система автоматически создаёт StockMovement: Приход (15 колец на основной склад).

### Что НЕ происходит (но многие ждут)

- **Начальник производства не выписывает сдельную ЗП** мастерам — это вне модуля (HR-функция).
- **Начальник не распределяет сдельную стоимость работ** — у нас идёт через модуль Финансы (себестоимость, разница маржи).

**⚠️ Уточнение (АНАЛИЗ-П1):** Задачи типа «Услуга» (монтаж) и «Работа» (проектирование) после выполнения **фиксируются как завершённые** в статусе `done`, но **не создают StockMovement** (в склад не приходят). Их выполнение учитывается только через прогресс ЗК (% выполненных задач). Связь с Финансами — через себестоимость услуги в OrderClosing (если услуга имела себестоимость).

Фильтры:

- Количество план / факт
- Цех
- Ответственный
- Кнопки: «Начать работу», «Частично выполнено (ввести фактКоличество)», «Готово», «Сообщить о проблеме»

**Не в первой версии** — отложено. См. раздел 9.

## 5. Бизнес-правила

### 5.1 Правила создания ЗК

1. **Триггер — статус КП «Оплачено».** Только из оплаченного КП создаётся ЗК. Никаких других путей создания ЗК нет (вручную создать ЗК нельзя).
2. **Один КП = один ЗК.** Повторно создать нельзя. После «Оплачено» второй раз нажать — ошибка.
3. **ЗК создаётся автоматически — статус «Создан».** Никаких ручных корректировок на этом этапе (только начальник производства в «Планируется» может править).
4. **Позиции копируются из КП как snapshot.** Каждое наименование, количество, ед. измерения — фиксируются в момент создания ЗК. Изменение справочника товаров после не влияет.

### 5.2 Правила для Задач (ProductionTask)

1. **Одна позиция КП может породить одну или несколько задач.** Например «Кольцо 15 шт.» → одна задача «Изготовить 15 шт. в Цехе металлоконструкций».
2. **«Монтаж» (тип «Услуга») → отдельная задача** в цехе «Выездная бригада», с датой после готовности основной продукции.
3. **Тип «Работа» → задача** в соответствующем цехе (например, «Разработка чертежа» → цех «Конструкторский отдел»).
4. **Частичная готовность** — мастер может отметить «готово 7 из 15». Система пересчитывает прогресс.
5. **[GAP] V-015**: Услуги и работы — должны ли они попадать в ЗК? Варианты:
   - A: Да, все позиции КП попадают в ЗК как задачи (просто разные цеха). Логично и единообразно.
   - B: Только товары, услуги/работы вне производства. Усложняет логику «по услугам не нужен заказ».
   - **Рекомендация продакт-оунера**: вариант A (проще).

### 5.3 Правила для переходов

1. **«Завершён»** — ставится автоматически когда ВСЕ задачи ЗК выполнены (фактКоличество = quantity).
2. **«Отменён»** — ТОЛЬКО начальник производства или директор. Повышенные права.
3. **«Архив»** — автоматически через 90 дней после «Завершён» или «Отменён». Вручную — админ.

### 5.4 Правила для Цехов

1. Цех — фиксированный справочник (настраивается админом).
2. Мастер может работать в нескольких цехах (если указано в его профиле), но обычно один основной.
3. Задача привязана к одному цеху. Если задача разделяется между цехами — создавать две задачи.

## 6. Связи с другими модулями

### 6.1 Связь с КП (главная)

- Создаётся автоматически при `Proposal.status = "оплачено"`.
- `ProductionOrder.parentProposalId → Proposal.id` (FK, RESTRICT — нельзя удалить КП, пока есть ЗК).
- **Обратная связь из ЗК → КП в статусе «Завершён»? — НЕТ в первой версии.** КП остаётся в «Оплачено» (отдельный вопрос Q2 в разделе 8).

### 6.2 Связь с Договором

- `ProductionOrder.parentContractId → Contract.id` (FK, не строгий — может быть NULL).
- Заполняется автоматически при создании ЗК, если КП был сконвертирован в Договор.
- Если КП ещё не сконвертирован (например, оплата пришла до подписания договора) — поле NULL.

### 6.3 Связь со Складом — КРИТИЧНАЯ

> **Ссылка на полную спецификацию:** см. [`../04_Склад/МОДУЛЬ-СКЛАД.md`](../04_Склад/МОДУЛЬ-СКЛАД.md) (обзорный каркас) → [`МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md`](../04_Склад/МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md) §3 (StockMovement) + §6 (Shipment) + [`../99_Справочники/SCHEMA-CONSOLIDATED.md`](../99_Справочники/SCHEMA-CONSOLIDATED.md) §3.5 (enum StockMovementType).

- **При переходе ЗК в «Завершён»** автоматически создаётся `StockMovement: type='IN', reason='PRODUCTION'` для каждой задачи-товара (`Product.kind='ITEM'`). Услуги (`SERVICE`/`WORK`/`INSTALLATION`) в склад НЕ приходят (закреплено в `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §3 правило 6).
- **Кладовщик видит** ЗК как ожидаемую поставку ещё до завершения (в статусе «В работе» видна «плановая дата прихода» через `StockMovement.expectedAt`).

### 6.4 Связь с Финансами

- `OrderClosing.productionOrderId → ProductionOrder.id` (FK, основной).
- Финансы ЗАВИСЯТ от ЗК: OrderClosing можно создать только когда ЗК = «Завершён» (по archive/ANALYSIS-KP-priority.md, ON DELETE SetNull для OrderClosing — если ЗК удаляется, OrderClosing остаётся с NULL).

### 6.5 Связь с Картотекой сделки (ранее — «Пакет документов»)

- **Картотека = ВИРТУАЛЬНОЕ ПРЕДСТАВЛЕНИЕ** а не отдельная сущность в БД. Все документы сделки (КП, Договоры, ЗК) опционально тегируются общим `packageTag` (по `customerId`), директор/админ через фильтр `packageTag=X` видят «всё по сделке» в одном окне. Никаких FK `coveredByPackageId` — это было архитектурно избыточно.

## 10. Открытые вопросы — ✅ ВСЕ 6 ЗАКРЫТЫ (24.06.2026)

> Источник: `OPEN-QUESTIONS-MASTER.md`. Каждое решение применено в §5 (бизнес-правила) + §6 (связи с модулями) + §8 (RBAC).

(Историческое содержание раздела сохранено ниже для архивных целей.)

> **✅ СПОР-12 (24.06.2026):** При отмене ЗК **статус КП остаётся «Оплачено»**, возврат средств происходит через **отдельную сущность `Refund`** в Модуле Финансы (формальный канон — см. `МОДУЛЬ-ФИНАНСЫ.md` §6 «Правила Refund» + §10 «Поля Refund» + `SCHEMA-CONSOLIDATED.md` §6.1 «Инвариант Refund»). Refund НЕ реализуется через отрицательный `Payment` (это другая сущность Сторно для корректировки опечатки бухгалтера). Финансы — единая точка правды для денежных движений. Альтернатива «вернуть КП в «Принято»» — отклонена, так как юридически деньги были переведены.

### Q1 (V-027). ✅ РЕШЕНО: B (финальное — СПОР-12 + GAP-023 ✅)

**Решение:** КП остаётся «Оплачено». ЗК → Cancellation передаёт сигнал в Модуль Финансы → бухгалтер оформляет `Refund` (отдельная сущность, строго > 0, вычитается математически из `Order.paidAmount`). `ProductionOrder.status = CANCELLED` → сигнал в Финансы через UI-флоу. Прямой FK `Refund.linkedProductionOrderId` для трассировки (см. МОДУЛЬ-ФИНАНСЫ.md §10). Если ЗК отменён ДО оплаты (статус ЗК = CREATED без оплаты) → Refund не оформляется, КП возвращается вручную через «Создать новую версию» (см. GAP-011). Подробнее в `OPEN-QUESTIONS-MASTER.md` + `ЖУРНАЛ-ПРОГОНА.md` GAP-023.

(Описание вопроса ниже — сохранено для истории.)
- Если ЗК отменён (например, после оплаты клиент захотел вернуть деньги), что с КП и с деньгами?
- **Вариант A:** КП возвращается в «Принято», со специальным ярлыком «Производство отменено».
- **Вариант B:** КП остаётся в «Оплачено». ЗК → Cancellation передаёт сигнал в Модуль Финансы → бухгалтер оформляет Refund.
- **✅ РЕШЕНО: B (финальное, 24.06.2026 — СПОР-12 + GAP-023 ✅).** КП остаётся «Оплачено». Финансы оформляют `Refund` (отдельная сущность, строго > 0, вычитается математически из `Order.paidAmount`). `ProductionOrder.status = CANCELLED` → сигнал в Финансы через UI-флоу: начальник производства отменяет ЗК → видит «Бухгалтер, оформите Refund» в карточке Order. Прямой FK `Refund.linkedProductionOrderId` для трассировки (см. МОДУЛЬ-ФИНАНСЫ.md §10). Дополнительно: если ЗК отменён ДО оплаты (статус ЗК = CREATED без оплаты) → Refund не оформляется, КП возвращается вручную через «Создать новую версию» (см. GAP-011).
