# AGENT-ENTRYPOINT.md — Точка входа для ИИ-агентов

> **Назначение.** Этот документ — **главная входная точка** для любого агента (ИИ или человек), который заходит в проект KPPDF CRM v6 чтобы работать с теоретической базой (чертёж приложения).
>
> **Когда читать:** ПЕРВЫМ, до любых других документов проекта.
>
> **Главная цель проекта:** никакого кода. Только теоретическая часть. Все артефакты — обычные текстовые `.md` файлы с описаниями.

---

## 0. Про что этот документ

Этот файл отвечает на три вопроса:

1. **Кто ты** (какую роль берёшь на себя — Архитектор? Бизнес-аналитик? и т.д.)
2. **Как работать** (по какому методу проходишь по проекту — что сначала, что потом)
3. **Когда «готово»** (по каким критериям оцениваешь свою работу перед сдачей)

Полные ответы — в 5 топических файлах (см. §2 «Навигация»).

### 0.1 PROJECT-STATE-LOG.md — журнал инкрементальных изменений

Каждое глобальное изменение проекта (изменение типа поля, добавление нового статуса, переименование папки, новая политика) **должно быть записано** в файле [`PROJECT-STATE-LOG.md`](PROJECT-STATE-LOG.md) в корне проекта с указанием: **Дата** (YYYY-MM-DD) / **Тип** (schema/process/structure/terminology/critical_fix) / **Модуль** / **Описание** / **Причина** / **Затронутые файлы** / **Автор** / **Связанные OQ**.

**Правило:** каждый агент при старте читает `AGENT-ENTRYPOINT.md` и **`PROJECT-STATE-LOG.md`** (последние 10 записей). Это даёт контекст «куда ветер дует» без перечитывания всей базы.

## 1. Контекст проекта

### 1.1 Что такое KPPDF CRM v6

**KPPDF CRM v6** — это система управления коммерческим циклом для компании, занимающейся производством и поставкой спортивного оборудования. Пять модулей в порядке зависимостей:

```
КП (Коммерческое предложение)
   ↓ конвертация
Договор (Contract)
   ↓ подписание → авто-Order
Производственный заказ (ЗК)
   ↓ завершение → авто-приход
Склад (Warehouse, Shipment)
   ↓ отгрузка → авто-расход
Финансы (Order, Invoice, Payment, Refund)
```

Сейчас проект в **фазе Phase 0 (теоретическая подготовка)**. Код не пишем. Все артефакты — `.md` файлы.

### 1.2 Главная боль

| Было | Стало (после v6) |
|---|---|
| 15 минут на «КП для другого юрлица» | 30 секунд + авто-применение наценки/НДС |
| Долгая сборка корзины товаров | Инлайн-поиск в пустой строке таблицы |
| Расхождения по бумаге | Все операции = документы с автором/датой |
| Не понятно кто что может | Единая RBAC-матрица по 7 ролям |

### 1.3 Главный принцип документации

> **Каждое утверждение в документе — это либо правило, либо поле, либо переход, либо таблица, либо сценарий «если X → то Y».**
>
> «Вода» (нарратив без конкретики, истории «что пытались в прошлом», «бесило менеджеров») — **запрещена**.

## 2. Навигация по инструкциям

Все правила и шаблоны для работы агентов вынесены в 5 топических файлов. **Читай их по необходимости:**

| # | Файл | Когда читать | Содержимое |
|---|---|---|---|
| 1 | [`AGENT-ROLES.md`](AGENT-ROLES.md) | Перед началом работы | 7 ролей, атомарность + исключение STUB, pipeline + полный цикл с петлёй QA → coordinator |
| 2 | [`AGENT-METHOD.md`](AGENT-METHOD.md) | Перед началом работы | Шаги прохождения + правила фиксации dыр (проектные vs локальные) + шаблон OQ |
| 3 | [`AGENT-FORMAT.md`](AGENT-FORMAT.md) | При написании любого документа | Принципы П1-П8 + форматирование + анти-паттерны (A1-A11) |
| 4 | [`AGENT-REVIEW.md`](AGENT-REVIEW.md) | Перед сдачей работы | Чек-лист MUST/SHOULD (включая hard limit 400 строк) |
| 5 | [`AGENT-PROMPTS.md`](AGENT-PROMPTS.md) | При запуске нового агента | Промпт-шаблоны для 5 ролей + табличный формат отчёта Координатора |

**Минимальный маршрут для типовой задачи:**
1. Этот файл (5 мин)
2. [`AGENT-ROLES.md`](AGENT-ROLES.md) (5 мин)
3. Выполнение задачи (основная работа)
4. [`AGENT-REVIEW.md`](AGENT-REVIEW.md) перед сдачей (5 мин)

**Полное погружение — в [`AGENT-METHOD.md` §2.1](AGENT-METHOD.md).**
