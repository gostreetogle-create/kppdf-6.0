## 0. Глоссарий терминов

Используются единые названия операций и сущностей. Термины пересекаются с `МОДУЛЬ-СКОЛАД-ПОДРОБНЫЙ.md` — там полные схемы, здесь — как они видны пользователю.

| Термин | Что значит (как видит пользователь) |
|---|---|
| **Склад (Warehouse)** | Физическое место хранения. Справочник: «Основной», «У партнёра», «Транзитный». Кладовщик выбирает из dropdown. |
| **Остаток (StockRecord)** | Текущее количество конкретного товара на конкретном складе. Видно как `availableQty` (которое можно отгрузить) и `reservedQty` (зарезервировано под КП). |
| **Движение (StockMovement)** | Любое изменение остатка. Видно как запись в журнале: «приход +50 / расход −10 / списание −3». Immutable — отменить можно только встречным движением. |
| **Приход от поставщика (SupplierDelivery)** | Документ приёмки товара от поставщика. Номер `СД-XXXX`. Создаётся кладовщиком по факту прибытия машины. |
| **Отгрузка клиенту (Shipment)** | Документ отгрузки. Номер `ОТК-XXXX`. Создаётся кладовщиком вручную или авто-предложением при завершении ЗК. |
| **Акт списания (WriteOffAct)** | Документ списания брака/недостачи/утери. Номер `АС-XXXX`. Требует approve от accountant или director. |
| **Заявка на закупку (PurchaseOrder)** | Документ «купить N штук товара X у поставщика Y». Номер `ЗП-XXXX`. Создаётся закупщиком. |
| **Заявка на закупку (PurchaseRequest)** | «Лёгкая» заявка, которую создаёт система автоматически, когда `availableQty < minStock`. Закупщик решает: оформить на её основе PurchaseOrder или отклонить. |
| **Поставщик (Supplier)** | Организация из справочника с галкой «поставщик». Выбирается из dropdown. |
| **availableQty (доступно)** | = `quantity − reservedQuantity`. Число, которое кладовщик может физически отгрузить без перерасхода. |
| **Каркас-Kit** (как в КП-доке) | Глобальный дизайн-каркас приложения: левая витрина/фильтры, центральная таблица, правая панель действий. Используется на всех 5 страницах Склада единообразно. |
| **Снимок (snapshot)** | Копия данных о товаре на момент операции. Цена, название, фото фиксируются. Изменения в справочнике товаров потом не сломают старые документы. |
| **Транзитный приход (in_transit)** | Статус документа: поставщик отправил товар, машина в пути. Кладовщик видит в «Жду прихода». |
| **Свободная компоновка таблиц** | Как в КП-доке раздел 6. Все таблицы (приходов, отгрузок, списаний) настраиваются через фильтры — без жёсткого «главная таблица + детали». |

> **Префиксы документов в v1:**
> - `СД-XXXX` — SupplierDelivery (Приход) — отдельный счётчик
> - `ОТК-XXXX` — Shipment (Отгрузка) — отдельный счётчик
> - `АС-XXXX` — WriteOffAct (Акт списания) — отдельный счётчик
> - `ЗП-XXXX` — PurchaseOrder (Заявка на закупку) — отдельный счётчик
> Эти префиксы коррелируют с префиксами `КП-XXXX`, `Д-XXXX`, `ЗК-XXXX` других модулей через ту же sealed counter table логику. См. также **СПОР-13 (24.06.2026)** в `СПОРНЫЕ-МОМЕНТЫ.md` — нумерация НЕ синхронизирована между модулями, каждый счётчик независим (намеренно).

## 1. Для чего нужен этот модуль

**Модуль Склад** — это место, где кладовщик принимает товар, отгружает клиентам, списывает брак и планирует закупки. Все эти операции фиксируются как **документы** (не Excel-строки, не голословные пометки). Это защищает от расхождений между «учётными остатками» и «реальностью на полке».

### Для кого он работает

| Кто | Что делает |
|---|---|
| **Кладовщик (`storekeeper`)** | Принимает приходы (`СД`), оформляет отгрузки (`ОТК`), делает списания (`АС`), контролирует остатки в `/warehouse`. Главный пользователь всех 5 страниц. |
| **Менеджер по продажам (`manager`)** | Видит остатки в `/warehouse` (карточка товара показывает «в наличии N шт.»). Создаёт отгрузки от своего имени через RBAC ACL. |
| **Закупщик** (`manager` с расширенными правами или выделенная роль) | Работает в `/purchase`: видит заявки от системы (PurchaseRequest), создаёт PurchaseOrders, контролирует приходы по своим `ЗП`. |
| **Бухгалтер (`accountant`)** | Видит акты списания (для отчётов), approve мелких списаний (до 5 000 ₽) в `/write-off`. Видит приходы в `/supplier-deliveries` для сверки с нашими счетами. |
| **Директор (`director`)** | Видит всё (включая крупные WriteOffAct-ы на approve). Может отменить любую операцию. |
| **Администратор (`admin`)** | Настраивает справочник складов в `/warehouse` → вкладка «Склады». |

### Какую главную боль решает модуль

В прошлой версии (5.0) кладовщик вёл учёт в Excel:
- Менеджер вбил «отгрузили 10 шт.» → кладовщик не узнал, **расхождение** на складе.
- Пришло 100 шт., продали 50, в Excel осталось 50, а на полке 47: **3 потеряны** без следа.
- Брак списывали «по-тихому», аналитика «сколько списано за год» — **недоступна**.

В v6 все операции **документальные**:
- Каждое движение — это документ (приход/отгрузка/списание) с автором и временем.
- Остаток физический = `availableQty` в карточке товара, виден всем.
- Любое списание — это акт с approve-ом, не «удалено из Excel».

### Концепция «Единое окно Склада»

Как и в модуле КП («Каркас-Kit» — 3-зонный макет с витриной слева, татом по центру, действиями справа), на всех 5 страницах Склада используется **единый макет**:

```
┌──────────────────────────────────────────────────────────────────┐
│ [Левая панель — фильтры и витрина]    │ [ЦЕНТР — таблица]    │ [Правая панель — действия]
```

Но в отличие от КП-дока (где в центре лист А4), здесь в центре — **живая таблица с фильтрами и сортировкой на клиенте**. Это типичный «админский UI» (как в 1С, МойСклад).

**Адрес страниц** (URL-маршруты):
- `/warehouse` — главная страница Склада (дашборд остатков + журнал движений)
- `/supplier-deliveries` — список приходов `СД-XXXX`
- `/shipments` — список отгрузок `ОТК-XXXX`
- `/purchase` — заявки на закупки `ЗП-XXXX` + PurchaseRequest очередь
- `/write-off` — акты списания `АС-XXXX` (включая approve-очередь для director/accountant)

Между страницами переход через **главное меню** (слева в навигации): «Склад» с подпунктами.

## 2. Жизненный цикл операций Склада

В модуле Склад **4 базовых операции** (по числу базовых документов из `МОДУЛЬ-СКОЛАД-ПОДРОБНЫЙ.md`):

```
═════════════════════════════════════════════════════════════════════
                          ПОЛНАЯ ЦЕПОЧКА ОПЕРАЦИЙ
═════════════════════════════════════════════════════════════════════

   ╔════════════════════════╗      ╔════════════════════════╗
   ║ PURCHASE ORDER (ЗП)    ║ ───▶ ║ SUPPLIER DELIVERY (СД)║
   ║ «закупить N шт.»       ║      ║ «принять приход»      ║
   ╚════════════════════════╝      ╚══════════╤═════════════╝
   draft → sent → confirmed                  │
           → in_transit → received            │ при завершении (completed)
                       → partially_received   │ создаёт
                       → closed               │ StockMovement: type=in
                                              ▼
                                       ┌──────────────┐
                                       │ StockRecord  │  ←─── изменение
                                       │ quantity +N  │      остатка
                                       └──────┬───────┘
                                              │
                                              ▼
                  ╔═══════════════════════╗  ╔════════════════════════╗
                  ║ PRODUCTION ORDER (ЗК) ║  ║    WRITE-OFF (АС)      ║
                  ║ «произвести/собрать»  ║  ║ «списать брак/утерю»  ║
                  ╚═════════════════╤═════╝  ╚════════════╬═══════════╝
                                    │                    │
                                    │ when completed     │ создаёт
                                    ▼                    ▼ StockMovement:
                            ╔════════════════╗          type=out
                            ║ SHIPMENT (ОТК) ║          reason=write_off
                            ║ «отгрузить»    ║
                            ╚════════╤═══════╝
                                     │ при shipped
                                     ▼
                              StockRecord: quantity −N
                              OrderClosing.progress +=N
```

### Сводная таблица статусов 4 документов

| Документ | Статусы (жизненный цикл) | Триггер переходов |
|---|---|---|
| **PurchaseOrder (ЗП)** | `draft → sent → confirmed → in_transit → received (or partially_received) → closed` (или `cancelled`) | Закупщик → авто при SupplierDelivery |
| **SupplierDelivery (СД)** | `planned → in_transit → received → partially_received → completed` (или `cancelled`) | Кладовщик → авто StockMovement при completed |
| **Shipment (ОТК)** | `planned → packed → shipped → delivered` (или `partial`, `cancelled`) | Кладовщик → авто StockMovement при shipped |
| **WriteOffAct (АС)** | `draft → pending_approval → approved → completed` (или `cancelled`) | Создатель → approver → авто StockMovement при completed |

Каждый переход подробно расписан в `МОДУЛЬ-СКОЛАД-ПОДРОБНЫЙ.md` разделы 5.3, 6.3, 7.3, 8.3. Здесь — вкратце.

## 3. КЛЮЧЕВОЙ СЦЕНАРИЙ: «Принять приход за 3 минуты»

Это самая частая операция кладовщика. Как должно работать в идеале (v6):

**Ситуация.** Закупщик позавчера оформил `PurchaseOrder ЗП-0042` поставщику ООО «Металлопром» на 100 шт. арматуры A500C, ожидаемая дата прихода — сегодня. Сегодня утром приехала машина. Кладовщик Сидоров начинает приёмку.

**Что должно происходить:**

1. Сидоров открывает `/supplier-deliveries`, видит список «Ожидаемые приходы» (вверху страницы, отдельная вкладка/фильтр). Там `ЗП-0042` подсвечен жёлтым («ожидается сегодня»).
2. Нажимает на `ЗП-0042`. Открывается экран «Принять приход».
3. Система **автоматически предзаполняет форму** на основе ЗП: поставщик, склад, позиции с количеством 100 (ожидаемое).
4. Сидоров вводит **фактическое количество** (приехало 98 — 2 в пути потерялись). Ну и ещё 3 шт. с дефектом (видно сразу).
5. В таблице позиций видит 3 строки на каждую позицию ЗП: «Ожидалось 100», «Принято 98», «Брак 3» (последняя стала отдельной записью для будущего `АС`).
6. Нажимает **«Принять приход»** в правой панели.
7. Появляется мини-форма: «Создать (1) SupplierDelivery СД-0017 и (2) WriteOffAct АС-0003 на 3 шт.? [Да, оба] / [Только СД] / [Отмена]». По умолчанию — «Да, оба».
8. Сидоров нажимает «Да, оба». Система **мгновенно**:
   - Создаёт `СД-0017` со статусом `completed`.
   - Создаёт `АС-0003` со статусом `pending_approval` (ждёт approve от accountant).
   - Автоматически создаёт `StockMovement: Приход` +98 по товару.
   - Обновляет `ЗП-0042` → `partially_received`.
   - Фиксирует остаток на складе (теперь +98 −0 = +98).
9. Сидорову остаётся только проверить визуально: товар на месте, место на складе освободилось.

**По времени — 3 минуты**, не больше.

### Что НЕ происходит при приёмке

- **Не блокируется работа других** — пока Сидоров принимает приход, Петров в `/shipments` тоже работает.
- **Не требуется ручной ввод каждой позиции** — все позиции предзаполняются из ЗП.
- **Не требуется approve для СД** — только для списания (АС).

## 4. Экраны модуля (5 страниц)

Все 5 страниц следуют единому макету **Каркас-Kit** (3-зонный):

```
┌──────────────────────────────────────────────────────────────────┐
│ [Левая панель — фильтры и витрина]    │ [ЦЕНТР — таблица]    │ [Правая панель — действия]
│                                              │
│ ─ Фильтры ─                               │
│ [Поиск по товару/номеру]                  │
│ [Склад ▼] [Статус ▼] [Период ▼]         │
│ [☐ Архивные]                              │
│                                              │
│ ─ Витрина (опционально) ─                 │
│ (карточки товаров или Suppliers)          │
│                                              │
│ [Клик на товар → форма в правой панели]  │
```

### 4.1. `/warehouse` — Главная страница Склада (дашборд остатков)

Это **главный экран модуля** — обычно выставлен по умолчанию при заходе в «Склад». Кладовщик открывает здесь смену и держит открытым весь день.

#### Что в центре (главная таблица)

**Вкладка «Остатки»** (открыта по умолчанию):

| Колонка | Что в ней | Редактируется? |
|---|---|---|
| SKU | Артикул товара | Нет |
| Название | Полное название | Нет |
| Категория | Категория из справочника товаров | Нет |
| Единица | шт / комплект / м² / м.п / кг / час | Нет |
| Склад | Из dropdown (по умолчанию «Все склады») | Фильтр |
| **Физически** | `quantity` (всего на складе) | Нет |
| **Зарезервировано** | `reservedQuantity` (под КП/ЗК) | Нет |
| **Доступно** | `= quantity − reservedQuantity`. **Главная цифра для менеджера по продажам!** | Авто-расчёт |
| minStock | Минимум, при падении ниже — красная подсветка + иконка ⚠️ | Нет |
| Последнее движение | Дата+время (отсортировано DESC по умолчанию) | Нет |

**Клик на строку → открывает карточку товара в правой панели** с историей движений по этому товару (вкладки: «Приходы», «Отгрузки», «Списания»).

Внизу таблицы — блок **«Итого по странице»**: `∑ quantity`, `∑ reservedQuantity`, `∑ availableQty` + кнопка «Экспорт в Excel» (для инвентаризации).

**Вкладка «Движения»** (журнал StockMovement):

| Колонка | Что в ней |
|---|---|
| Дата/время | `createdAt` движение |
| Тип | `in` / `out` / `transfer` / `write_off` (с цветной иконкой: + зелёный, − красный) |
| Причина | `purchase` / `production` / `sale` / `write_off` / `manual` |
| Товар | SKU + название + ед. |
| Количество | > 0 (направление — в типе) |
| Документ-источник | Ссылка на `СД-XXXX` / `ОТК-XXXX` / `АС-XXXX` / `ЗК-XXXX` |
| Автор | ФИО пользователя |
| Склад | Из dropdown |

Без возможности правки! Движения **immutable**. Если ошибка — создаётся встречное (return / re-add).

**Вкладка «Склады»** (только для admin):

> Содержимое: список всех `Warehouse` в системе с кнопками «Добавить», «Архивировать», «Редактировать». Inline-форма для создания нового склада.

#### Что в левой панели

- **Поиск по товару/артикулу**: текстовое поле, мгновенная фильтрация.
- **Фильтр по складу**: dropdown «Все склады» / «Основной» / «У партнёра» / «Транзитный».
- **Фильтр по категории товара**: dropdown.
- **Чекбокс «Показать только с availableQty < minStock»** — критичные товары.
- **Чекбокс «Только с движениями за последние N дней»** — N выбирается dropdown (1 / 7 / 30 / 90).
- **Чекбокс «Архивные товары»**: по умолчанию ВЫКЛ.

#### Что в правой панели

- **Карточка товара** (открывается при клике на строку):
  - Фото + название + категория + ед.
  - Остатки по всем складам (если товар есть на 3 складах — таблица из 3 строк).
  - Вкладки: «История движений», «Резервы», «Подробности».
  - Кнопки (по RBAC): «Переместить между складами», «Создать списание», «Открыть карточку товара в справочнике».
- **Карточка склада** (если админ):
  - Редактор полей `Warehouse`.
- **Ничего** (если ничего не выбрано): краткая статистика — «Товаров в наличии: N / Складов: M / Ожидают прихода: 3 / Ожидают отгрузки: 7».

#### Кнопки действий (по RBAC)

| Кнопка | Кто видит | Действие |
|---|---|---|
| **Создать приход (`СД`)** | storekeeper, admin, manager (только свой поставщик), director | Открывает мини-форму «Новая SupplierDelivery» в правой панели |
| **Создать отгрузку (`ОТК`)** | storekeeper, admin, manager (только своё КП/Договор), production (только свой ЗК), director | Открывает мини-форму «Новая Shipment» |
| **Создать списание (`АС`)** | storekeeper, admin, director (без approve), accountant (только inventory) | Открывает мини-форму «Новый WriteOffAct» в статусе `draft` |
| **Запланировать закупку (`ЗП`)** | manager (если назначен), admin, storekeeper (если назначен), director | Открывает экран `/purchase` |

### 4.2. `/supplier-deliveries` — Список приходов

Типичная «однотабличная» страница: список всех `СД` с быстрыми переходами в детальный режим.

#### Структура

**Левая панель** — фильтры:

- **Вкладка «Ожидаемые приходы»** (по умолчанию): только статусы `planned` + `in_transit`. Сверху — крупная кнопка «Принять приход» (открывает форму на основе самого «горящего» ЗП).

## 5. Бизнес-правила

### 5.1. Правила для остатков и движений

1. **`availableQty ≥ 0` всегда.** Система **блокирует** создание любого `StockMovement: out`, если `quantity < quantityAfterMovement`. Нельзя уйти в минус физически.
2. **`reservedQuantity ≥ 0`** — резерв не может превышать `quantity`. Создание резерва блокируется, если не хватает физического остатка.
3. **`StockMovement immutable`.** После создания нельзя редактировать. Для исправления — только встречное движение (return / re-add / transfer).
4. **`availableQty = quantity − reservedQuantity`** — единственная цифра для отображения «сколько можно отгрузить». Менеджер по продажам видит именно её.

### 5.2. Правила для приходов (`СД`)

1. **Принимать приход может только `storekeeper`** (или admin/director — emergency).
2. **«Принять частично» обязательно = «Оформить `АС`»** — если `quantityReceived < quantityExpected`, система **требует** либо (а) оформить акт списания на разницу, либо (б) выбрать причину «потеря в пути без вины поставщика». Без одного из двух — приход остаётся в статусе `partially_received` и **считается незавершённым**.
3. **Оплата поставщику (Payment) — отдельный процесс** в Модуле Финансы. Здесь только факт приёмки. `paymentStatus` обновляется вручную бухгалтером (в v1 нет автоматической связи).
4. **Приход без ЗП** — допустим (машина приехала без предупреждения). `supplierPurchaseOrderId = NULL`. Закупщик должен оформить ЗП постфактум, чтобы привязать приход.

### 5.3. Правила для отгрузок (`ОТК`)

1. **Авто-предложение при завершении ЗК**, НЕ авто-создание — чтобы избежать случайных двойных отгрузок. Подробнее: `МОДУЛЬ-СКОЛАД-ПОДРОБНЫЙ.md` §6.4.
2. **`shipped → StockMovement: out` автоматически** — при нажатии «Отгрузить» в одной транзакции: создаётся ShipmentItem → StockMovement → обновляется StockRecord.quantity. Если транзакция обрывается — откат, **никаких half-done состояний**.
3. **`delivered → OrderClosing.progress += quantityActual`** — в Модуле Финансы (не здесь).
4. **Мульти-склад отгрузка — два `ОТК`** (если товар на 2 разных складах) — проще, чем усложнять items[].

### 5.4. Правила для списаний (`АС`)

1. **Авто-маршрутизация approve** по сумме:
   - `< 5 000 ₽` → **accountant** (если недоступен — director).
   - `5 000 – 50 000 ₽` → **director**.
   - `≥ 50 000 ₽` → **director** + бухгалтерская фиксация в журнале.
2. **`inventory_shortage`** — отдельная категория: требует **director + комиссия** (минимум 3 человека в `commissionMembers`).
3. **`completed → StockMovement: out, reason=write_off`** — в одной транзакции с обновлением StockRecord.
4. **Отмена списания возможна только в `draft`**. После `pending_approval` — только возврат в `draft` через «Отклонить» approver-ом.

### 5.5. Правила для закупок (`ЗП`)

1. **Авто-создание из `PurchaseRequest`** (когда закупщик нажал «Оформить как ЗП») — `purchaseOrderId` в PR сохраняется для трассировки.
2. **Цена в ЗП — это `unitCost` от поставщика** (закупочная цена). Используется в `StockMovement.unitCost` для расчёта себестоимости в Модуле Финансы.
3. **`ЗП → received` (авто)** при получении `СД` со статусом `completed`. `partially_received` — если хотя бы один `СД` завершился с неполным приходом.

### 5.6. Правила для UI (все 5 страниц)

1. **Автосохранение фильтров** в localStorage — менеджер возвращается на страницу → видит те же фильтры (как в КП-док §8).
2. **Inline-поиск в каждой таблице** — мгновенная фильтрация на клиенте (≤ 100 строк). Если больше — серверный запрос с debounce 300ms.
3. **Optimistic Locking на approve `АС`** — если два approver-а открыли один акт одновременно, второй видит предупреждение: «Акт уже утверждён первым пользователем. Обновить?». По аналогии с автосохранением КП.
4. **Печатные формы** (`Print PDF`):
   - `СД` → накладная ТОРГ-1 (базовая, некрасивая).
   - `ОТК` → накладная ТОРГ-12 (базовая).
   - `АС` → акт на списание (простая форма).
   - `ЗП` → заявка поставщику (простая форма).
   - Все 4 PDF содержат номер, дату, контрагента, сумму, позиции, печать организации.

## 6. Связи с другими модулями

### 6.1. С Модулем КП (`МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md`)

- **Поле `Product.minStock`** + `StockRecord.quantity` отображается в витрине товаров в `/proposals/new`. Карточка товара показывает «есть на складе N шт.» (как уже описано в КП-доке §4).
- **КП «отправлено» / «принято»** → создаётся `Reservation` под товары. Резерв уменьшает `availableQty`.
- **КП «отменено»** → резерв снимается, `availableQty` восстанавливается.
- **КП «Склад-напрямую» (сценарий R1)** — менеджер видит «товар на складе, без ЗК». В этом случае отгрузка `ОТК-XXXX` создаётся **напрямую по КП** (`proposalId` в Shipment, без ЗК).

### 6.2. С Модулем Договор (`МОДУЛЬ-ДОГОВОР.md`)

- **`Contract.parentProposalId`** = родительский КП-источник → отгрузки по Договору имеют `contractId` в `Shipment`.
- **`Contract` «Подписан клиентом»** → автоматически НЕ создаёт отгрузку (требуется ручное создание кладовщиком через UI).
- **Тендерные закупки** — `PurchaseOrder.parentContractId` ссылается на Договор с поставщиком.

### 6.3. С Модулем Производство (`МОДУЛЬ-ПРОИЗВОДСТВО.md`)

- **`ProductionOrder → Завершён`** — авто-предложение создать `Shipment` (см. §4.3).
- **`ProductionTask.type === ITEM`** — только такие задачи создают StockMovement при завершении ЗК (R2 закрыт через СПОР-12).
- **`ProductionTask.status === done` + ЗК содержит только задачи типа INSTALLATION/SERVICE/WORK** — **никаких StockMovement**, кладовщик получает **информационное сообщение** в `/warehouse` (закрывает R1 + GAP-009).

### 6.4. С Модулем Финансы (`МОДУЛЬ-ФИНАНСЫ.md`)

- **`Shipment.delivered`** → `OrderClosing.progress += quantityActual` (в v1 — ручное обновление бухгалтером через UI ОрдерКлозинга, v2 — автоматический триггер).
- **`WriteOffAct.completed`** → уменьшение `costOfGoodsSold` в `OrderClosing` (для подсчёта маржи).
- **`SupplierDelivery.paymentStatus`** → обновляется вручную бухгалтером (оплата поставщику происходит через `Payment` в Финансах).
- **`OrderClosing.trigger`** для v1 — «отгрузка delivered + оплата поступила» — оба условия должны быть выполнены.

### 6.5. Со Справочником Организаций и Товаров

- **`Organization.isSupplier = true`** обязательно для `PurchaseOrder.supplierId`. Если в dropdown-е нет поставщиков — показать жёлтую плашку «Добавьте хотя бы одного поставщика в справочник».
- **`Organization.isActive = false`** — поставщик/клиент НЕ показывается в dropdown-ах **НО** существующие документы продолжают быть видимы.
- **`Product.isActive = false`** — товар НЕ показывается в витрине КП И в фильтрах отгрузки **НО** приходы/отгрузки с этим товаром **остаются неизменными** (с snapshot-ом названия/SKU).

### 7.1. Пояснения

- **`manager` ограничен** — может создавать `СД` только по своему поставщику (где `createdById = manager`), `ОТК` — только по своему КП/Договору/ЗК (где он `createdById` или ответственный). Это чтобы менеджер не отгрузил чужой товар.
- **`production` создаёт `ОТК` только по своему `ЗК`** — где он `responsibleUserId` или среди `assigneesProductionOrder`. Потому что начальник производства знает, когда его цех закончил и можно отгружать.
- **`storekeeper`** — главный пользователь всех 5 страниц. Без него склад не работает.
- **`accountant`** — approve мелких списаний + участие в инвентаризации. **Не имеет права** на рутинные операции (приход/отгрузка) — это зона кладовщика.
- **`director`** — универсальный fallback для emergency. На approve крупных `АС` — обязательно.

### 7.2. Настройка прав (админка)

Администратор настраивает RBAC через `/admin/rbac` (отдельный экран). Для модуля Склад:

- Какие роли имеют доступ к каким страницам (по умолчанию все, кроме `viewer` только чтение).
- Какие роли могут approve `АС` на каких суммах.
- Какие роли могут создавать `ЗП` (отдельно от обычных `manager`, чтобы дать только назначенным закупщикам).

## 8. Что НЕ делаем в первой версии (границы MVP)

### UI не входит в v1

- **Drag-and-drop** для строк в таблицах — порядок меняется стрелочками «вверх/вниз» в контекстном меню.
- **Мобильный интерфейс** для кладовщика (Android-приложение со сканером штрих-кодов) — v2.
- **Тёмная тема** — отдельный вопрос.
- **Многоязычность** (русский + английский) — отдельный вопрос.
- **Real-time обновления** через WebSocket — менеджер обновляет страницу вручную (F5) или ждёт 30 секунд. **Без push-уведомлений** в v1.

### Функции не входят в v1

- **Штрих-коды** для сканирования товара — v2.
- **Статистика оборачиваемости** (turnover days) — v2.
- **ABC/XYZ анализ** — v2.
- **Прогнозирование спроса** (ML) — **никогда** (по решению `СТЕК-ПРЕДПИСАНИЕ.md` §6.10).
- **Интеграция с 1С / МойСклад** — v2.
- **Multi-currency** — v2 (закреплено СПОР-14, в v1 только RUB).
- **CDN для фото товаров** — v2 (закреплено СТЕК-ПРЕДПИСАНИЕ §6.10).
- **Аудит-лог изменений** (кто что менял задним числом) — v2. В v1 — только `createdById`/`approvedById` на каждом документе, без истории.
- **Возврат от клиента (R2 / S3)** — отложено в v2. В v1 только `WriteOffAct` для внутреннего брака.

### Шаблоны документов — не в v1

- Все 4 документа (СД, ОТК, АС, ЗП) в v1 — **только табличный view**. Красивые шаблоны с логотипом и печатью — v2.
- Это осознанное упрощение: кладовщику важнее скорость работы, чем красота печатной формы.

## 9. Открытые вопросы (нужны продуктовые решения)

### Q1. (R1 + GAP-009 — закрыто в `МОДУЛЬ-СКОЛАД-ПОДРОБНЫЙ.md`) ЗК без товаров
- ✅ **Решение v1:** при завершении ЗК только с услугами кладовщик получает **информационное сообщение** в `/warehouse` (без Shipment, без StockMovement).

### Q2. (S3 / R2 — отложено) Возврат клиентом
- В v2 — новый статус `Shipment.status = returned` + `StockMovement.reason = sale_return` + Refund в Финансах.
- **В v1:** нет UI для возврата клиентом. Если клиент вернул товар — оформить через `WriteOffAct.reason = inventory_shortage` (без связи с клиентом).

### Q3. (Новое для UI) Виджет «Жду прихода» — где разместить?
- **Вариант A:** отдельная вкладка в верхней части `/supplier-deliveries` (как сейчас в §4.2).
- **Вариант B:** крупный блок-сводка на `/warehouse` (на дашборде остатков).
- **Вариант C:** в правой панели `/warehouse` (боковой).
- **Рекомендация:** A + ссылка «Показать на дашборде остатков» в B.

### Q4. (Новое) Кнопка «Принять все ожидаемые приходы» (массовая операция)
- Если в 9 утра приехало 5 машин — удобно ли одним кликом принять все?
- **Вариант A:** Есть кнопка «Принять все ожидаемые» в `/supplier-deliveries` (5 одновременно).
- **Вариант B:** Только по одному.
- **Рекомендация v1:** B (безопаснее — кладовщик должен проверить каждую машину; массовое принятие рискует пропустить брак).

### Q5. (Новое) Звуковое уведомление при новом «Жду прихода»
- Закупщик утром создал ЗП — кладовщик должен узнать. Push-уведомление через колонки?
- **Решение v1:** нет (WebSocket не в v1). Только визуально — счётчик «Ждут прихода: N» в правом верхнем углу `/warehouse` обновляется при следующем F5.

### Q6. (Новое) Drag-and-drop между двумя страницами (Shipment → WriteOffAct)
- Может ли кладовщик «перетащить» строку из `/shipments` в `/write-off` (если обнаружил брак при отгрузке)?
- **Решение v1:** нет. Это два разных процесса — `АС` оформляется через явную форму, не drag-and-drop.

### Q7. (Новое) Bulk-списание через barcode-сканер
- В v2 — кладовщик сканирует все бракованные товары, система автоматически собирает `АС`.
- **Решение v1:** ручной ввод строк (через форму `+`).

### Q8. (Новое) Подтверждение прихода через Telegram-бот
- Кладовщик получает push в Telegram «Машина с ЗП-0042 приехала. Принять? [Да] [Нет]».
- **Решение v1:** нет (Telegram-бот — отдельный проект).
