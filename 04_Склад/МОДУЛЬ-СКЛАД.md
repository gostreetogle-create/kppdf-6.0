# Модуль «Склад»

> **Назначение.** Входная точка для понимания модуля «Склад» в v6. Это **обзорный каркас для менеджера/ИИ-агента**: что этот модуль делает, как он работает, кто им пользуется, как связан с другими модулями.
>
> **Структура модуля.** Модуль разделён на **3 файла** в этой папке:
> - **`МОДУЛЬ-СКЛАД.md`** (этот документ) — обзорный каркас для быстрого погружения.
> - **`МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md`** — полные схемы 4 сущностей (`SupplierDelivery` / `Shipment` / `WriteOffAct` / `PurchaseOrder`) + детальный RBAC.
> - **`МОДУЛЬ-СКЛАД-UI.md`** — UI-каркасы 5 страниц на «Каркас-Kit».
>
> **Для разработчика кода:** сначала прочитайте этот файл (общая картина), затем переходите к Подробному (схемы БД) и UI (как выглядит).
>
> **Дата:** 24.06.2026.

## 0. Глоссарий терминов модуля

| Термин | Что значит (как для менеджера/ИИ-агента) |
|---|---|
| **Склад (Warehouse)** | Физическое место хранения товара. Справочник (например, «Основной», «У партнёра», «Транзитный»). |
| **Остаток (StockRecord)** | Текущее количество конкретного товара на конкретном складе = `availableQty = quantity − reservedQuantity`. |
| **Зарезервировано (reservedQuantity)** | Количество товара, «замороженное» под активные КП/ЗК. Не отгружается, но физически на месте. |
| **Движение (StockMovement)** | Неизменяемая запись изменения остатка: приход/расход/перемещение/списание. Историю нельзя править задним числом. |
| **Приход от поставщика (SupplierDelivery)** | Документ приёмки товара от поставщика по `PurchaseOrder`. Префикс `СД-XXXX`. Шесть статусов (`planned → in_transit → received → partially_received → completed → cancelled`). |
| **Отгрузка клиенту (Shipment)** | Документ отгрузки готового товара клиенту. Префикс `ОТК-XXXX`. Шесть статусов (`planned → packed → shipped → delivered → partial → cancelled`). |
| **Акт списания (WriteOffAct)** | Документ списания брака / недостачи / утери. Префикс `АС-XXXX`. **Требует approve** (двухступенчатый процесс). |
| **Заявка на закупку (PurchaseOrder)** | Внутренний документ «закупить N товаров у поставщика». Префикс `ЗП-XXXX`. Восемь статусов (черновик → отправлен → подтверждён → в пути → принят → закрыт). |
| **Заявка от системы (PurchaseRequest)** | Автоматическая заявка при `availableQty < minStock`. Закупщик решает: оформить ЗП или отложить. |
| **Резерв (Reservation)** | Запись «заморозки» товара под КП. Снимается при отклонении/архивации КП. |
| **Каркас-Kit** (как в КП-доке) | Глобальный 3-зонный дизайн-каркас: левая витрина / центральная таблица / правая панель действий. Используется на всех 5 страницах Склада. |
| **Снимок (snapshot)** | Копия данных о товаре (`productSku, productName, unit`) на момент операции — защита от ретро-правок в справочниках. |
| **Источник (`source*Id`)** | FK обратной связи в `StockMovement`: какое событие создало это движение (ЗК / КП / СД / ОТК / АС). |
| **approve (для списания)** | Двухступенчатый процесс: создатель формулирует акт → approver (accountant до 5000₽, director выше) утверждает → система проводит через StockMovement. |

> Полный глоссарий (30+ терминов проекта) — в `99_Справочники/GLOSSARY-MASTER.md`.

## 1. Для чего нужен этот модуль

**Склад** в v6 — это **операционная точка контроля движения товара**: что пришло, что ушло, что списали, что нужно закупить.

### Боль, которую решает

В прошлой версии (kppdf v5) учёт вёлся в Excel. Менеджер вбивал «отгрузили 10 шт.», а кладовщик не знал — потому что Excel-файл лежит на одном компьютере. Возникали:
- **Расхождения** между «учётными остатками» и «реальностью на полке».
- **Потери** (3 шт. списали «по-тихому» — аналитика «сколько списано за год» недоступна).
- **Пересортица** при приёмке (приехало не то, что заказывали, кладовщик разбирается сам).

В v6 **все операции фиксируются документами** (приход/отгрузка/списание) с автором, датой, основанием. Это **невозможно «забыть» или «потерять»**.

### Для кого работает (5 ролей)

| Роль | Что делает в модуле Склад |
|---|---|
| **Кладовщик (`storekeeper`)** | **Главный пользователь.** Принимает приходы, оформляет отгрузки, делает списания, контролирует остатки в `/warehouse`. Имеет права на все операции. |
| **Менеджер по продажам (`manager`)** | Видит остатки при подготовке КП («вот на этом складе 12 шт., на другом 8, итого 20»). Создаёт отгрузки от своего имени в пределах RBAC. |
| **Закупщик** (`manager` с расширенными правами или выделенная роль) | Работает в `/purchase`: видит заявки от системы, создаёт заявки на закупку, контролирует приходы по своим `ЗП`. |
| **Бухгалтер (`accountant`)** | Видит акты списания (для финансовой отчётности). **Approve мелких списаний** до 5 000 ₽ в `/write-off`. Участвует в инвентаризации как член комиссии. |
| **Директор (`director`)** | Видит все движения. **Approve крупных списаний** (5 000+ ₽) и инвентаризационных актов. Emergency fallback для операций. |

## 2. 4 базовые операции = 4 документа = 4 префикса

```
═══════════════════════════════════════════════════════════════════════
                         ОПЕРАЦИИ СКЛАДА v6
═══════════════════════════════════════════════════════════════════════

   ╔════════════════════════╗      ╔════════════════════════╗
   ║ PURCHASE ORDER (ЗП)    ║ ──▶  ║ SUPPLIER DELIVERY (СД) ║
   ║ «закупить N шт.        ║      ║ «принять приход»      ║
   ║ у поставщика»          ║      ║                       ║
   ╚════════════════════════╝      ╚════════╤═══════════════╝
   draft → sent → confirmed →             │
   in_transit → received →                 │ при completed:
   partially_received → closed             │ StockMovement:in
                                            ▼
                                      ┌──────────────┐
                                      │ StockRecord  │ ← изменение остатка
                                      │ quantity ≥0  │
                                      └──────┬───────┘
                                             │
                                             ▼
              ╔════════════════════════╗  ╔════════════════════╗
              ║ PRODUCTION ORDER (ЗК) ║  ║   WRITEOFF (АС)    ║
              ║ «произвести/собрать»  ║  ║ «списать брак»     ║
              ╚══════════════╤═════════╝  ╚════════╤═══════════╝
                             │                    │
                             │ when completed     │ создаёт
                             │ (только ITEM)       │ StockMovement:out
                             │                    │ reason=write_off
                             ▼                    ▼
                      ┌──────────────────┐
                      │  SHIPMENT (ОТК)  │
                      │  «отгрузить      │
                      │   клиенту»      │
                      └────────┬─────────┘
                               │ при shipped
                               ▼
                        StockRecord: −qty
                        OrderClosing.progress +=qty
                        (Финансы → Маржа)
```

**Ключевые соответствия:**
- **`СД`** = приход → StockMovement: `IN`, reason=`purchase`
- **`ЗК → Завершён`** (только для `Product.kind='ITEM'`) → StockMovement: `IN`, reason=`production`
- **`ОТК → shipped`** → StockMovement: `OUT`, reason=`sale`
- **`АС → completed`** → StockMovement: `OUT`, reason=`write_off`

## 3. Ключевые правила (одной строкой)

> **Каждое правило подробно расписано в `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` соответствующем разделе.**

1. **Остаток не уходит в минус** — `quantity >= 0` всегда (БД-чек + UI-warning).
2. **Резерв не превышает остаток** — `reservedQuantity <= quantity`.
3. **Все движения immutable** — `StockMovement` не редактируется задним числом. Только создаётся + можно создать встречное корректирующее движение.
4. **Snapshot в позициях** — `productSku/productName/productUnit` копируются из `Product` в момент создания `*Item`. Изменения в справочнике `Product` потом не сломают старые документы.
5. **Списание требует approve** — двухступенчатый процесс: `DRAFT → PENDING_APPROVAL → APPROVED → COMPLETED`. Авто-маршрутизация по сумме.
6. **Услуги и работы не приходят в Склад** — авто-приход от ЗК создаётся **только для `Product.kind='ITEM'`**. Задачи `SERVICE`/`WORK`/`INSTALLATION` идут в Склад как «информационное сообщение» (без реального движения).
7. **`СД-XXXX`, `ОТК-XXXX`, `АС-XXXX`, `ЗП-XXXX`** имеют **отдельные sealed counter tables** (см. `99_Справочники/SCHEMA-CONSOLIDATED.md` §4) — для защиты от race conditions при параллельном создании.
8. **Currency = RUB жёстко в v1** (multi-currency v2, закреплено СПОР-14).

**Условные обозначения:**
- ✅ = разрешено полностью
- ❌ = запрещено
- ⚠️ = разрешено с ограничением (своё / до порога / назначение)

**Где граничные условия подробнее:**
- `manager ⚠️ свой` = только по своим КП/Договорам/ЗК (где `createdById = manager`). Подробнее: `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §9.1.
- `accountant ⚠️ inv` = только `inventory_shortage`/`inventory_surplus` (не брак, не потеря — только инвентаризация). Подробнее: `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §7.4.
- `accountant <5000₽` = approve списаний до 5 000 ₽; свыше — director. Подробнее: `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §7.4.

## 7. Связи с другими модулями

### 7.1 Входящие потоки (модуль Склад получает данные от других модулей)

```
КП «оплачено» → авто-создание ЗК
   ↓
ЗК «завершён» (только для Product.kind='ITEM')
   ├─► авто-создание StockMovement:IN, reason='production'
   └─► авто-предложение кладовщику создать Shipment (ОТК)
         ↓
      ОТК «shipped»
         ├─► авто-создание StockMovement:OUT, reason='sale'
         └─► Shipment «delivered» → OrderClosing.progress ++ (Финансы)
```

### 7.2 Исходящие потоки (Склад отдаёт данные другим модулям)

| Этот модуль → | Связь | Когда |
|---|---|---|
| `Shipment` → `OrderClosing` | Себестоимость + прогресс | При `delivered` Shipment → FINANCE |
| `WriteOffAct` → `OrderClosing` | Списание уменьшает `costOfGoodsSold` | При `completed` WriteOffAct → FINANCE |
| `StockRecord.availableQty` → `Product.costPrice + minStock` | UI КП витрина | Каждый load витрины |
| `SupplierDelivery.paymentStatus` → `Payment` | Связь «прибыл → оплачен» | Вручную бухгалтером (v1) |

### 7.3 FK-связи (бизнес-инварианты)

| Связь | FK | ON DELETE | Что блокирует |
|---|---|---|---|
| `StockRecord.Warehouse` | `warehouseId` | **RESTRICT** | Нельзя удалить Склад с ненулевыми остатками |
| `StockRecord.Product` | `productId` | **RESTRICT** | Нельзя архивировать Товар с ненулевым остатком |
| `SupplierDelivery.Supplier` | `supplierId` | **RESTRICT** | Нельзя удалить Поставщика с активными `СД` |
| `Shipment.Customer` | `customerId` | **RESTRICT** | Нельзя удалить Клиента с активными `ОТК` |
| `WriteOffAct.ApprovedBy` | `approvedById` | **SET NULL** | Уволенный approver — запись остаётся (видимость NULL) |

Полная таблица FK — в `МОДУЛЬ-СКЛАД-ПОДРОБНЫЙ.md` §3 «Сводная таблица FK и ON DELETE».

## 10. Концепции (для быстрого погружения)

### 10.1 Принцип «immutable movements»

`StockMovement` — **никогда не редактируется задним числом**. Если движение ошибочно — создаётся **встречное движение** (return / re-add / transfer). Это защищает аудит-трейл.

### 10.2 Snapshot-поля

Все `*Item` таблицы (`SupplierDeliveryItem`, `ShipmentItem`, `WriteOffItem`, `PurchaseOrderItem`) хранят **snapshot полей `productSku, productName, unit`** на момент создания. Изменения в `Product` потом не сломают старые документы.

### 10.3 Soft-delete

Все 4 новые сущности имеют `isActive: Boolean @default(true)`. «Удаление» = `isActive = false`. Физическое удаление в v1 **запрещено** по конвенции всего проекта v6.

### 10.4 Отдельные счётчики

Префиксы `СД-XXXX`, `ОТК-XXXX`, `АС-XXXX`, `ЗП-XXXX` имеют **отдельные sealed counter tables** (см. `99_Справочники/SCHEMA-CONSOLIDATED.md` §4.2) — для защиты от race conditions при параллельном создании.

`ЗП-0042` может существовать рядом с `КП-0042`, `Д-0042`, `ЗК-0042`, `ОТК-0042`, `АС-0042`, `СД-0042`, `Inv-0042`, `Pay-0042` — это **нормально** (разные счётчики, см. СПОР-13).

### 10.5 ЗК без товаров (GAP-009)

Если ЗК содержит только услуги/работы (`SERVICE`/`WORK`/`INSTALLATION`) — **никаких `StockMovement` не создаётся**, **`Shipment` не создаётся**. Кладовщик получает **только информационное сообщение** в `/warehouse`: «ЗК-0023 завершён, в Склад прихода нет (только услуги)».

### 10.6 Финансовое закрытие через внешние события, не через каскад

R4 из АНАЛИЗ-П1: «Order закрыт → что с КП/Договором/ЗК?» — закреплено: **никаких каскадных переходов**. Каждый модуль управляет своими статусами. Финансы **ЗАВИСЯТ** от отгрузки (`Shipment.delivered`), но **НЕ НАОБОРОТ**.
