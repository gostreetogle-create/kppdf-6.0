# PROJECT-STATE-LOG.md — Журнал инкрементальных изменений

> **Назначение.** Этот файл фиксирует **все глобальные изменения** проекта KPPDF CRM v6 во времени. Каждый агент при старте читает `AGENT-ENTRYPOINT.md` + **этот файл** (последние 10 записей) — это даёт контекст «куда ветер дует» без перечитывания всей теоретической базы.
>
> **Когда писать:** при любом изменении, затрагивающем **больше одного модуля** или **общую схему** (см. §0.1 AGENT-ENTRYPOINT.md).
>
> **Когда НЕ писать:** локальные правки внутри одного файла одного модуля — это не инкрементальное изменение, это обычная работа.

---

## 0. Схема записи

Каждая запись **обязана** иметь следующие поля:

| Поле | Описание | Обязательное |
|---|---|---|
| **Дата** | ISO формат `YYYY-MM-DD` | да |
| **ID** | `PSL-NNN` (порядковый номер, монотонно растёт) | да |
| **Тип** | `schema` / `process` / `structure` / `terminology` / `critical_fix` | да |
| **Модуль** | `Универсально` / `КП` / `Договор` / `ЗК` / `Склад` / `Финансы` | да |
| **Описание** | что именно изменилось | да |
| **Причина** | почему (ссылка на GAP / СПОР / OQ если есть) | да |
| **Затронутые файлы** | список файлов где надо искать следствие | да |
| **Автор** | роль агента + ник (например «Архитектор / Claude-Sonnet-4.5») | да |
| **Связанные OQ** | OQ-XXX из `OPEN-QUESTIONS-MASTER.md` или `00-spr/00-otkrytye-voprosy.md` (если есть) | нет |

### Что считается «глобальным изменением»

| Писать в LOG | НЕ писать в LOG |
|---|---|
| Изменили тип поля `Price` с Int на Float (затрагивает КП + Договор + Счет) | Поправили опечатку в одном файле модуля КП |
| Добавили новый статус `EXPIRED` (затрагивает все модули) | Переставили абзацы в README модуля |
| Переименовали папку `01_КП/01_шаблоны/` → `01_КП/01-shablon/` | Добавили сценарий в USER-JOURNEYS |
| Ввели новую политику (например, ставка НДС по умолчанию = 0% для экспорта) | Добавили edge-кейс в Бизнес-правила модуля КП |
| Поменяли RBAC-права роли `manager` | Исправили кросс-ссылку между двумя файлами одного модуля |

### Правила ведения

- **ID монотонный:** `PSL-001`, `PSL-002`, ... — никогда не переиспользовать
- **Сортировка:** новые записи сверху (свежее = выше)
- **Не редактировать старые записи** — для изменений задним числом создавать новую запись со ссылкой на старую («см. PSL-XXX, обновлено»)
- **Связи с OQ:** каждое инкрементальное изменение, закрывающее дыру, должно ссылаться на OQ-XXX
