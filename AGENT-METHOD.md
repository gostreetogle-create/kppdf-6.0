# AGENT-METHOD.md — Метод прохождения по проекту

> **Назначение.** Пошаговый метод работы ИИ-агента: что читать, в каком порядке, как реагировать на противоречия и дыры.
>
> **Обязательное условие:** до ознакомления с этим файлом агент ОБЯЗАН прочитать [`AGENT-ENTRYPOINT.md`](AGENT-ENTRYPOINT.md). Также рекомендуется [`AGENT-ROLES.md`](AGENT-ROLES.md) для понимания своей роли.

---

## 0. Контекст

Метод построен из двух уровней: (а) **быстрый старт** для типовых задач (~17 минут), (б) **полное погружение** для задач с правкой схемы или новых модулей (~62 минуты).

## 3. Декомпозиция плоского файла

**Правило 3.1.** Если файл > 400 строк и ещё не структурирован:
1. Прочитай ВЕСЬ файл целиком.
2. Найди все сущности: организации, товары, документы, действия, edge-cases («если», «но», «кроме»).
3. Распредели по 5 папкам в порядке зависимостей:
   - `00-spr/` — Справочники (фундамент)
   - `01-shablon/` — Скелеты (как создаётся визуальный макет)
   - `02-konstruktor-*/` — Сборка (UI + механика)
   - `03-zhiznennyj-cikl/` — Процесс
   - `04-pravila/` — Ограничения
4. Внутри каждой папки — файлы target 80-250 строк. Hard limit 400 (иначе дробление). Меньше 40 — объединение с соседним.
5. Каждый файл: контекст → определение → правила → edge-кейсы.

> **Примечание для Архитектора:** [`AGENT-ROLES.md` §2.1](AGENT-ROLES.md) ограничивает Архитектора «структурой, не бизнесом». Декомпозиция по 5 тематическим папкам — это структурное решение (как удобнее читать / поддерживать), не бизнес-классификация. Распределение **файлов по папкам** — входит в роль Архитектора. Содержимое файлов — нет.

## 4. Противоречия

**Правило 4.1.** При обнаружении противоречия между правилами/утверждениями:
- **Шаг 1.** Зафиксируй в файле `ПРОТИВОРЕЧИЕ-N: [описание]` в месте где оно обнаружено.
- **Шаг 2.** В отчёте своему «заказчику» (продакт-оунеру) перечисли все найденные противоречия.
- **Шаг 3.** **Не пытайся разрешать сам** — это работа Координатора + продакт-оунера.
- **Шаг 4.** Если противоречие критично для твоей работы — пометь как 🔴 P0 и подожди решения.

## 5. Дыры в покрытии и STUB-исключение

### 5.1 Фиксация дыр

**Правило 5.1.** Дыры в покрытии фиксируются на двух уровнях:
- **Проектные дыры** (затрагивают несколько модулей или общую схему) → `99_Справочники/OPEN-QUESTIONS-MASTER.md`.
- **Локальные дыры** (только в текущем модуле) → `<модуль>/00-spr/00-otkrytye-voprosy.md`.

Тип: missing_rule / contradiction / undefined_behavior / broken_link / terminology_drift.
Приоритет: 🔴 P0 / 🟡 P1 / 🟢 P2.

### 5.2 Отсутствующий файл и STUB

**Правило 5.2.1.** Если при прохождении метода обещанный файл не найден (например, `README.md`, `BIG-BOOK.md`, `99_Справочники/SCHEMA-CONSOLIDATED.md`):
- **Шаг 1.** Проверь явно через `glob` или `list_directory` — может файл лежит в другой подпапке.
- **Шаг 2.** Если файл действительно отсутствует — это дыра в покрытии. Зафиксируй в `00-otkrytye-voprosy.md` (или в `OPEN-QUESTIONS-MASTER.md` если проектная) с приоритетом 🔴 P0 и пометкой «BLOCKING: агент не может стартовать без этого файла».
- **Шаг 3.** Если роль и масштаб позволяют — создай минимальный файл-заглушку:
  - Минимум: заголовок + блок `> **Назначение.**` (1-2 строки) + `## 0. Контекст`.
  - Пометь явно: `# ⚠️ STUB: создан агентом, требует проверки координатором`.
  - НЕ пытайся наполнить содержанием «от себя» — это работа других ролей.
- **Шаг 4.** Если создавать файл вне твоей роли — стоп, зафиксируй дыру, отдай Координатору.

**Правило 5.2.2 (граница с атомарной ответственностью).** Создание STUB-файла **не является нарушением** атомарности:
- (а) Применяется только к **не-существующим** файлам. Никаких правок существующего контента, который принадлежит другой роли.
- (б) STUB содержит **только заголовок + маркер `⚠️ STUB` + заглушку** без содержания.
- (в) Как только другая роль начинает работать с STUB, **создатель больше его никогда не трогает** — это явный хендофф, зафиксированный в `PROJECT-STATE-LOG.md`.

Атомарность защищает от «дружеской» правки чужого контента, но не блокирует создание новой инфраструктуры.

## 6. Шаблон фиксации дыры (OQ-XXX)

Каждая запись **обязана** иметь следующие поля:

| Поле | Описание |
|---|---|
| **ID** | `OQ-XXX` (порядковый номер, уникальный в пределах файла) |
| **Модуль** | КП / Договор / Универсально / ... |
| **Дата обнаружения** | YYYY-MM-DD |
| **Обнаружил** | роль агента (Архитектор / QA-валидатор / ...) |
| **Тип** | missing_rule / contradiction / undefined_behavior / broken_link / terminology_drift |
| **Приоритет** | 🔴 P0 (блокер MVP) / 🟡 P1 (важно) / 🟢 P2 (мелочь) |
| **Что хотел сказать автор** | контекст — почему дыра вообще возникла |
| **Что реально отсутствует/противоречит** | конкретное описание проблемы |
| **Минимальное исправление** | предложение как починить (или «открыто для PO») |
| **Статус** | 🔵 OPEN / 🟢 RESOLVED / 🟡 DEFERRED |

**Пример:**

```markdown
## OQ-007 — Нет правила что делать если корзина пустая при формировании КП

| Поле | Значение |
|---|---|
| **ID** | OQ-007 |
| **Модуль** | КП |
| **Дата обнаружения** | 2026-06-26 |
| **Обнаружил** | Бизнес-аналитик |
| **Тип** | missing_rule |
| **Приоритет** | 🟡 P1 |
| **Что хотел сказать автор** | Менеджер случайно нажал «Сформировать КП» с пустой корзиной |
| **Что реально отсутствует** | В `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md §4` это поведение не описано |
| **Минимальное исправление** | Добавить явное правило: «Ошибка: Добавьте хотя бы один товар» |
| **Статус** | 🔵 OPEN |
```
