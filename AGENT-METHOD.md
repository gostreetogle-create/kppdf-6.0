# AGENT-METHOD.md — Метод прохождения по проекту

> **Назначение.** Пошаговый метод работы ИИ-агента: что читать, в каком порядке, как реагировать на противоречия и дыры.
>
> **Обязательное условие:** до ознакомления с этим файлом агент ОБЯЗАН прочитать [`AGENT-ENTRYPOINT.md`](AGENT-ENTRYPOINT.md). Также рекомендуется [`AGENT-ROLES.md`](AGENT-ROLES.md) для понимания своей роли.

Метод построен из двух уровней: (а) **быстрый старт** для типовых задач (~17 минут), (б) **полное погружение** для задач с правкой схемы или новых модулей (~62 минуты).

## 3. Декомпозиция плоского файла

**Правило 3.1.** Если файл > 400 строк и ещё не структурирован:
1. Прочитай ВЕСЬ файл целиком.
2. Найди все сущности: организации, товары, документы, действия, edge-cases («если», «но», «кроме»).
3. Распредели по 5 папкам в порядке зависимостей:
   - `00-spr/` — Справочники (фундамент)
   - `01-shablon/` — Скелеты (как создаётся визуальный макет)
   - `02-konstruktor-*/` — Сборка (UI + механика)
   - `03-zhiznennyj-cikl/` — Процесс
   - `04-pravila/` — Ограничения
4. Внутри каждой папки — файлы target 80-250 строк. Hard limit 400 (иначе дробление). Меньше 40 — объединение с соседним.
5. Каждый файл: контекст → определение → правила → edge-кейсы.

> **Примечание для Архитектора:** [`AGENT-ROLES.md` §2.1](AGENT-ROLES.md) ограничивает Архитектора «структурой, не бизнесом». Декомпозиция по 5 тематическим папкам — это структурное решение (как удобнее читать / поддерживать), не бизнес-классификация. Распределение **файлов по папкам** — входит в роль Архитектора. Содержимое файлов — нет.

## 4. Противоречия

**Правило 4.1.** При обнаружении противоречия между правилами/утверждениями:
- **Шаг 1.** Зафиксируй в файле `ПРОТИВОРЕЧИЕ-N: [описание]` в месте где оно обнаружено.
- **Шаг 2.** В отчёте своему «заказчику» (продакт-оунеру) перечисли все найденные противоречия.
- **Шаг 3.** **Не пытайся разрешать сам** — это работа Координатора + продакт-оунера.
- **Шаг 4.** Если противоречие критично для твоей работы — пометь как 🔴 P0 и подожди решения.

## 5. Дыры в покрытии и STUB-исключение

### 5.1 Фиксация дыр

**Правило 5.1.** Дыры в покрытии фиксируются на двух уровнях:
- **Проектные дыры** (затрагивают несколько модулей или общую схему) → `99_Справочники/OPEN-QUESTIONS-MASTER.md`.
- **Локальные дыры** (только в текущем модуле) → `<модуль>/00-spr/00-otkrytye-voprosy.md`.

Тип: missing_rule / contradiction / undefined_behavior / broken_link / terminology_drift.
Приоритет: 🔴 P0 / 🟡 P1 / 🟢 P2.

### 5.2 Отсутствующий файл и STUB

**Правило 5.2.1.** Если при прохождении метода обещанный файл не найден (например, `README.md`, `BIG-BOOK.md`, `99_Справочники/SCHEMA-CONSOLIDATED.md`):
- **Шаг 1.** Проверь явно через `glob` или `list_directory` — может файл лежит в другой подпапке.
- **Шаг 2.** Если файл действительно отсутствует — это дыра в покрытии. Зафиксируй в `00-otkrytye-voprosy.md` (или в `OPEN-QUESTIONS-MASTER.md` если проектная) с приоритетом 🔴 P0 и пометкой «BLOCKING: агент не может стартовать без этого файла».
- **Шаг 3.** Если роль и масштаб позволяют — создай минимальный файл-заглушку:
  - Минимум: заголовок + блок `> **Назначение.**` (1-2 строки) + `## 0. Контекст`.
  - Пометь явно: `# ⚠️ STUB: создан агентом, требует проверки координатором`.
  - НЕ пытайся наполнить содержанием «от себя» — это работа других ролей.
- **Шаг 4.** Если создавать файл вне твоей роли — стоп, зафиксируй дыру, отдай Координатору.

**Правило 5.2.2 (граница с атомарной ответственностью).** Создание STUB-файла **не является нарушением** атомарности:
- (а) Применяется только к **не-существующим** файлам. Никаких правок существующего контента, который принадлежит другой роли.
- (б) STUB содержит **только заголовок + маркер `⚠️ STUB` + заглушку** без содержания.
- (в) Как только другая роль начинает работать с STUB, **создатель больше его никогда не трогает** — это явный хендофф, зафиксированный в `PROJECT-STATE-LOG.md`.

Атомарность защищает от «дружеской» правки чужого контента, но не блокирует создание новой инфраструктуры.

### 5.3 Граница решений: когда спрашивать, когда решать

> **Контекст.** Зафиксировано 2026-06-26 по вербальному запросу PO (см. [`PROJECT-STATE-LOG.md` PSL-008](PROJECT-STATE-LOG.md)). Меняет поведение ВСЕХ будущих агентских запусков. До этого раздела действовало правило «все multi-choice ИИ → `ask_user`»; теперь это правило ограничено тремя конкретными случаями.

**Правило 5.3.1 (принцип автономии).** Агент работает автономно **по умолчанию**. Категорически НЕ спрашивай PO:

- «Продолжить ли?», «Точно ли это?», «Применить ли рекомендацию?», «Вы уверены?» — это блокирует прогресс.
- Любой вопрос типа «можно ли начать следующий шаг?» — **никогда**. Продолжай, фиксируй через `PSL-NNN`.
- Технические сомнения (выбор между A и B на уровне архитектуры/кода/библиотеки) — **никогда**. Выбери по своим критериям, зафиксируй через `PSL-NNN` с полем «Причина».

**Правило 5.3.2 (когда ОБЯЗАН спрашивать через `ask_user`).** Спрашивай ТОЛЬКО в одном из трёх случаев:

| Случай | Почему именно через `ask_user` | Пример |
|---|---|---|
| **(а) UX / usability / визуальное поведение** | Эмоционально-визуальная территория PO как конечного пользователя. Агент не решит, что «красивее» или «удобнее» | «Где разместить кнопку "Сохранить": сверху или снизу правой панели?» |
| **(б) Бизнес-перекрёсток (contract change)** | Две валидные альтернативы, обе меняют контракт с пользователем/клиентом/контрагентом | «Конвертация КП→Договор ЗАПРЕЩЕНА из статуса "Оплачено" vs РАЗРЕШЕНА?» |
| **(в) Необратимое действие с тяжёлыми последствиями** | `git push --force`, удаление БД, изменение прод-конфига, миграция в prod-окружении | «Применить миграцию `prisma migrate deploy` в прод?» |

**Правило 5.3.3 (фиксация каждого решения).** Любое автономное решение фиксируется как новая запись `PSL-NNN` в [`PROJECT-STATE-LOG.md`](PROJECT-STATE-LOG.md) с явным полем «Причина: принято автономно согласно Правилу 5.3.1». Это создаёт audit trail — если решение неверное, его можно откатить конкретной записью («см. PSL-NNN, обновлено: PSL-NNN+1»).

**Правило 5.3.4 (приоритет при конфликте 4 UX-принципов ниже).** Если применимо несколько принципов и они конфликтуют — приоритет по возрастанию значимости:

1. **Единообразие** (consistency, паттерн Windows/macOS) — **самый высокий**. Изменение единого паттерна ломает UX всего продукта.
2. **Интеллектуальные подсказки** (smart suggestions).
3. **Логичная последовательность** (intuitive flow).
4. **Минимизация трения** (low friction) — **самый низкий** из четырёх; конфликт с (1) всегда решается в пользу (1).

### 5.4 UX-принципы PO (канон для UI-роли)

> **Источник.** PO сформулировал голосом 2026-06-26 (вербальный запрос, без формального OQ). Зафиксировано канонически здесь + в [`PROJECT-STATE-LOG.md` PSL-008](PROJECT-STATE-LOG.md). Применяется к каждому `LAUNCH-UX` запуску и к ЛЮБОМУ UI-решению агента (Архитектор при структуре макета, Бизнес-аналитик при формулировке правил видимости, UX-дизайнер — главный потребитель).

#### Принцип 1. Единообразие (Consistency)

Все интерактивные элементы ведут себя **одинаково во всём продукте** — независимо от того, на какой странице / в каком модуле они находятся. Эталон: Windows / macOS / Linux DE — узнаваемость по общему паттерну.

**Конкретные правила для KPPDF CRM v6:**

| Элемент | Паттерн |
|---|---|
| Кнопка «Сохранить» | Всегда: иконка дискеты + текст «Сохранить»; shortcut `Ctrl/Cmd+S`; правый верхний угол формы; primary-цвет |
| Кнопка «Отмена» | Всегда: серый outline; shortcut `Esc`; левый нижний угол модалки |
| Dropdown | Единый стиль стрелки, одинаковая высота строк (32px), иконка поиска внутри при >5 элементах |
| Modal | Единый header (заголовок + крестик справа); footer с кнопками в правом нижнем углу; backdrop blur |
| Toast notification | Всегда справа снизу; авто-закрытие 5 сек; один формат (иконка + 1 строка текста) |
| Подтверждение удаления | Всегда: «Удалить X? Это действие нельзя отменить» + кнопки «Удалить» (danger) + «Отмена» |

#### Принцип 2. Интеллектуальные подсказки (Smart Suggestions)

Система **активно помогает** пользователю, а не пассивно ждёт ввода. Реализуется через:

| Возможность | Где применимо |
|---|---|
| **Автокомплит** в полях: ИНН клиента, артикул товара, название организации | КП-редактор, форма создания Клиента |
| **Контекстные suggestions** в dropdown: «Иванов Иван Иванович (по ИНН 7707083893, ООО Клиент)» — НЕ просто ФИО | Dropdown выбора клиента |
| **Защита от опечаток** — Zod-runtime валидация в момент ввода + красная подсветка + tooltip с форматом | Все формы v1 |
| **Inline-валидация** — пока пользователь печатает, поле подсвечивается 🟢 если валидно | Все обязательные поля |
| **Show recent / favorite** — последние 5 КП менеджера в выпадашке (серверная подгрузка) | Корневой dropdown выбора КП |
| **WYSIWYG preview А4** — центральная зона редактора КП **точно соответствует** печатной форме | Редактор КП («живой preview А4» из правки 4.9) |

**Что это значит для стека:** `react-hook-form` + `@hookform/resolvers/zod`; `Mantine Autocomplete` с `limit={10}`; debounce **250мс** для автокомплита.

#### Принцип 3. Логичная последовательность (Intuitive Flow)

Каждое следующее действие **очевидно** из текущего состояния экрана. Пользователь не должен читать инструкции, чтобы понять «что дальше».

**Проявления:**

- **Прогрессивное раскрытие** — кнопки / поля появляются по мере необходимости, а не все сразу. «Опция → действие».
- **Контекстные подсказки** — около сложного поля `?` icon → tooltip с примером и форматом.
- **Стрелки переходов / breadcrumbs** — показывают, где пользователь в workflow («КП-2026-0421 → Шаг 3 из 5»).
- **Жёлтая ⚠️ по умолчанию + режим руководителя** (правка D) — мягкое привлечение внимания без агрессии.
- **Back / Cancel всегда доступны** — никакой irrevocability без явного подтверждения.

#### Принцип 4. Минимизация трения (Low Friction)

Минимум кликов для типовых сценариев. Обучение «на лету» через визуальные подсказки — никаких onboarding-modal на старте.

**Конкретные оптимизации:**

- **«Типовой КП» → одна кнопка**: «Создать типовой КП из шаблона X» + автозаполнение позиций.
- **Авто-сохранение каждые 5-10 сек** (правка C) — пользователь НЕ думает о «Сохранить».
- **Enter → следующее поле** (не сабмит формы); сабмит только по «Сохранить» или `Ctrl+Enter`.
- **Drag-and-drop товаров в корзину** из любого места (глобальный поиск / список товаров).
- **Клавиатурные shortcuts**: `Ctrl+K` (глобальный поиск), `Ctrl+N` (новый КП), `Esc` (отмена/закрыть модалку).
- **Дефолты умные**: ставка НДС по последнему КП; валюта = RUB; язык = ru; юрлицо = последнее использованное этим менеджером.

#### Что НЕ входит в эти 4 принципа

- ❌ **Визуальный дизайн** (цвета, шрифты, spacing, иконки) — отдельный эстетический документ `UI-VISUAL-PRINCIPLES.md` (TODO Phase 2).
- ❌ **Архитектурные решения** (папки, ORM, SSR) — это технические решения, не UX.
- ❌ **Бизнес-логика** (правила, RBAC, state-машина) — территория Бизнес-аналитика и PO как бизнес-заказчика (Правило 5.3.2(б)).

### 5.5 Применение правил границы решений на практике

| Сценарий PO говорит | Реакция агента |
|---|---|
| «Сделай X» | Прочитай X, действуй по соответствующему Правилу 5.3. Никаких уточнений, если X однозначен. |
| «Сделай что считаешь нужным» | Максимальная автономия. Зафиксируй выбор в `PSL-NNN`, обоснуй критерии. |
| «Что лучше — A или B?» | Если A vs B это технический выбор → реши сам + `PSL-NNN`. Если визуальное/бизнес — `ask_user`. |
| «Можно ли я сделаю X?» (PO предлагает) | Соглашайся по умолчанию, фиксируй через `PSL-NNN`. |
| «Доделай что осталось» | Пройди §2 «Полное погружение», выбери самое приоритетное по открытым OQ / V-проверкам / PSL, начни. |

### 5.6 Pre-action Checklist + Post-action Checkpoint (обязательный паттерн)

> **Контекст.** Зафиксировано 2026-06-27 по вербальному запросу PO (см. [`PROJECT-STATE-LOG.md` PSL-009](PROJECT-STATE-LOG.md)). Применяется к КАЖДОМУ work-сеансу агента. Цель: **сохранение контекста при обрыве связи** (интернет, браузер, перезагрузка) + **audit trail** что сделано за сеанс. Это **закон**, а не рекомендация.

**Правило 5.6.1 (Pre-action Checklist появляется в начале каждой значимой работы).** Перед НАЧАЛОМ любой значимой работы агент ОБЯЗАН составить письменный план. Два варианта размещения:

- **(а) В первом ответе сеанса** (когда работает без отдельного большого файла) — отдельный блок «## Pre-action Checklist» сразу после приветствия / до первого действия.
- **(б) В НАЧАЛЕ изменяемого/создаваемого файла** (когда работа идёт в конкретном файле) — блок `# IN-WORK CHECKLIST` после шапки файла.

Обязательные поля:

| Поле | Пример |
|---|---|
| **Цель работы** | Построить канонический REGISTRY-OF-RULES.md согласно ТЗ-001. |
| **Объём** | 4 файла: REGISTRY ≈2500 строк + LOG + REPORT + AMBIGUITIES. |
| **Подход** | Последовательное прохождение 23 файлов из §5 ТЗ → извлечение правил в 12 разделов → self-verify §10 → handoff через REPORT. |
| **Шаги (в порядке)** | 1. Прочитать §5.1 приоритетный список. 2. Извлечь RBAC-правила. ... N. Завершить REPORT. |
| **Что НЕ делаю (out of scope)** | Не редактирую МОДУЛЬ-доки. Не пишу код. |
| **Что может пойти не так** | REGISTRY превысит 4000 строк → разбить на PART-1/2/3. Нашёл противоречие → AMBIGUITIES. |

**Правило 5.6.2 (Post-action Checkpoint появляется в КОНЦЕ каждого сеанса или перед долгой паузой).** Обязательный формат:

| Поле | Пример |
|---|---|
| **Сделано** | [x] Прочитан AGENT-METHOD.md — 100% сделано. [x] Извлечено 87 RBAC-правил. |
| **Не сделано / требует продолжения** | [ ] Cross-module триггеры — остановился на TRI-007 из-за сложности. |
| **Открытые вопросы** | Как разрешить противоречие PROT-002 (см. AMBIGUITIES)? |
| **Если связь оборвалась** | Следующий агент продолжит с TRI-008 → §5 «cross-module». Прочитать 01-LOG.md последнюю запись. |

**Правило 5.6.3 (где искать).** Pre-action и Post-action фиксируются в одном из двух мест:
- **(а)** В изменяемом файле (если файл переписывается / создаётся) — `## IN-WORK CHECKLIST` секция после заголовка.
- **(б)** В `01-LOG.md` или специализированном work-log файле (если работа без большого файла).

**Правило 5.6.4 (когда ОБЯЗАТЕЛЬНО).** Применяется к:
- Phase 1 Bootstrap прогонов Аналитика (Run 1/5).
- Phase 1 Bootstrap миграций Prisma.
- Phase 2 реализации модуля КП.
- Любой задаче, результат которой > 200 строк или > 3 файлов.

Не применяется к: микро-правкам (<10 строк, 1 файл), ответам на вопросы без правок.

**Правило 5.6.5 (зачем именно так — два сценария обрыва).**
- **(1) Обрыв интернета / краш браузера** → следующий агент читает самый свежий изменённый файл, видит блок `# IN-WORK CHECKLIST` → точно стыкуется с тем, что было сделано. Не нужно переспрашивать «что делал».
- **(2) Агент остановился в середине работы** → PO / следующий агент видит «[x] Сделано A, [x] Сделано B, [ ] Не сделано C, причина: Z» → может либо завершить, либо передумать.

Без этих слоёв контекст **теряется при любом обрыве** и приходится переспрашивать или повторять работу. **См. также:** §6 «Протокол возобновления (потеря контекста)» — это 6-шаговая процедура для случая **полного** обрыва контекста (новая сессия, краш браузера); §5.6 — это паттерн для **текущего сеанса** (до того, как обрыв произойдёт). Два слоя работают последовательно: §5.6 предотвращает потерю, §6 восстанавливает если потеря случилась.

**Правило 5.6.6 (стандартный план «стандартной работы» для Бизнес-аналитика / Архитектора).** Применяется как шаблон Pre-action для типовых задач:

```markdown
## Pre-action Checklist
**Цель:** …
**Объём:** …
**Подход:** …

### Шаги
1. [ ] Прочитать приоритетный список §X AGENT-METHOD.
2. [ ] Извлечь правила/правила/сущности согласно §Y.
3. [ ] Создать файл Z со структурой по §W.
4. [ ] Пройти self-check (AGENT-REVIEW.md).
5. [ ] Создать PSL-NNN запись в PROJECT-STATE-LOG.md с причиной.

### Не делаю
- (список)

### Риски
- (вероятные ситуации → действия)
```

(Полный пример для ТЗ-001 / REGISTRY см. в `99_Справочники/TASKS/ТЗ-001-КАТАЛОГ-ПРАВИЛ.md` §17.)

## 6. Шаблон фиксации дыры (OQ-XXX)

Каждая запись **обязана** иметь следующие поля:

| Поле | Описание |
|---|---|
| **ID** | `OQ-XXX` (порядковый номер, уникальный в пределах файла) |
| **Модуль** | КП / Договор / Универсально / ... |
| **Дата обнаружения** | YYYY-MM-DD |
| **Обнаружил** | роль агента (Архитектор / QA-валидатор / ...) |
| **Тип** | missing_rule / contradiction / undefined_behavior / broken_link / terminology_drift |
| **Приоритет** | 🔴 P0 (блокер MVP) / 🟡 P1 (важно) / 🟢 P2 (мелочь) |
| **Что хотел сказать автор** | контекст — почему дыра вообще возникла |
| **Что реально отсутствует/противоречит** | конкретное описание проблемы |
| **Минимальное исправление** | предложение как починить (или «открыто для PO») |
| **Статус** | 🔵 OPEN / 🟢 RESOLVED / 🟡 DEFERRED |

**Пример:**

```markdown
## OQ-007 — Нет правила что делать если корзина пустая при формировании КП

| Поле | Значение |
|---|---|
| **ID** | OQ-007 |
| **Модуль** | КП |
| **Дата обнаружения** | 2026-06-26 |
| **Обнаружил** | Бизнес-аналитик |
| **Тип** | missing_rule |
| **Приоритет** | 🟡 P1 |
| **Что хотел сказать автор** | Менеджер случайно нажал «Сформировать КП» с пустой корзиной |
| **Что реально отсутствует** | В `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md §4` это поведение не описано |
| **Минимальное исправление** | Добавить явное правило: «Ошибка: Добавьте хотя бы один товар» |
| **Статус** | 🔵 OPEN |
```
