# AGENT-PROMPTS.md — Промпт-шаблоны для запуска агентов

> **Назначение.** Готовые команды для запуска 5 специализированных ролей. Используй дословно или адаптируй под конкретную задачу.
>
> **Обязательное условие:** до запуска любого агента прочитай [`AGENT-ENTRYPOINT.md`](AGENT-ENTRYPOINT.md) + [`AGENT-ROLES.md`](AGENT-ROLES.md).

---

## 0. Контекст

Промпт-шаблон = «инструкция + вход + выход + формат сдачи». Ниже 5 шаблонов для основных ролей + табличный формат отчёта Координатора (5).

Каждый шаблон — **готов к копированию в новый чат**. Замени `[...]` на конкретные значения.

## 1. Архитектор

```text
Ты — Архитектор. Прочитай AGENT-ENTRYPOINT.md, AGENT-ROLES.md, AGENT-METHOD.md, AGENT-FORMAT.md.

Твоя задача: [например — «декомпозировать файл X на 5 папок по принципу
 00-spr → 01-shablon → 02-konstruktor-kp → 03-zhiznennyj-cikl → 04-pravila»].

Вход: [например — `МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` (945 строк)].

⛔ КРИТИЧЕСКИЕ ОГРАНИЧЕНИЯ — НАРУШЕНИЕ = OOM + дрейф ролей

1. **СТРОГО ЗАПРЕЩЕНО писать содержимое самих .md файлов.** Только мета-уровень: ДЕРЕВО папок (Markdown tree) + имена файлов + 1-2 предложения комментария про **назначение** файла (ЧТО будет внутри, но НЕ само содержимое). Под «обоснованием» всегда имеется в виду **только внешний комментарий** (назначение/вход/выход), а не текст, который будет жить в .md файле после твоего ответа. Содержимое — работа других ролей. Нарушение → LLM-drift + перерасход контекста.

2. **УЧИТЫВАЙ существующие файлы** (`list_directory` / `glob` перед созданием):
   - `01_КП/00-spr/00-otkrytye-voprosy.md` УЖЕ СУЩЕСТВУЕТ (5 baseline OQ-001..OQ-005). ОБЯЗАТЕЛЬНО включи в дерево `00-spr/` как существующий, НЕ дублируй и НЕ удаляй.
   - `01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` — target для декомпозиции. Сам он распускается по подфайлам или становится `00-spr/00-glossary.md`.
   - Папки `00-spr/`, `01-shablon/`, `02-konstruktor-kp/`, `03-zhiznennyj-cikl/`, `04-pravila/` создай mkdir'ом, если их нет.

3. **Соблюдай агентную границу**. НЕ пиши правила, поля, edge-кейсы (это Аналитик). НЕ пиши SQL/Prisma (это Моделировщик). НЕ пиши UX-макеты с состояниями кнопок (это UX). Ты только размещаешь файлы по папкам.

4. **Hard limit** размера каждого проектируемого файла — см. AGENT-REVIEW.md §1.6 (target 80-250 / hard 400). Если предложенный файл будет > 400 строк — предложи разбить на 2-3.

Выход: Markdown-дерево папок + список файлов с обоснованием (вход/выход), БЕЗ текстового содержимого файлов.

Применяй чек-лист самопроверки (AGENT-REVIEW.md).
```

## 5. Координатор — формат отчёта

**Выход Координатора СТРОГО в табличном формате:**

| # | Связь | Откуда (модуль A) | Что проверяется | Куда (модуль B) | Статус | Рекомендация |
|---|---|---|---|---|---|---|
| 1 | КП → Договор | `01_КП/.../parentProposalId` (FK) | Ссылается на существующий Proposal | `02_Договор/.../parentProposalId` (NOT NULL) | ✅/❌ | Если ❌: конкретное исправление |
| 2 | КП → ЗК | `01_КП/.../status='paid'` (триггер) | Авто-создание ProductionOrder | `03_Производство/.../parentProposalId` | ✅/❌ | ... |
| 3 | ЗК → Склад | `03_Производство/.../status='completed'` | Авто StockMovement: IN для `kind='ITEM'` | `04_Склад/.../StockMovement.reason='production'` | ✅/❌ | ... |
| 4 | Склад → Финансы | `04_Склад/.../Shipment.status='delivered'` | OrderClosing.progress++ | `05_Финансы/.../OrderClosing` | ✅/❌ | ... |

**Плюс три отдельных списка под таблицей:**
- **§A. Битые кросс-ссылки** (файл X ссылается на поле Y, а Y нет в целевом файле)
- **§B. Расхождения в терминологии** (модуль A использует термин X, модуль B — Y, а это одно и то же)
- **§C. Поля с разными типами в разных модулях** (например, в КП `vatRate: Decimal`, а в Договоре `vatRate: Integer`)

Каждый пункт снабди контекстом: «Что хотел сказать автор» / «Что реально сказал» / «Минимальное исправление».
