## 0. Самообслуживание чек-листа (правила ведения)

**Когда обновлять этот файл** — только если изменилась **глобальная маршрутизация** (новая роль, новый документ-источник, новый pipeline run, новый модуль). Для точечных правок — не трогать, обновите источник по ссылке.

**Когда НЕ обновлять:**
- Изменился один STUB в `01_КП/` → править сам STUB, не этот файл.
- Закрылась одна OQ → править `99_Справочники/OPEN-QUESTIONS-MASTER.md`, не этот файл.
- Закрылся один RUN Pipeline → править `PROJECT-STATE-LOG.md` (новая запись PSL-NNN), не этот файл.

**Правила содержания:**
1. **Не дублировать** таблицы решений и журнал PSL — только ссылки (анти-паттерн A5).
2. **Каждый маршрут** = одна строка «Если хочешь X → читай Y» (анти-паттерн A3 — без вводных абзацев).
3. **Все варианты выбора** (multi-choice от AI в конце ответа) — фиксируются через `ask_user` или добавляются в `PROJECT-STATE-LOG.md` как запись PSL-NNN («Причина» = «зафиксирован выбор пользователя»).
4. **Hard limit:** при превышении 250 строк → немедленно разбить на `CHECKLIST.md` + `CHECKLIST-TRACKER.md` с явной cross-link (анти-паттерн A9).
5. **Формат:** следовать [`AGENT-FORMAT.md`](AGENT-FORMAT.md) (П1-П8): нумерация правил, цифры вместо слов, кросс-ссылки вместо копий.

## 6. Протокол возобновления (потеря контекста)

**Если контекст утерян (обрыв интернета, новая сессия, краш):**

1. **Открой `CHECKLIST.md`** (этот файл — точка входа).
2. **Прочитай `PROJECT-STATE-LOG.md`** — последние 5 записей PSL-NNN сверху вниз. Определи последний завершённый шаг.
3. **Спроси PO ровно одну фразу:** *«Контекст восстановлен. Последний шаг был **PSL-NNN — <название>**. Продолжить с того же места, начать новую задачу или закрыть run?»*
4. **Если PO говорит «продолжить»:** открой соответствующий `LAUNCH-X.md` (§4) и следуй его §1 «Файлы для attach».
5. **Если нужно сменить роль:** см. §4 → новый запуск через новый LAUNCH-пакет.
6. **Если задача уже сделана (PSL-NNN = DONE):** переходи к следующему run / новой задаче без повторов.

**Признак восстановления успешно:** ПО подтвердил «continue» + ты видишь §3 текущего состояния + последний PSL-запись совпадает с памятью.

**Правило: НЕ удалять.** Завершённые элементы складируются в архивные зоны:

| Архивная зона | Что туда |
|---|---|
| `git rm` + git history | исходные файлы, которые распускаются (пример: `01_КП/МОДУЛЬ-КОММЕРЧЕСКОЕ-ПРЕДЛОЖЕНИЕ.md` 945 строк → 20 STUB; исходник в git history) |

| `archive/` подпапка (если создана в будущем) | версионированные отчёты прошлых версий |

**Перед архивированием:** убедись что информация зафиксирована в git через коммит + push (см. ## 9. «Задавай умные вопросы» (защита от ломания сделанного)

Прежде чем что-то менять, спроси себя (через `ask_user` если неуверен):

| Вопрос | Когда задать | Если ответ «не знаю» |
|---|---|---|
| Это сделано? | Каждый раз перед стартом (§2) | Сначала проверь, потом спрашивай PO |
| Это сломает PSL-NNN? | Перед edit/rm любого .md | Найди PSL-NNN для file → см. причину → спроси PO |
| Это накладывается на закрытую OQ? | Перед введением нового правила | Поиск по `OPEN-QUESTIONS-MASTER.md` |
| Это накладывается на закрытый СПОР? | Перед изменением RBAC/enum | Поиск по `СПОРНЫЕ-МОМЕНТЫ.md` (15 закрытых) |
| Это hard limit (250/400 строк)? | Перед созданием .md > 200 строк | Разбей на 2-3 файла + cross-link |
| Это в правильной роли (Правило 3.1)? | Принимая задачу | Проверь [`AGENT-ROLES.md`](AGENT-ROLES.md) §2 атомарность — если пересекается с двумя ролями, разбей на sub-task |

**Правило «не ломай сделанного»:** Если ответ на любой из вопросов «да» — остановись и спроси PO через `ask_user`. Если PO уверен «override» — фиксируй новое решение как новый PSL-NNN со ссылкой на переопределённую запись (PSA: «см. PSL-XXX, обновлено»).
